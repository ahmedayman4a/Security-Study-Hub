Kerckhoffs
Kerckhoffs's Principle states that the security of a cipher should depend on:
aThe secrecy of the algorithm, not the key.
bThe secrecy of the key, not the algorithm.
cUsing a key of at least 128 bits.
dEncryption and decryption using identical operations.
Explanation
✓ Correct: (b)
Kerckhoffs's principle: the algorithm may be made public; security must rest entirely on the secrecy of the key. This is why AES is fully documented — knowing how AES works gives an attacker no advantage without the key. It enables public scrutiny and peer review of algorithms without compromising deployed systems.
2
Avalanche Effect
The avalanche effect in block ciphers means:
aA small change in the key causes a proportionally small change in the ciphertext.
bIncreasing the key size always doubles the effective security.
cFlipping 1 bit in the plaintext or key should cause roughly half the ciphertext bits to flip.
dThe cipher output grows in size proportional to the number of rounds.
Explanation
✓ Correct: (c)
The avalanche effect: a single-bit change in the input (plaintext or key) should flip approximately 50% of the output bits. This ensures ciphertexts appear completely random and hides any statistical relationship between plaintext and ciphertext. Ciphers lacking this property leak structural information — you could find partial plaintext by observing which ciphertext bits changed.
3
Confusion vs Diffusion
Which best describes Shannon's two design properties, confusion and diffusion?
aConfusion spreads bits across ciphertext; diffusion hides the key using non-linear substitutions.
bConfusion hides the key-to-ciphertext relationship (via substitution); diffusion spreads plaintext influence across ciphertext (via permutation).
cBoth confusion and diffusion are properties of stream ciphers only.
dDiffusion prevents differential cryptanalysis; confusion prevents linear cryptanalysis exclusively.
Explanation
✓ Correct: (b)
Shannon identified two essential properties: confusion — making the relationship between key and ciphertext as complex/non-linear as possible (achieved via S-boxes, substitutions); diffusion — ensuring each plaintext bit influences many ciphertext bits (achieved via permutations, MixColumns). In AES: SubBytes provides confusion, ShiftRows+MixColumns provide diffusion.

AES Parameters
AES operates on which block size, with which key sizes?
a64-bit block, 56-bit key only.
b64-bit block, 128/192/256-bit key.
c128-bit block, 56-bit key only.
d128-bit block, 128/192/256-bit key.
Explanation
✓ Correct: (d)
AES (Rijndael) always uses a 128-bit (16-byte) block, regardless of key length. Supported key lengths are 128, 192, or 256 bits. The corresponding round counts are 10, 12, and 14. Compare with DES: 64-bit block with a 56-bit key (8 parity bits are stripped from the 64-bit key input).
5
AES Rounds
How many rounds does AES-128 use, and how many does AES-256 use?
aAES-128: 10 rounds; AES-256: 14 rounds.
bAll AES variants use 16 rounds.
cAES-128: 8 rounds; AES-256: 16 rounds.
dThe number of rounds depends only on the plaintext, not the key size.
Explanation
✓ Correct: (a)
AES round counts scale with key size: AES-128 → 10 rounds, AES-192 → 12 rounds, AES-256 → 14 rounds. More rounds provide a higher security margin, compensating for the additional key material. Note: each round consists of SubBytes, ShiftRows, MixColumns (omitted in the final round), and AddRoundKey.
6
ShiftRows
What does the AES ShiftRows operation do to the 4×4 state matrix?
aIt cyclically rotates bytes in column \(i\) by \(i\) positions.
bIt cyclically rotates bytes in row \(i\) by \(i\) positions to the left.
cIt multiplies each byte by a fixed constant in \(\text{GF}(2^8)\).
dIt XORs every byte with the corresponding round key byte.
Explanation
✓ Correct: (b)
ShiftRows cyclically shifts each row of the 4×4 state: row 0 unchanged, row 1 shifted left by 1, row 2 by 2, row 3 by 3. This is a permutation that provides inter-column diffusion — bytes originally in the same column end up in four different columns, so subsequent MixColumns mixes bytes from different original columns.
7
MixColumns
The AES MixColumns operation treats each column of the state as:
aA 32-bit integer XORed with the next column.
bA lookup index into a 256-entry S-box.
cA polynomial over \(\text{GF}(2^8)\), multiplied by a fixed matrix polynomial.
dA 4-byte sequence that is rotated and added to the round key.
Explanation
✓ Correct: (c)
MixColumns treats each 4-byte column as a degree-3 polynomial over \(\text{GF}(2^8)\) and multiplies it by the fixed polynomial \(3x^3 + x^2 + x + 2\) (mod \(x^4+1\)). This is implemented as matrix-vector multiplication over \(\text{GF}(2^8)\). Each output byte depends on all four input bytes in the column — providing strong intra-column diffusion.

Key Schedule g()
In the AES key schedule, the function \(g()\) is applied to every 4th word. What does it do?
aIt XORs consecutive words together to create the next word.
bIt rotates the 4-byte word, applies SubBytes to each byte, then XORs with a round constant (Rcon).
cIt reverses the bit order of the word and adds it to the previous round key.
dIt doubles the key length by appending a permuted copy of the current key.
Explanation
✓ Correct: (b)
The \(g()\) function (applied to \(W[4k-1]\) when generating round key \(k\)): (1) RotWord — rotate the 4 bytes left by one position; (2) SubWord — apply the AES S-box to each of the 4 bytes; (3) XOR with Rcon[k], the round constant for round \(k\). This non-linear transformation prevents the key schedule from being inverted from partial key knowledge.

Caesar Cipher
The Caesar cipher has how many distinct, meaningful key values?
a26 (one for each letter)
b25 (shifts 1 through 25)
c128 (ASCII printable range)
d26! (all alphabet permutations)
Explanation
✓ Correct: (b)
A shift of 0 is the identity (no encryption), so only shifts 1–25 are meaningful, giving 25 keys. Brute-force needs at most 25 trials — trivially broken. Compare with the monoalphabetic substitution cipher which has \(26! \approx 4 \times 10^{26}\) possible keys, yet is still broken in seconds using frequency analysis.
10
Vigenère Weakness
The Vigenère cipher is broken in practice because:
aThe key is always exactly 26 characters, making it equivalent to a monoalphabetic cipher.
bOnce the key period is found (via Kasiski test or index of coincidence), each position reduces to an independent Caesar cipher broken by frequency analysis.
cIts output has the same letter frequency distribution as the original plaintext.
dIt is perfectly secure for any key length shorter than the message.
Explanation
✓ Correct: (b)
The Vigenère cipher's repeating key is its fatal flaw. The Kasiski examination finds repeated ciphertext segments (which likely correspond to the same plaintext encrypted with the same key fragment), revealing the key length. The index of coincidence independently confirms the key length. Once the period \(L\) is known, the ciphertext is split into \(L\) groups, each encrypted with a single Caesar shift — broken instantly by frequency analysis.

MITM on 2DES
Why does the meet-in-the-middle (MITM) attack reduce 2DES's effective security from 2112 to approximately 257?
aAn algebraic flaw in DES makes the combined key factorizable.
b2DES is equivalent to single DES because DES forms a group under composition.
cEncrypt known plaintext with all 256 possible K₁ values and store the results; decrypt known ciphertext with all 256 possible K₂ values and look for a match — the total cost is 2×256 ≈ 257 with 256 space.
dThe attacker guesses K₁ from the IV used in CBC mode.
Explanation
✓ Correct: (c)
MITM: given known plaintext \(P\) and ciphertext \(C = E_{K_2}(E_{K_1}(P))\), build a table of \(E_{K_1}(P)\) for all \(2^{56}\) values of \(K_1\). Then compute \(D_{K_2}(C)\) for all \(2^{56}\) values of \(K_2\) and look for a collision with the table. A collision gives a candidate \((K_1, K_2)\). Total work: \(2 \times 2^{56} \approx 2^{57}\) — far less than the expected \(2^{112}\).
12
3DES EDE
In 3DES EDE (Encrypt-Decrypt-Encrypt) mode, the operations applied are:
a\(E_{K_1} \to E_{K_2} \to E_{K_3}\)
b\(E_{K_1} \to D_{K_2} \to E_{K_3}\)
c\(D_{K_1} \to E_{K_2} \to D_{K_3}\)
d\(E_{K_1} \to E_{K_2} \to D_{K_3}\)
Explanation
✓ Correct: (b)
3DES EDE: Encrypt with \(K_1\), Decrypt with \(K_2\), Encrypt with \(K_3\). The "Decrypt in the middle" enables backward compatibility: if \(K_1 = K_2 = K_3\), the EDE sequence is \(E \circ D \circ E = E\) (the two cancel), reducing 3DES to plain DES. With three independent 56-bit keys, the effective security is approximately 112 bits (not 168, due to time-space tradeoff attacks).
13
ECB Weakness
The critical weakness of ECB (Electronic Codebook) mode is:
aIt requires a secret IV that must be transmitted alongside the ciphertext.
bIdentical plaintext blocks always produce identical ciphertext blocks, leaking structural information.
cA single-bit ciphertext error corrupts the entire message.
dECB cannot be parallelized, making it too slow for practical use.
Explanation
✓ Correct: (b)
In ECB, each block is encrypted independently with the same key. Identical plaintext blocks → identical ciphertext blocks. This is catastrophically insecure for structured data — the "ECB penguin" (an image of Tux encrypted with ECB still shows the penguin outline) is the canonical demonstration. ECB also enables block-substitution attacks where an adversary rearranges blocks. Never use ECB for any real data.
14
CBC IV
In CBC mode, what is the primary purpose of the Initialization Vector (IV)?
aThe IV is XORed with the key before each block encryption to vary the round keys.
bThe IV XORs with the first plaintext block to randomize the first ciphertext, ensuring two identical plaintexts produce different ciphertexts.
cThe IV must match the last ciphertext block for decryption to authenticate correctly.
dThe IV determines the number of rounds the cipher executes.
Explanation
✓ Correct: (b)
CBC mode: \(C_i = E_K(P_i \oplus C_{i-1})\), with \(C_0 = \text{IV}\). The IV provides a random starting value for the XOR chain. Without a fresh, random IV, two messages beginning with the same plaintext would produce identical initial ciphertext — leaking that the messages share a prefix. The IV need not be secret but must be unpredictable (to resist chosen-IV attacks, as in BEAST).

CTR Mode
A key advantage of CTR (Counter) mode over CBC is:
aCTR provides built-in message authentication that CBC lacks.
bEncryption and decryption in CTR are the same operation (XOR with keystream), the cipher's inverse is never needed, and blocks can be processed in parallel.
cCTR automatically pads messages that are not a multiple of the block size.
dCTR mode eliminates the need for an IV entirely.
Explanation
✓ Correct: (b)
CTR generates keystream blocks by encrypting \(\text{Nonce} \| \text{Counter}_i\). The plaintext is XORed with the keystream — encryption and decryption are identical. Advantages: (1) only the block cipher's encryption direction is used; (2) all blocks are independent → full parallelism; (3) random-access decryption of any block; (4) works on arbitrary-length messages (no padding required). CTR is essentially a stream cipher built from a block cipher.

Kerberos AS
In Kerberos, the Authentication Server (AS) is responsible for:
aIssuing service tickets for every individual application server.
bVerifying the client's identity (via password-derived key) and issuing a Ticket-Granting Ticket (TGT) plus a session key for the TGS.
cActing as a man-in-the-middle to decrypt and re-encrypt all service communications.
dStoring all session keys used between clients and services indefinitely.
Explanation
✓ Correct: (b)
The Kerberos AS is the first contact in the authentication flow. The client sends its identity; the AS looks up the client's long-term key \(K_c\) (derived from the password). It replies with: (a) the TGT, encrypted under the TGS's long-term key \(K_{TGS}\) (the client cannot read it), and (b) a session key \(K_{c,TGS}\), encrypted under \(K_c\). The client decrypts to get \(K_{c,TGS}\) and uses the TGT when requesting service tickets from the TGS.

Needham-Schroeder Nonce
In the Needham-Schroeder protocol, why does the client include a nonce \(N_A\) in its request to the KDC?
aThe nonce becomes the session key shared between the two parties.
bThe KDC includes \(N_A\) in its encrypted response, proving the response is fresh and not a replay of a previous valid reply.
cThe nonce replaces timestamps, which Needham-Schroeder does not use.
dThe nonce is the client's public component in a key-exchange operation.
Explanation
✓ Correct: (b)
A nonce (number used once) is a random value chosen fresh for each protocol run. The client sends \(N_A\) to the KDC. The KDC includes \(N_A\) in its reply, encrypted under \(K_A\) (the client's master key). Only the genuine KDC can produce a correctly-encrypted response containing \(N_A\) — and since \(N_A\) was chosen just now, the response cannot be a replay of any previous exchange.

RC4
RC4's internal state consists of:
aA 128-bit shift register updated by a non-linear feedback function.
bA 256-byte permutation array \(S[0..255]\) and two indices \(i, j\), initialized by the key via the KSA.
cAn 8-byte state updated by a Feistel network with 16 rounds.
dA fixed 128-bit key expanded into a 2048-bit key schedule.
Explanation
✓ Correct: (b)
RC4 uses a 256-byte permutation \(S\) initialized to \([0,1,\ldots,255]\) and then scrambled by the Key Scheduling Algorithm (KSA) using the secret key. The PRGA outputs one keystream byte per step: increment \(i\), update \(j = (j + S[i]) \bmod 256\), swap \(S[i] \leftrightarrow S[j]\), output \(S[(S[i]+S[j]) \bmod 256]\). Known weakness: the first bytes of the keystream are biased and should be discarded (as in WEP's failure).

Kerberos SSO
Kerberos achieves Single Sign-On (SSO) because:
aThe user's password is embedded in each service ticket for re-authentication.
bAfter the initial password-based login, the TGT allows the client to obtain service tickets for any Kerberized service without re-entering the password.
cThe AS stores a persistent session and automatically authenticates all future requests.
dAll services share the same session key derived from the user's master key.
Explanation
✓ Correct: (b)
After the user enters their password once (to decrypt the TGT), the client caches the TGT (valid for ~8 hours). To access any Kerberized service, the client presents the TGT to the TGS and receives a service-specific ticket — no password re-entry. The TGT is opaque to the client (encrypted under the TGS's key), and each service ticket is opaque to the client (encrypted under the service's key), preventing forgery.

Needham-Schroeder Replay
The classic replay vulnerability in the original Needham-Schroeder protocol allows:
aAn attacker to factor the master key from two intercepted sessions.
bAn attacker who previously obtained an old session key to replay the old ticket to B, tricking B into believing it is starting a fresh session with A.
cThe KDC to be bypassed entirely because nonces are predictable.
dAny passive observer to derive both parties' master keys from the transcript.
Explanation
✓ Correct: (b)
Denning & Sacco (1981): if Eve compromises an old session key \(K_{AB}\), she can replay the old \(E_{K_B}(K_{AB} \| A)\) ticket to B. B decrypts it, sees a valid session key for A, and believes a fresh session has been established — even though \(K_{AB}\) is known to Eve. The fix (used in Kerberos) is to include timestamps in tickets, letting B reject stale tickets outside an allowed window.
21
OFB Bit Errors
In OFB (Output Feedback) mode, a single-bit error in the transmitted ciphertext:
aCauses all subsequent blocks to be unrecoverable.
bCorrupts only the corresponding bit in the decrypted plaintext, with no propagation to other bits.
cCorrupts the entire current block and the next block.
dHas no effect because OFB generates keystream independently of the ciphertext.
Explanation
✓ Correct: (b)
OFB generates keystream by feeding the block cipher's output back into itself — entirely independent of the plaintext or ciphertext. Decryption XORs the received ciphertext with the same keystream. A single-bit flip in \(C_i\) XORs just that one bit with the correct keystream bit, flipping exactly one plaintext bit. The keystream generator remains synchronized, so future blocks are unaffected. This makes OFB suitable for noisy channels.
22
CFB Error Propagation
In CFB (Cipher Feedback) mode, a 1-bit error in ciphertext block \(C_i\) causes:
aNo error — CFB self-synchronizes and corrects errors automatically.
bA 1-bit error in \(P_i\) and exactly one additional corrupted bit somewhere in \(P_{i+1}\), after which synchronization resumes.
cAll blocks from \(P_i\) onward are completely garbled.
dOnly \(P_i\) is corrupted; \(P_{i+1}\) is unaffected.
Explanation
✓ Correct: (b)
CFB feeds the ciphertext into the shift register used for the next keystream block. A 1-bit error in \(C_i\): (1) the XOR in decrypting \(P_i\) directly flips 1 bit of \(P_i\); (2) the corrupted \(C_i\) is fed into the shift register, producing a garbled keystream input for block \(i+1\), corrupting exactly one output bit in \(P_{i+1}\). After block \(i+1\), the shift register contains only correct \(C\) values again and synchronization is restored.

Euler's Theorem
RSA's correctness relies on Euler's theorem. Which statement correctly applies it?
aEuler's theorem proves that all keys generated by RSA are distinct.
bSince \(ed \equiv 1 \pmod{\phi(n)}\), we have \(m^{ed} = m^{1+k\phi(n)} \equiv m \pmod{n}\), so decryption inverts encryption.
cEuler's theorem shows that RSA encryption is a one-way function.
dIt proves that key generation terminates in polynomial time.
Explanation
✓ Correct: (b)
Euler's theorem: for \(m\) coprime to \(n\), \(m^{\phi(n)} \equiv 1 \pmod{n}\). Since \(ed \equiv 1 \pmod{\phi(n)}\), write \(ed = 1 + k\phi(n)\). Then \(m^{ed} = m \cdot (m^{\phi(n)})^k \equiv m \cdot 1^k = m \pmod{n}\). This guarantees that applying \(d\) to \(m^e\) recovers \(m\) — RSA decryption is the correct inverse of RSA encryption.

Private Key
In RSA, the private key \(d\) is computed as:
a\(d = e^2 \bmod \phi(n)\)
b\(d = e \bmod (p + q)\)
c\(d \equiv e^{-1} \pmod{\phi(n)}\), i.e., \(e \cdot d \equiv 1 \pmod{\phi(n)}\)
d\(d = \phi(n) - e\)
Explanation
✓ Correct: (c)
\(d\) is the modular multiplicative inverse of \(e\) modulo \(\phi(n)\). The extended Euclidean algorithm computes \(d\) such that \(e \cdot d \equiv 1 \pmod{\phi(n)}\). This inverse exists because \(\gcd(e, \phi(n)) = 1\) was required when choosing \(e\). Knowing \(\phi(n)\) is what allows computing \(d\) — which is why an attacker who can factor \(n\) into \(p, q\) and compute \(\phi(n) = (p-1)(q-1)\) breaks RSA.

Factoring Hardness
RSA's security is based on the integer factorization problem. What makes factoring the modulus \(n\) hard in practice?
aRSA moduli are kept secret so the attacker does not know \(n\).
b\(n\) is a product of two large secret primes; no polynomial-time factoring algorithm is known, and sub-exponential algorithms require impractical computational resources for \(n \geq 2048\) bits.
cFactoring \(n\) is computationally easy, but deriving \(d\) from \(e\) is hard even given the factorization.
dThe RSA standard mandates 64-bit primes, making factoring provably impossible.
Explanation
✓ Correct: (b)
The best known factoring algorithm, the General Number Field Sieve (GNFS), runs in sub-exponential time but is still impractical for \(n \geq 2048\) bits. Note: if factoring were easy, RSA would be completely broken — knowing \(p, q\) immediately gives \(\phi(n)\) and thus \(d\). Also note: no proof exists that factoring is hard; it is an unproven computational assumption, which is why post-quantum alternatives are being developed.

Square-and-Multiply
The square-and-multiply algorithm computes \(m^e \bmod n\) by:
aPrecomputing all powers \(m^1, m^2, m^4, \ldots\) and selecting the relevant subset to multiply.
bReducing \(m^e \bmod n\) by GCD tricks before exponentiation.
cProcessing the binary representation of \(e\) from MSB to LSB: always square, and additionally multiply by \(m\) for each 1-bit.
dSplitting the exponent using the Chinese Remainder Theorem into two half-size problems.
Explanation
✓ Correct: (c)
Algorithm: set \(\text{result} = 1\). Read \(e\)'s binary digits from MSB to LSB. For each bit: (1) always square: \(\text{result} \leftarrow \text{result}^2 \bmod n\); (2) if the bit is 1: multiply: \(\text{result} \leftarrow \text{result} \times m \bmod n\). Total operations: \(\leq 2\lfloor\log_2 e\rfloor\). For a 2048-bit exponent, only ~3000 multiplications suffice instead of \(2^{2048}-1\).

RSA-CRT
How does the Chinese Remainder Theorem (CRT) speed up RSA decryption?
aCRT allows using a shorter private key \(d\) without reducing security.
bDecryption is split into two exponentiations modulo \(p\) and \(q\) (each half the size of \(n\)), which are ~4× faster each, then combined via Garner's formula.
cCRT replaces modular exponentiation with polynomial division.
dCRT applies to encryption only; decryption still requires full exponentiation mod \(n\).
Explanation
✓ Correct: (b)
CRT approach: compute \(m_p = c^{d_p} \bmod p\) where \(d_p = d \bmod (p-1)\), and \(m_q = c^{d_q} \bmod q\) where \(d_q = d \bmod (q-1)\). Since \(p, q \approx n^{1/2}\), each exponentiation is roughly \((1/2)^3 = 1/8\) the cost of the full exponentiation (exponent is half as long, modulus is half as large). Combining the two results via Garner's formula costs little. Net speedup: approximately 4×.

Small Exponent Attack
The small public exponent attack on RSA (e.g., \(e = 3\)) works when:
aThe same message \(m\) is sent unpadded to 3 different recipients with \(e_1 = e_2 = e_3 = 3\), allowing recovery of \(m^3\) via CRT and then taking the cube root over the integers.
bUsing \(e = 3\) means \(d\) is also small, making the private key easy to factor.
cSetting \(e = 3\) always results in \(m^3 < n\), so reduction mod \(n\) never occurs, making the ciphertext equal to \(m^3\) in the integers.
dA small \(e\) reduces the number of valid plaintexts to only 256 values.
Explanation
✓ Correct: (a)
If message \(m\) is sent to three recipients with moduli \(n_1, n_2, n_3\) and all use \(e=3\), the attacker has \(c_i = m^3 \bmod n_i\). By CRT, compute \(X \equiv c_i \pmod{n_i}\) for \(i=1,2,3\): since \(m < \min(n_i)\), we have \(m^3 < n_1 n_2 n_3\), so \(X = m^3\) exactly (no modular reduction). Taking the integer cube root gives \(m\). Fix: always use OAEP padding before RSA encryption.

Non-Repudiation
RSA provides non-repudiation in digital signatures because:
aRSA encrypts messages with the recipient's public key, proving the sender's identity.
bThe sender signs with their private key \(d\) (known only to them); anyone can verify with the public key \(e\), so only the private key holder could have produced the valid signature.
cNon-repudiation requires both parties to share the same symmetric key.
dRSA non-repudiation relies on the CA to confirm message authenticity after the fact.
Explanation
✓ Correct: (b)
Non-repudiation: Alice computes \(s = m^d \bmod n\). Anyone can verify: \(s^e \bmod n = m\). Since only Alice holds \(d\), only Alice can produce a valid signature. She cannot later claim someone else signed — this is non-repudiation. Compare with MACs (shared key): both parties can compute a valid MAC, so no non-repudiation. Digital signatures require asymmetric keys.

Hybrid Encryption
Why is hybrid encryption (RSA + AES) used instead of pure RSA for bulk data?
aRSA is faster for large data; AES is used only for the key exchange.
bRSA is orders of magnitude slower than AES for bulk data; a random AES session key is encrypted with RSA (small), and the bulk data is encrypted with AES (fast).
cThe combination of RSA and AES provides quantum resistance.
dHybrid encryption is mandated by law for commercial communications.
Explanation
✓ Correct: (b)
RSA operations involve modular exponentiations with thousands-of-bit numbers — typically 1000–10,000× slower than AES. For a 1 GB file: RSA encryption would be impractical. The hybrid solution: (1) generate a fresh random 256-bit AES session key, (2) encrypt the file with AES-256 (fast), (3) encrypt only the 32-byte session key with RSA (tiny). The recipient decrypts the RSA-wrapped key then uses it to decrypt the file. This is how TLS and PGP work.

CCA on Textbook RSA
Textbook RSA (without padding) is malleable. A chosen-ciphertext attack exploits this by:
aMultiplying a target ciphertext \(c = m^e\) by a known \(r^e\) to get \((mr)^e\), submitting it for decryption to obtain \(mr\), then dividing by \(r\) to recover \(m\).
bCCAs only work against stream ciphers, not RSA.
cRSA is immune to CCAs because private key operations are irreversible.
dA CCA requires the attacker to know the plaintext before encryption.
Explanation
✓ Correct: (a)
Textbook RSA is multiplicatively homomorphic: \(E(m_1) \cdot E(m_2) = E(m_1 m_2)\). An attacker who wants to decrypt \(c = m^e\) picks any \(r\), computes \(c' = c \cdot r^e = (mr)^e\), and submits \(c'\) to the decryption oracle. The oracle returns \(mr\). The attacker divides by \(r\) to get \(m\). OAEP padding destroys this malleability by randomizing the plaintext before encryption, making RSA-OAEP CCA-secure.

Miller-Rabin
The Miller-Rabin primality test is:
aA deterministic test that proves a number is prime with certainty.
bA deterministic polynomial-time primality test.
cA probabilistic test where each round with a random witness reduces the false-positive probability by at least a factor of 4; it can prove compositeness but not primality.
dA test that factors \(n\) to verify no prime divides it.
Explanation
✓ Correct: (c)
Miller-Rabin with witness \(a\): if the test says "composite," the number is definitely composite. If it says "probably prime," there is a ≤1/4 chance of error. After \(k\) independent rounds with different witnesses, the error probability is ≤\((1/4)^k\). For \(k=40\), this is negligible (\(\approx 10^{-24}\)). Note: AKS (2002) is a deterministic polynomial-time test but is slower; Miller-Rabin with 40 rounds is the practical standard for RSA key generation.

Digital Certificate
The core purpose of a digital certificate is to:
aEncrypt messages using the certificate owner's public key automatically.
bBind a public key to an identity, with the issuing CA's digital signature as the proof of that binding.
cStore the certificate owner's private key in an encrypted form.
dProvide a list of all currently revoked public keys.
Explanation
✓ Correct: (b)
A digital certificate is a signed data structure: subject identity + subject's public key + validity dates + issuer + CA's digital signature over all the above. Any party that trusts the CA can verify the CA's signature and thereby trust that the included public key belongs to the claimed identity. This solves the key distribution problem — you don't need a secure channel to receive someone's public key if you can verify their certificate.

Certificate Chain
When validating an X.509 certificate chain, the validator:
aContacts the certificate issuer directly for each validation request.
bVerifies each certificate's signature using its issuer's public key, walking up the chain until reaching a locally trusted root CA.
cValidates only the end-entity certificate; intermediate CAs are trusted implicitly.
dA chain with more than two certificates is automatically rejected for security.
Explanation
✓ Correct: (b)
Chain validation: start from the end-entity cert → verify its signature using the issuer's public key (from the next certificate in the chain) → continue until reaching a root CA whose certificate is in the local trust store (pre-installed in browsers/OS). Each link must have: (1) valid cryptographic signature, (2) validity period not expired, (3) appropriate key usage. If any link fails, the chain is invalid.

CRL
A Certificate Revocation List (CRL) is:
aA list of all valid certificates currently issued by a CA.
bA CA-signed list of certificate serial numbers that have been revoked before their expiration date.
cA database of users who have forgotten their passwords.
dThe set of root CA certificates pre-installed in a browser's trust store.
Explanation
✓ Correct: (b)
A certificate must be revocable before expiry when the private key is compromised or the owner changes status. The CA publishes a signed CRL listing revoked serial numbers with revocation reasons and dates. Limitations: CRLs can be large, grow over time, and must be periodically downloaded (introducing latency and staleness). OCSP (Online Certificate Status Protocol) is a real-time alternative that checks the status of a specific certificate by serial number.

DH Shared Secret
In a Diffie-Hellman exchange with public parameters \((g, p)\), Alice sends \(g^a \bmod p\) and Bob sends \(g^b \bmod p\). The shared secret is:
a\(g^{a+b} \bmod p\)
b\(g^{ab} \bmod p\)
c\(g^{a \oplus b} \bmod p\)
d\((g^a + g^b) \bmod p\)
Explanation
✓ Correct: (b)
Alice computes \((g^b)^a = g^{ab} \bmod p\). Bob computes \((g^a)^b = g^{ab} \bmod p\). Both arrive at \(g^{ab} \bmod p\). An eavesdropper sees \(g^a\) and \(g^b\) but cannot compute \(g^{ab}\) without solving the Computational Diffie-Hellman (CDH) problem — which is as hard as the discrete logarithm problem (DLP) in \(\mathbb{Z}_p^*\).
37
DH MITM
The fundamental reason DH is vulnerable to man-in-the-middle attacks is:
aDH uses the same key for encryption and authentication, creating a circular dependency.
bDH provides no authentication — Eve can substitute her own values for both Alice's and Bob's, establishing shared secrets with each without either party knowing.
cThe discrete logarithm problem was recently solved, breaking DH's security.
dDH is vulnerable because all deployments use the same prime \(p\).
Explanation
✓ Correct: (b)
In unauthenticated DH: Eve intercepts Alice's \(g^a\), sends her own \(g^e\) to Bob. She intercepts Bob's \(g^b\), sends her own \(g^{e'}\) to Alice. Alice derives shared secret \(g^{ae'}\) (with Eve), Bob derives \(g^{be}\) (with Eve). Eve decrypts, reads, re-encrypts, and forwards — a transparent relay. Neither Alice nor Bob detects Eve. Fix: authenticate the DH values using certificates or pre-shared keys (as in TLS, where the server's certificate proves ownership of its DH public key).
38
Perfect Forward Secrecy
Perfect Forward Secrecy (PFS) means:
aThe same session key is reused across sessions to provide consistent security.
bEven if the long-term private key is later compromised, past session keys cannot be derived from it, protecting previously recorded sessions.
cPFS is achieved by switching to a new symmetric algorithm for each session.
dPFS requires the receiver to delete their private key after each communication session.
Explanation
✓ Correct: (b)
PFS is achieved using ephemeral key pairs (DHE/ECDHE): a fresh random DH key pair is generated for each session and discarded after use. The session key derives entirely from the ephemeral exchange; the long-term private key is used only for authentication (to prevent MITM). If the long-term key is leaked years later, an adversary who recorded old sessions still cannot decrypt them — the ephemeral session keys are gone. TLS 1.3 mandates PFS.

ElGamal Nonce Reuse
In ElGamal encryption or DSS, reusing the random nonce \(K\) for two different messages allows an attacker to:
aIn encryption: divide the two ciphertexts to cancel the keystream, recovering the ratio of plaintexts. In DSS: solve a linear equation to recover the private key \(x\).
bNonce reuse only increases the ciphertext expansion factor but does not break security.
cNonce reuse makes ElGamal equivalent to RSA encryption.
dNonce reuse only affects the DSS variant, not ElGamal encryption.
Explanation
✓ Correct: (a)
Encryption nonce reuse: both ciphertexts share the same \(C_1 = g^K\) and keystream \(y^K\). The second components are \(C_2 = m_1 \cdot y^K\) and \(C_2' = m_2 \cdot y^K\). Their ratio is \(C_2 / C_2' = m_1/m_2\) — if either message is known, the other is recovered. DSS nonce reuse is even worse: from two signatures with the same \(K\), one can solve \(K = (m_1 - m_2)(s_1 - s_2)^{-1} \bmod (p-1)\) and then recover \(x = r^{-1}(Ks - m) \bmod (p-1)\). Sony's PS3 private key was recovered exactly this way.
40
ElGamal Ciphertext Size
Compared to the plaintext, ElGamal encryption produces a ciphertext that is:
aThe same size as the plaintext.
bApproximately twice the size, because the ciphertext is a pair \((C_1, C_2)\), each the size of the group modulus \(p\).
cExactly 256 bits regardless of plaintext length.
d256 bits shorter than the RSA equivalent for the same security level.
Explanation
✓ Correct: (b)
ElGamal encryption: \(C_1 = g^K \bmod p\) and \(C_2 = m \cdot y^K \bmod p\). Both \(C_1\) and \(C_2\) are elements of \(\mathbb{Z}_p^*\), so each has the same bit-length as \(p\). Encrypting a message \(m\) that is also \(|p|\) bits yields a ciphertext pair of \(2|p|\) bits — double the size. This 2× expansion is a notable disadvantage of ElGamal compared to RSA, which does not inherently expand the message size.
41
Logjam Attack
The Logjam attack (2015) against TLS/DH exploited:
aA flaw in RSA key generation that allows factoring 1024-bit moduli.
bThe ability to downgrade TLS connections to use 512-bit "export-grade" DH primes, for which discrete logarithms had been precomputed offline.
cNonce reuse in ElGamal to recover TLS session keys.
dA birthday-paradox collision attack against SHA-1 certificates.
Explanation
✓ Correct: (b)
Logjam: a MITM attacker intercepts the TLS handshake and changes the offered DH group to a 512-bit "export" prime. Many servers accepted this downgrade (legacy from 1990s US export restrictions). Because many TLS implementations reused a small set of standard 512-bit primes, researchers precomputed discrete log tables for those primes. Once the discrete log of \(g^a\) was computed, the attacker derived the shared secret and read all traffic in near-real-time. Fix: disable export cipher suites, use ≥2048-bit primes.
42
ElGamal DSS Verification
In ElGamal DSS, given public key \(y = g^x \bmod p\), signature \((r, s)\) on message \(m\), the verification condition is:
a\(r^s \equiv g^m \pmod{p}\)
b\(g^m \equiv y^r \cdot r^s \pmod{p}\)
c\(s \equiv m \cdot r^{-1} \pmod{p-1}\)
d\(m \equiv r^y \cdot s^g \pmod{p}\)
Explanation
✓ Correct: (b)
Signing: choose fresh random \(K\), compute \(r = g^K \bmod p\), then \(s = K^{-1}(m - xr) \bmod (p-1)\). Verification: check \(g^m \equiv y^r \cdot r^s \pmod{p}\). This works because \(y^r \cdot r^s = g^{xr} \cdot g^{Ks} = g^{xr + Ks} = g^{xr + (m-xr)} = g^m\). Each signature requires a fresh, secret, random \(K\) — reuse immediately compromises the private key \(x\).
🔏 Lecture 05: Cryptographically Secure Hashing Q43–Q52
43
Pre-Image Resistance
First pre-image resistance of a hash function means:
aGiven a hash output \(h\), it is computationally infeasible to find any message \(m\) such that \(H(m) = h\).
bGiven a message \(m_1\), it is infeasible to find \(m_2 \neq m_1\) with \(H(m_1) = H(m_2)\).
cIt is computationally infeasible to find any two distinct messages with the same hash.
dThe hash function must complete in constant time for all input lengths.
Explanation
✓ Correct: (a)
Pre-image resistance (one-wayness): the hash should be a one-way function. Given \(h\), finding any \(m\) such that \(H(m) = h\) requires \(O(2^n)\) random trials for an \(n\)-bit hash — brute force. This is distinct from (b), which is second pre-image resistance (given a specific \(m_1\), find another), and (c), which is collision resistance (find any pair). Pre-image attacks require inverting the hash, which is the hardest property to break.
44
Second Pre-Image
Second pre-image resistance (weak collision resistance) means:
aGiven \(h\), it is infeasible to find any \(m\) with \(H(m) = h\).
bGiven a specific message \(m_1\), it is infeasible to find a different message \(m_2 \neq m_1\) such that \(H(m_1) = H(m_2)\).
cIt is infeasible to find any two messages with the same hash, regardless of starting point.
dThe hash is resistant to length extension attacks.
Explanation
✓ Correct: (b)
Second pre-image resistance (also called weak collision resistance): you are handed a specific \(m_1\) and must find a collision for it specifically. This is harder than a general collision attack (where you choose both messages freely) because your search is constrained. An \(n\)-bit hash provides \(2^n\) security against second pre-image attacks. This property is critical for document signing: an attacker should not be able to substitute a different document that hashes to the same value as a legitimately signed one.
45
Birthday Attack
The birthday attack reduces collision finding from \(O(2^n)\) to \(O(2^{n/2})\) because:
aThe attacker knows part of the key, halving the search space.
bThe attacker is free to choose both colliding messages, exploiting the birthday paradox: among \(\approx 2^{n/2}\) random messages, two will share a hash value with probability \(\approx 50\%\).
cMerkle-Damgård hashes have a structural weakness that allows collision chains.
dCollision resistance is only required at the \(n/2\)-bit level by the hash standard.
Explanation
✓ Correct: (b)
Birthday paradox: in a room of \(\approx\sqrt{2N}\) people, the probability of any two sharing a birthday exceeds 50%. For an \(n\)-bit hash with \(N = 2^n\) possible outputs, after generating \(\approx 2^{n/2}\) random hash values, a collision appears with probability ≈ 50%. This is why SHA-256 (256-bit output) offers only 128-bit collision security — despite 256-bit output. For a secure hash, you need the output to be at least twice your target security level in bits.
46
Merkle-Damgård
The Merkle-Damgård construction builds a variable-length hash function from a fixed-input compression function by:
aHashing all message blocks in parallel and XORing the outputs.
bPadding the message, dividing it into fixed-size blocks, and iterating: \(H_i = f(H_{i-1}, M_i)\) starting from a fixed IV, then outputting the final \(H_k\).
cHashing only the message length and combining with a MAC.
dHashing each message bit independently and concatenating the results.
Explanation
✓ Correct: (b)
Merkle-Damgård: pad \(M\) to a multiple of the block size (padding typically includes the message length — Merkle-Damgård strengthening). Split into blocks \(M_1, \ldots, M_k\). Iterate: \(H_0 = \text{IV}\), \(H_i = f(H_{i-1}, M_i)\) for the compression function \(f\). Output \(H_k\). Security reduction: if \(f\) is a collision-resistant compression function, the resulting hash is also collision resistant. Vulnerability: length extension attacks exploit the exposed final state \(H_k\).
47
HMAC
The HMAC construction is defined as:
a\(\text{HMAC}(K, m) = H(K \| m)\)
b\(\text{HMAC}(K, m) = H\bigl((K \oplus \textit{opad}) \| H((K \oplus \textit{ipad}) \| m)\bigr)\)
c\(\text{HMAC}(K, m) = H(m) \oplus K\) truncated to the hash output length
d\(\text{HMAC}(K, m) = H(K) \oplus H(m)\)
Explanation
✓ Correct: (b)
HMAC uses the key twice with different paddings: ipad = 0x36 repeated to the block size, opad = 0x5C repeated. The inner hash \(H((K \oplus \text{ipad}) \| m)\) binds the message to the key. The outer hash \(H((K \oplus \text{opad}) \| \text{inner})\) prevents length extension attacks against the inner hash. Options (a) and (c) are insecure: (a) allows length extension, (c) is not a proper MAC. HMAC is provably secure if the underlying hash satisfies mild assumptions.
48
Length Extension Attack
A length extension attack against \(H(K \| m)\) allows an attacker to:
aCompute \(H(K \| m \| \textit{pad} \| \textit{extra})\) for any chosen extra without knowing \(K\), using only \(H(K \| m)\).
bExtend the key \(K\) to increase the MAC's security.
cRecover \(K\) from any single MAC output in linear time.
dCompute \(H(K \| m)\) from \(H(m)\) alone.
Explanation
✓ Correct: (a)
In Merkle-Damgård, the final hash state \(H_k\) is the complete internal state after processing \(K \| m\). An attacker who knows \(H_k = H(K \| m)\) can set it as the starting state and continue hashing more data \((\text{extra})\) — producing a valid \(H(K \| m \| \text{padding} \| \text{extra})\) without ever knowing \(K\). This allows forging MACs for extended messages. HMAC's double-keyed construction prevents this because the outer hash hides the inner state behind an additional keyed application.
49
SHA-256 Output
SHA-256 produces a digest of:
a128 bits
b160 bits (SHA-1)
c256 bits
d512 bits (SHA-512)
Explanation
✓ Correct: (c)
SHA-256 produces a 256-bit (32-byte) digest, giving 128-bit collision security (birthday bound). SHA family: SHA-1 → 160 bits (deprecated for most uses), SHA-224 → 224 bits, SHA-256 → 256 bits, SHA-384 → 384 bits, SHA-512 → 512 bits. SHA-3 (Keccak, 2015 NIST standard) uses a fundamentally different sponge construction but also supports 256-bit output (SHA3-256).
50
MAC vs Signature
The key difference between a MAC and a digital signature is:
aMACs use asymmetric keys; digital signatures use symmetric keys.
bDigital signatures provide non-repudiation (only the private key holder could sign); MACs do not, because the MAC secret is shared between parties.
cMACs are based on public-key cryptography; signatures use symmetric keys.
dDigital signatures authenticate only the key, not the message content.
Explanation
✓ Correct: (b)
A MAC uses a shared secret key — both the sender and receiver know \(K\) and can compute a valid MAC. A third party cannot determine which party produced the MAC, so there is no non-repudiation. A digital signature uses the signer's private key, which only they possess. Any party with the public key can verify the signature, but only the private key holder could have created it — providing non-repudiation. This is why legal contracts use digital signatures, not MACs.
51
HMAC Security
HMAC guarantees:
aMessage confidentiality alongside integrity.
bMessage authentication and integrity: only the party knowing the secret key \(K\) can compute or verify the tag; any message modification changes the tag.
cEquivalent security to encrypting the message with AES-256.
dNon-repudiation equivalent to an RSA digital signature.
Explanation
✓ Correct: (b)
HMAC provides: (1) integrity — any change to \(m\) produces a different tag; (2) authentication — only parties sharing \(K\) can produce or verify a valid tag. HMAC does NOT provide confidentiality (the message \(m\) is transmitted in clear) nor non-repudiation (the key is shared). In TLS, HMAC is used in the record layer to verify that messages haven't been tampered with in transit, typically combined with encryption for confidentiality.
52
SHA-256 vs MD5
Why should SHA-256 be used instead of MD5 for security-critical applications?
aSHA-256 is always faster than MD5 for all input sizes.
bPractical collisions in MD5 have been demonstrated; its 128-bit output provides only \(2^{64}\) collision security, which has been broken in practice.
cMD5 pads its output to 256 bits internally, creating a structural weakness.
dSHA-256 does not use Merkle-Damgård, making it fundamentally more secure than MD5.
Explanation
✓ Correct: (b)
MD5 produces 128-bit digests → \(2^{64}\) birthday-attack collision complexity. Wang et al. (2004) demonstrated practical collisions in MD5 in minutes; this was exploited to forge SSL certificate chains (2009 rogue CA attack). SHA-256 provides 256-bit output → \(2^{128}\) collision complexity, currently considered computationally infeasible. Note: both use Merkle-Damgård (so both are vulnerable to length extension); use HMAC-SHA-256 rather than raw hash prepended with a key for MAC construction.

53
TRNG vs PRNG
The fundamental difference between a TRNG and a PRNG is:
aTRNGs are faster software algorithms; PRNGs rely on physical phenomena.
bPRNGs are deterministic algorithms — given the same seed, they produce the same sequence; TRNGs draw entropy from unpredictable physical processes.
cTRNGs are faster but less random than PRNGs.
dPRNGs cannot be seeded and produce independent values each run.
Explanation
✓ Correct: (b)
A TRNG derives randomness from physical entropy sources (thermal noise in resistors, shot noise in semiconductors, radioactive decay, photon arrival times). Its output is non-deterministic and cannot be reproduced. A PRNG is a deterministic algorithm: \(x_{n+1} = f(x_n)\). The same seed always produces the same sequence. A CSPRNG (Cryptographically Secure PRNG) is a PRNG that passes the next-bit test: given any prefix of the output, the next bit is computationally indistinguishable from random.
54
LCG Weakness
Linear Congruential Generators (LCGs) are unsuitable for cryptography because:
aLCGs generate numbers too slowly for any practical use.
bLCGs are fully deterministic; observing a few outputs allows solving for the state and predicting all future (and past) values.
cLCGs require a hardware entropy source to produce any output.
dLCGs are cryptographically secure but unsuitable for simulation.
Explanation
✓ Correct: (b)
LCG recurrence: \(X_{n+1} = (aX_n + c) \bmod m\). Given even three consecutive outputs, simple linear algebra recovers all parameters \((a, c, m)\) if unknown — or directly recovers the seed if the parameters are public. Once the state is known, all future and past values are determined. LCGs are fine for simulations and random number quality tests, but are cryptographically broken: never use for key generation, nonces, or anything security-sensitive.
55
Blum Blum Shub
The Blum Blum Shub (BBS) generator's security is based on:
aThe hardness of computing discrete logarithms modulo a prime.
bThe hardness of factoring \(n = p \cdot q\), where \(p \equiv q \equiv 3 \pmod{4}\) are both prime (a Blum integer).
cBBS is unconditionally secure and requires no hardness assumption.
dThe hardness of the shortest vector problem in lattices.
Explanation
✓ Correct: (b)
BBS: choose Blum primes \(p \equiv q \equiv 3 \pmod{4}\), set \(n = pq\). Pick seed \(x_0\) as a random quadratic residue mod \(n\). Recurrence: \(x_i = x_{i-1}^2 \bmod n\), output the LSB(s) of each \(x_i\). Security reduces to the Quadratic Residuosity Problem, which is computationally equivalent to factoring \(n\). This makes BBS provably secure under the factoring assumption — but it is also extremely slow compared to AES-CTR-based PRNGs, limiting its practical use.
56
/dev/random vs /dev/urandom
On Linux, the difference between /dev/random and /dev/urandom is:
a/dev/random always produces strong output; /dev/urandom produces weak pseudo-random output.
b/dev/random blocks when the entropy pool is exhausted; /dev/urandom never blocks — it continues using a CSPRNG seeded from available entropy.
c/dev/urandom uses network traffic as entropy; /dev/random uses keyboard and mouse input only.
dBoth are identical on modern kernels and produce the same output.
Explanation
✓ Correct: (b)
Linux collects entropy from hardware events (keyboard/mouse timing, disk seeks, network arrival times). /dev/random is conservative: it blocks (stalls the read) when the estimated entropy count falls too low. /dev/urandom uses a CSPRNG (ChaCha20 on recent kernels) to stretch available entropy and never blocks. Cryptographers broadly recommend /dev/urandom for normal use — blocking is rarely necessary once the pool has been initially seeded at boot.
57
Shannon Entropy
The Shannon entropy of a fair coin flip (two equally likely outcomes) is:
a0 bits (coin flips are deterministic in quantum mechanics)
b1 bit
c2 bits
d\(\log_2(n)\) bits where \(n\) is the number of flips performed
Explanation
✓ Correct: (b)
Shannon entropy: \(H = -\sum_x p(x) \log_2 p(x)\). For a fair coin: \(p(H) = p(T) = 0.5\). \[H = -(0.5 \cdot \log_2 0.5 + 0.5 \cdot \log_2 0.5) = -(0.5 \cdot (-1) + 0.5 \cdot (-1)) = 1 \text{ bit}\] Each fair coin flip provides exactly 1 bit of entropy. A biased coin (e.g., \(p=0.9\)) has lower entropy (\(\approx 0.47\) bits) because the outcome is more predictable — less information is gained per flip.
58
ANSI X9.17
ANSI X9.17 generates pseudo-random keys by:
aApplying a simple LCG seeded with the current system time.
bUsing 3DES: the output block \(R_i = \text{3DES}_K(\text{DT}_i \oplus V_i)\), and the next seed \(V_{i+1} = \text{3DES}_K(R_i \oplus \text{DT}_i)\), where \(\text{DT}_i\) is a timestamp and \(V_i\) is the running seed.
cHashing the concatenation of message data and a counter using SHA-256.
dUsing BBS to extract LSBs of a Blum-integer squaring sequence.
Explanation
✓ Correct: (b)
ANSI X9.17 (and its successor X9.31): uses 3DES with a secret key \(K\). Per iteration: (1) compute the output \(R_i = \text{EDE}_K(\text{DT}_i \oplus V_i)\) where \(\text{DT}_i\) is the current timestamp and \(V_i\) is the current seed/state; (2) update the seed: \(V_{i+1} = \text{EDE}_K(R_i \oplus \text{DT}_i)\). The timestamp prevents state repetition; the chained seed maintains unpredictability. X9.31 is the AES-based successor.
59
Intel RDRAND
The Intel RDRAND instruction provides cryptographically random numbers by:
aReading bytes from the OS's /dev/urandom via a system call.
bFeeding on-chip thermal noise entropy into a DRBG (AES-CTR-based Deterministic Random Bit Generator) that produces the output.
cHashing the CPU cycle counter (RDTSC) to produce an unpredictable value.
dReturning a fixed pseudo-random sequence updated only at system power-on.
Explanation
✓ Correct: (b)
Intel's Digital Random Number Generator (DRNG): on-chip thermal noise is harvested via special analog circuits (the entropy source). This raw entropy seeds an SP 800-90A AES-CTR DRBG. RDRAND returns the DRBG output — high quality and fast (hundreds of MB/s). RDSEED returns the raw conditioned entropy before the DRBG, which is slower but provides entropy for seeding other generators. Intel's implementation is a black box, which has led to concerns about possible backdoors — production systems typically mix RDRAND with OS-collected entropy.

CSPRNG Requirements
A CSPRNG (Cryptographically Secure PRNG) must satisfy the next-bit test, meaning:
aIt must produce exactly one new bit per clock cycle.
bGiven any number of past output bits, no polynomial-time algorithm can predict the next bit with probability significantly better than 1/2.
cThe output must pass all NIST statistical randomness tests with 99% confidence.
dIt must require a hardware entropy source to re-seed after every 1000 outputs.
Explanation
✓ Correct: (b)
The next-bit test (Yao, 1982) is the gold standard for CSPRNGs: given the first \(k\) bits of output, no polynomial-time algorithm can predict bit \(k+1\) with probability > 1/2 + negligible. This computational unpredictability is essential: cryptographic nonces, session keys, and IV values must be unpredictable. Statistical randomness tests (NIST, Diehard) are necessary but not sufficient — a PRNG can pass all statistical tests while still being predictable if its state is exposed. LCGs pass statistical tests but fail the next-bit test.
