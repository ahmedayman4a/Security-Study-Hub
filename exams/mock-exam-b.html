Monoalphabetic Cipher
Despite having a key space of \(26!\), the monoalphabetic substitution cipher is easily broken by:
aFrequency analysis — letter frequencies in the ciphertext mirror those in the language, revealing the substitution mapping.
bBrute-force search over all \(26!\) possible keys.
cMeet-in-the-middle attack exploiting the cipher's Feistel structure.
dDifferential cryptanalysis using chosen-plaintext pairs.
Explanation
✓ Correct: (a)
A monoalphabetic cipher applies a fixed permutation to every letter. Because each letter is always replaced by the same ciphertext letter, the frequency distribution is preserved. In English, 'e' is most frequent (~13%), 't' second (~9%), etc. By matching ciphertext letter frequencies to known English frequencies, the substitution is revealed without trying any keys. Key space size is irrelevant when the cipher leaks such strong statistical patterns.

Playfair Cipher
The Playfair cipher improves over monoalphabetic substitution by:
aUsing a longer key — up to 26 characters instead of one shift value.
bEncrypting each letter with a different Caesar shift derived from the key.
cEncrypting pairs of letters (digraphs) using a 5×5 key matrix, so single-letter frequency analysis no longer applies.
dApplying a transposition after substitution, making it a product cipher.
Explanation
✓ Correct: (c)
Playfair constructs a 5×5 matrix from a keyword (I/J share a cell). Each pair of plaintext letters (digraph) is encrypted by one of three rules based on their positions in the matrix. Because the unit of encryption is a digraph (26² = 676 combinations) rather than a single letter, simple single-letter frequency analysis fails. However, digraph frequency analysis with enough ciphertext still breaks it. The key insight: Playfair is a digraph substitution cipher.

Hill Cipher
The Hill cipher encrypts an \(m\)-letter block as \(\mathbf{c} = K\mathbf{p} \bmod 26\). Decryption requires:
aThe inverse of each element of \(K\) individually.
bThe matrix inverse \(K^{-1} \bmod 26\), which exists only if \(\gcd(\det(K), 26) = 1\).
cTransposing the key matrix \(K\).
dRe-applying the same matrix \(K\) a second time to the ciphertext.
Explanation
✓ Correct: (b)
Hill cipher decryption: \(\mathbf{p} = K^{-1}\mathbf{c} \bmod 26\). The matrix inverse mod 26 exists only when \(\gcd(\det(K), 26) = 1\) — equivalently, the determinant must be invertible modulo 26 (i.e., coprime to 26 = 2×13, so the determinant must be odd and not a multiple of 13). This is analogous to requiring a modular inverse: \(a^{-1} \bmod n\) exists iff \(\gcd(a,n)=1\). Vulnerability: Hill cipher is completely broken by a known-plaintext attack.

Transposition Cipher
A transposition cipher differs from a substitution cipher in that it:
aReplaces each plaintext letter with a different letter of the alphabet.
bXORs each byte with a keystream derived from the key.
cRearranges (permutes) the positions of plaintext characters without changing the characters themselves.
dDoubles the ciphertext length by inserting dummy characters at specified positions.
Explanation
✓ Correct: (c)
In a transposition cipher (e.g., columnar transposition), the plaintext characters are unchanged — only their positions are rearranged according to a key-defined permutation. Because the characters themselves are not altered, letter frequency analysis still applies to the ciphertext (the frequencies are identical to the plaintext). Combined with a substitution cipher (product cipher), transposition contributes diffusion while substitution provides confusion.

Feistel Structure
A key property of the Feistel structure is:
aThe round function \(F\) must be invertible for decryption to work.
bDecryption uses the exact same structure as encryption, just with round keys applied in reverse order — \(F\) does not need to be invertible.
cEach round doubles the block size to increase diffusion.
dThe Feistel structure requires a key that is exactly as long as the plaintext block.
Explanation
✓ Correct: (b)
Feistel round: split block into left half \(L\) and right half \(R\). Output: \(L' = R\), \(R' = L \oplus F(R, K_i)\). Decryption reverses this: \(R = L'\), \(L = R' \oplus F(L', K_i)\) — using the same \(F\) with the same key, just working backward. Because the XOR is self-inverse, decryption is just encryption with keys in reverse. This elegantly allows \(F\) to be any function — including non-invertible ones — without complicating decryption.

DES Parameters
DES (Data Encryption Standard) uses:
a128-bit block size, 128-bit key, 10 rounds.
b128-bit block size, 56-bit key, 16 rounds.
c64-bit block size, 56-bit effective key (from a 64-bit key input with 8 parity bits), 16 rounds.
d64-bit block size, 64-bit key, 32 rounds.
Explanation
✓ Correct: (c)
DES: 64-bit block, 16 rounds, key input is 64 bits but 8 bits are parity — the effective key is 56 bits, giving a key space of \(2^{56} \approx 7.2 \times 10^{16}\). In 1997, RSA's DES Challenge was broken by brute-force in 96 days. By 1999 the EFF's "Deep Crack" broke DES in under 22 hours. DES is completely insecure today — use AES (128-bit block, 128/192/256-bit key).

AES SubBytes
The AES SubBytes step achieves which design property?
aDiffusion — spreading plaintext bits across the entire state.
bConfusion — introducing non-linearity to obscure the relationship between key and ciphertext.
cKey expansion — deriving round subkeys from the main key.
dAuthentication — verifying that the state has not been tampered with.
Explanation
✓ Correct: (b)
SubBytes applies the AES S-box to every byte of the 4×4 state independently. The S-box is designed using two steps: (1) compute the multiplicative inverse in \(\text{GF}(2^8)\) (highly non-linear), then (2) apply an invertible affine transformation. This non-linearity is what provides confusion — making the algebraic relationship between key and ciphertext virtually impossible to exploit. ShiftRows and MixColumns then provide diffusion.

AddRoundKey
In AES, the AddRoundKey operation:
aAdds the round key to each state byte modulo 256.
bXORs each byte of the state with the corresponding byte of the round key derived from the key schedule.
cMultiplies each state column by the round key in \(\text{GF}(2^8)\).
dSubstitutes state bytes using the round key as an index into the S-box.
Explanation
✓ Correct: (b)
AddRoundKey is bitwise XOR between the 16-byte state and the 16-byte round key (a portion of the expanded key schedule). XOR is chosen because it is its own inverse: \(A \oplus K \oplus K = A\). This makes the decryption step identical to the encryption step. AddRoundKey is applied at the beginning of the cipher (before round 1), after every round (in all 10/12/14 rounds), and is the only step that directly involves the secret key.

GF(2⁸) Arithmetic
Why does AES use arithmetic in \(\text{GF}(2^8)\) rather than ordinary integer arithmetic?
aOrdinary integer arithmetic would require larger registers and is slower on 8-bit hardware.
bGF(2⁸) arithmetic automatically provides key-dependent round constants.
cGF(2⁸) ensures every byte operation is invertible (every non-zero element has a multiplicative inverse), enabling exact decryption with no rounding errors.
dGF(2⁸) doubles the diffusion compared to integer multiplication.
Explanation
✓ Correct: (c)
\(\text{GF}(2^8)\) is a finite field: every non-zero element has an exact multiplicative inverse, and all arithmetic stays within 8 bits. This is critical: MixColumns must be perfectly invertible (to allow decryption), and SubBytes uses the multiplicative inverse in \(\text{GF}(2^8)\). Ordinary integer multiplication mod 256 would not guarantee invertibility (e.g., \(2 \times 128 = 256 \equiv 0 \pmod{256}\) — zero has no inverse). The irreducible polynomial \(x^8 + x^4 + x^3 + x + 1\) defines the field reduction.

Ideal Block Cipher
An "ideal block cipher" with \(n\)-bit blocks would require a key that specifies:
aExactly \(n\) bits — one bit per plaintext bit.
b\(2n\) bits — one round key of \(n\) bits per encryption direction.
c\(n \cdot 2^n\) bits — enough to define an arbitrary permutation over all \(2^n\) possible input blocks.
d\(n^2\) bits — one bit per cell in the \(n \times n\) state matrix.
Explanation
✓ Correct: (c)
An ideal block cipher selects a random permutation from the \(2^n!\) possible permutations on \(n\)-bit blocks. To specify an arbitrary permutation of \(2^n\) elements, you need to store the image of every possible input: \(2^n\) outputs each of \(n\) bits = \(n \cdot 2^n\) bits total. For \(n = 64\) bits: \(64 \cdot 2^{64} \approx 10^{21}\) bits. Clearly impractical — hence practical ciphers like AES use structured, efficient algorithms that approximate ideal behavior with manageable key sizes.

DES Group Property
If DES were a "group" under function composition, then double DES (2DES) would:
aBe twice as secure as single DES.
bOffer no additional security — 2DES would be equivalent to single DES with some key \(K_3\).
cAlways produce the same ciphertext as single DES for any two keys.
dRequire the MITM attack to be modified to work on 112-bit keys.
Explanation
✓ Correct: (b)
If the set of DES permutations were closed under composition (a group), then for any keys \(K_1, K_2\), there would exist \(K_3\) such that \(E_{K_2}(E_{K_1}(P)) = E_{K_3}(P)\) — 2DES would just be single DES. The security question of whether DES is a group was studied seriously; it is believed NOT to be a group. If it were, double DES would be pointless and triple DES would still only achieve single DES security.

3DES Effective Security
Why does 3DES with three independent 56-bit keys provide approximately 112 bits of security rather than 168?
aBecause 3DES reuses the middle decryption key, reducing the total key material.
bA time-space tradeoff attack can recover the key with work proportional to \(2^{112}\) using manageable storage, far less than \(2^{168}\).
cThe three keys are derived from each other, so only one is truly independent.
dBecause 56 × 3 = 168 but DES itself has only 64-bit block size, limiting security to 64 bits.
Explanation
✓ Correct: (b)
A meet-in-the-middle-style attack on 3DES: fix \(K_3\) and perform a MITM between \(E_{K_1}(P)\) and \(D_{K_3}(C)\), finding matching values with \(D_{K_2}\). The full attack requires time \(\approx 2^{112}\) and storage \(\approx 2^{56}\). This is much less than the theoretical \(2^{168}\) brute force but still impractical today. The conclusion: 3DES offers approximately 112-bit security, sufficient for current standards but declining as hardware improves. NIST deprecated 3DES in 2023.

CBC Parallelism
In CBC mode, which operation can be parallelized across blocks and which cannot?
aBoth encryption and decryption can be fully parallelized.
bNeither encryption nor decryption can be parallelized.
cDecryption can be parallelized (each \(P_i = D_K(C_i) \oplus C_{i-1}\) is independent); encryption cannot (each \(C_i\) depends on \(C_{i-1}\)).
dEncryption can be parallelized; decryption cannot.
Explanation
✓ Correct: (c)
CBC encryption: \(C_i = E_K(P_i \oplus C_{i-1})\). To compute \(C_i\), you need \(C_{i-1}\) first — strictly sequential. CBC decryption: \(P_i = D_K(C_i) \oplus C_{i-1}\). Since all ciphertext blocks are known (received from sender), \(D_K(C_i)\) can be computed for all \(i\) simultaneously. You just need to XOR with the appropriate \(C_{i-1}\). So decryption is fully parallelizable. This property makes CBC faster for decryption on multi-core hardware.

CTR Random Access
To decrypt block number 1000 of a CTR-mode ciphertext without decrypting blocks 1–999, you:
aMust first decrypt all preceding blocks since CTR maintains a running state.
bDirectly compute the keystream block by encrypting \(\text{Nonce} \| 1000\), then XOR with \(C_{1000}\).
cApply the cipher's decryption function to \(C_{1000}\) directly, as in ECB mode.
dXOR \(C_{1000}\) with \(C_{999}\) to recover the keystream, then XOR with the block cipher key.
Explanation
✓ Correct: (b)
CTR mode keystream block \(i\) = \(E_K(\text{Nonce} \| i)\). This is entirely independent of all other blocks. To decrypt block 1000: compute keystream block 1000 = \(E_K(\text{Nonce} \| 1000)\), then \(P_{1000} = C_{1000} \oplus E_K(\text{Nonce} \| 1000)\). No other blocks are needed. This makes CTR ideal for random-access applications (e.g., encrypted file systems, databases) and for parallel encryption of large files.

Needham-Schroeder KDC
In the Needham-Schroeder protocol, the Key Distribution Center (KDC) is trusted because:
aIt is the only entity capable of performing public-key operations.
bEvery principal shares a unique secret master key only with the KDC, allowing the KDC to issue and protect session keys between any pair of principals.
cThe KDC signs all session keys with an RSA private key that is regularly rotated.
dThe KDC uses quantum key distribution to securely deliver session keys.
Explanation
✓ Correct: (b)
The KDC model: each principal \(A\) shares a long-term master key \(K_A\) with the KDC (established offline). When \(A\) wants to communicate securely with \(B\), the KDC generates a session key \(K_{AB}\), encrypts it under \(K_A\) (for \(A\)) and under \(K_B\) (for \(B\), in the "ticket"). Neither \(A\) nor \(B\) needs to know the other's master key. The security assumption: the KDC is fully trusted and its master key store is secure. If the KDC is compromised, all communications are compromised.

TGT vs Service Ticket
In Kerberos, a Ticket-Granting Ticket (TGT) differs from a service ticket in that:
aThe TGT is encrypted under the client's password-derived key; the service ticket is not encrypted.
bThe TGT is encrypted under the TGS's long-term key and is used to obtain service tickets; a service ticket is encrypted under the specific service's long-term key and grants access to that service.
cThe TGT contains the user's plaintext password; the service ticket contains only the session key.
dTGTs are valid indefinitely; service tickets expire after each use.
Explanation
✓ Correct: (b)
The TGT is a ticket for the TGS (Ticket-Granting Service). It is encrypted under the TGS's long-term key \(K_{TGS}\), so only the TGS can read it. The client carries it opaquely and presents it to the TGS when requesting any service ticket. A service ticket (e.g., for a file server) is encrypted under that file server's long-term key \(K_{FS}\) — only the file server can decrypt it, revealing the session key for the client-server communication. Both the TGT and service tickets are opaque to the client.

Kerberos Timestamps
Kerberos uses timestamps to:
aEnforce an order on messages so they must be processed within 5 seconds.
bLimit the validity window of tickets and authenticators, preventing replay attacks by rejecting messages older than the allowed clock skew (typically ±5 minutes).
cAllow the KDC to log all authentication events with exact times for auditing.
dReplace nonces entirely, since synchronized clocks are equivalent to unique random values.
Explanation
✓ Correct: (b)
Kerberos's replay protection relies on timestamps embedded in authenticators and tickets. A replayed authenticator will have a timestamp outside the allowed window (default ±5 minutes from the server's current time). Servers cache recently-seen authenticators within the window to catch within-window replays. This requires clocks across the realm to be synchronized — Kerberos requires NTP (Network Time Protocol). Without timestamp checking, an attacker who captures a valid authenticator could replay it indefinitely.

RC4 WEP Weakness
The main cryptographic weakness of RC4 as used in WEP (Wi-Fi protocol) was:
aWEP used RC4 with a 40-bit key, giving only \(2^{40}\) security.
bWEP reused the RC4 keystream (by using the same base key with a short 24-bit IV that wraps), and the first bytes of the RC4 keystream are statistically biased, leaking key bytes.
cWEP used RC4 in CBC mode, which is incompatible with stream ciphers.
dRC4 generates identical keystreams for any two keys that share the first byte.
Explanation
✓ Correct: (b)
WEP prepended a 24-bit IV to the base key: \(\text{RC4}(\text{IV} \| K)\). With only \(2^{24} \approx 16\text{M}\) possible IVs, in a busy network the same IV+key pair repeats quickly — keystream reuse allows XOR-based plaintext recovery. Worse, Fluhrer-Mantin-Shamir (2001) showed that certain "weak" IVs leak key bytes through the first RC4 output byte, allowing the full key to be recovered from ~4 million packets. This completely broke WEP. The fix: WPA2 using AES-CCMP.

Session Key vs Master Key
The primary reason to use short-lived session keys rather than the long-term master key for data encryption is:
aSession keys are larger (256 bits) than master keys (128 bits) and thus more secure.
bLimiting the amount of data encrypted under any one key reduces the damage from a key compromise — past sessions remain secure (forward secrecy), and the master key is rarely used, reducing its exposure.
cMaster keys cannot be used for encryption — they are only used for MAC computation.
dSession keys are generated by the recipient, so the sender never learns them.
Explanation
✓ Correct: (b)
Key hygiene: using the same key for large amounts of data exposes it to more cryptanalysis. If a session key is compromised, only that session's data is exposed — other sessions used different keys. The master key, used rarely (only for key exchange), has minimal exposure and is not revealed to other parties. In Kerberos, master keys never leave the AS or the principal — session keys are what's distributed and used for actual communication.

CBC Error Propagation
In CBC mode decryption, a burst error affecting all bits in ciphertext block \(C_i\) will corrupt:
aOnly plaintext block \(P_i\).
bPlaintext block \(P_i\) entirely (because \(D_K(C_i)\) is garbled), plus only one bit in plaintext block \(P_{i+1}\) (for each bit flipped in \(C_i\)).
cAll plaintext blocks from \(P_i\) through the end of the message.
dPlaintext blocks \(P_i\) and \(P_{i+1}\) entirely, but no further blocks.
Explanation
✓ Correct: (b)
CBC decryption: \(P_i = D_K(C_i) \oplus C_{i-1}\). A fully-corrupted \(C_i\) means \(D_K(C_i)\) is completely garbled → \(P_i\) is entirely corrupted. But \(P_{i+1} = D_K(C_{i+1}) \oplus C_i\): the decryption \(D_K(C_{i+1})\) is correct (since \(C_{i+1}\) is untouched), and \(C_i\) is XORed in directly. Each bit of \(C_i\) that was flipped flips the corresponding bit of \(P_{i+1}\). From block \(P_{i+2}\) onward: \(C_{i+1}\) is correct, so decryption resumes normally. CBC has limited, self-recovering error propagation.

OFB as Stream Cipher
OFB (Output Feedback) mode is essentially a stream cipher because:
aIt processes one bit at a time rather than a full block at a time.
bIt generates a keystream independently of the plaintext and ciphertext by recursively encrypting an internal state, then XORs the keystream with the data.
cOFB uses RC4 internally instead of AES to generate the keystream.
dThe OFB keystream can only be 8 bits wide, making it equivalent to a byte-oriented stream cipher.
Explanation
✓ Correct: (b)
OFB keystream generation: \(O_0 = \text{IV}\), \(O_i = E_K(O_{i-1})\). The keystream block \(O_i\) depends only on the IV and the key — never on the plaintext or ciphertext. This is exactly how a stream cipher works: a deterministic keystream XORed with the data. Like a stream cipher, OFB requires: (1) the IV/nonce must never be reused (keystream reuse is catastrophic); (2) the keystream is pre-computable for any block position.

Which Mode Is Self-Synchronizing?
Which cipher mode is self-synchronizing — meaning it can recover from lost ciphertext bits after a fixed delay?
aECB (Electronic Codebook)
bOFB (Output Feedback)
cCFB (Cipher Feedback)
dCTR (Counter)
Explanation
✓ Correct: (c)
CFB's shift register is fed by the ciphertext. After a bit synchronization error, the corrupted region propagates through the shift register. Once the shift register is full of correct ciphertext bytes (after one full shift register length of data), the keystream generator resynchronizes automatically — decryption recovers. This "self-synchronizing" property made CFB attractive for serial communications where bits can be dropped. OFB is synchronous (not self-synchronizing); a single lost bit permanently desynchronizes the keystream.

RSA Key Pair
In RSA, the public key is the pair \((e, n)\) and the private key is the pair \((d, n)\). Confidentiality — sending a secret message to Alice — requires:
aEncrypting with Alice's private key \((d, n)\).
bEncrypting with Alice's public key \((e, n)\); only Alice can decrypt with her private key \((d, n)\).
cThe sender encrypting with their own private key, so Alice can verify the sender's identity.
dBoth parties first agreeing on a shared key, then using RSA to MAC the ciphertext.
Explanation
✓ Correct: (b)
RSA key usage: (1) Confidentiality: encrypt with the recipient's public key; only the recipient can decrypt with their private key. (2) Digital signature: sign with the signer's private key; anyone can verify with the signer's public key. Using Alice's private key for encryption (option a) would produce a signature-like operation — anyone with the public key could "decrypt" it, providing no confidentiality at all.

Why e=65537
The value \(e = 65537\) (\(= 2^{16}+1\)) is the most common RSA public exponent because:
a65537 is the smallest prime, giving the most efficient decryption.
bIt is prime, has a very short binary representation (only two 1-bits), enabling fast encryption via square-and-multiply, and is large enough to avoid the small-exponent attack with padding.
c65537 ensures \(d\) is always exactly 2048 bits long for standard key sizes.
dRSA standards mandate \(e = 65537\) for compliance.
Explanation
✓ Correct: (b)
\(e = 65537 = 10000000000000001_2\): exactly two 1-bits means square-and-multiply requires only 16 squarings and 1 multiplication — extremely fast encryption. It is prime (so unlikely to share a factor with \(\phi(n)\), ensuring \(d\) exists). It is large enough that direct encryption of small messages does not produce \(m^e < n\) without padding (unlike \(e=3\)). Smaller values like \(e=3\) or \(e=17\) are also used but require careful padding to avoid the small-exponent attack.

RSA φ(n)
For RSA modulus \(n = pq\) with \(p, q\) prime, Euler's totient function \(\phi(n)\) equals:
a\(p + q\)
b\(pq - 1\)
c\((p-1)(q-1)\)
d\(\frac{pq}{2}\)
Explanation
✓ Correct: (c)
\(\phi(n)\) counts integers in \([1, n]\) coprime to \(n\). For prime \(p\): \(\phi(p) = p-1\). For \(n = pq\) with distinct primes: \(\phi(pq) = \phi(p)\phi(q) = (p-1)(q-1)\). This equals \(pq - p - q + 1 = n - (p+q) + 1\). Knowing \(\phi(n)\) allows computing \(d = e^{-1} \bmod \phi(n)\) and thus breaking RSA. If an attacker knows \(n\) and \(\phi(n)\), they can compute \(p + q = n - \phi(n) + 1\) and then solve for \(p, q\) from the quadratic equation \(x^2 - (p+q)x + n = 0\).
26
Knowing φ(n) Breaks RSA
An attacker who knows \(\phi(n)\) but not \(p\) or \(q\) can:
aImmediately determine \(d\) only if \(e = 3\).
bCompute \(d = e^{-1} \bmod \phi(n)\) using the extended Euclidean algorithm, breaking RSA regardless of the key size.
cDetermine \(p\) and \(q\) only if \(n\) is odd.
dDecrypt only messages shorter than \(\sqrt{n}\) bits.
Explanation
✓ Correct: (b)
The private key \(d\) is defined entirely by \(e\) and \(\phi(n)\): \(d = e^{-1} \bmod \phi(n)\). Given \(\phi(n)\), the extended Euclidean algorithm finds \(d\) in \(O(\log^2 n)\) time. This is why RSA security requires that \(\phi(n)\) remains secret: it must be computationally infeasible to compute \(\phi(n)\) from \(n\) alone (equivalent to factoring). Any leak of \(\phi(n)\) — even partial information — is catastrophic.

RSA Signature
To digitally sign a message \(m\) using RSA, the signer computes:
a\(s = m^e \bmod n\) (encryption with the public key)
b\(s = H(m)^d \bmod n\) (apply a hash function, then raise to the private exponent)
c\(s = m \oplus d\) (XOR with the private key)
d\(s = H(m \| d)\) (hash the message concatenated with the private key)
Explanation
✓ Correct: (b)
In practice, the message is first hashed (to a fixed size and to prevent chosen-message attacks): compute \(h = H(m)\), then sign the hash: \(s = h^d \bmod n\). Verification: compute \(s^e \bmod n\) and check that it equals \(H(m)\). Signing the raw message \(m\) would be vulnerable to several attacks (existential forgery, chosen-message). Padding schemes like PKCS#1 v1.5 and RSA-PSS add additional randomization and structure before signing.

Fermat Factorization
If RSA primes \(p\) and \(q\) are chosen too close together (\(|p - q|\) is small), the modulus \(n\) is vulnerable to:
aA birthday attack that finds two messages with the same hash.
bFermat's factorization: since \(n = pq \approx \left(\frac{p+q}{2}\right)^2 - \left(\frac{p-q}{2}\right)^2\), trial values near \(\sqrt{n}\) quickly find the factorization.
cThe small public exponent attack exploiting CRT across multiple recipients.
dA chosen-ciphertext attack that multiplies the ciphertext by a known value.
Explanation
✓ Correct: (b)
Fermat factorization: write \(n = a^2 - b^2 = (a+b)(a-b)\) where \(a = \frac{p+q}{2}\) and \(b = \frac{p-q}{2}\). Start with \(a = \lceil\sqrt{n}\rceil\) and search for \(b\) such that \(a^2 - n\) is a perfect square. If \(p\) and \(q\) are close, \(b\) is small and the search terminates quickly. Fix: choose \(p\) and \(q\) independently and uniformly from large prime ranges, ensuring \(|p-q|\) is large — e.g., both at least \(2^{511}\) for a 2048-bit modulus.

Low Entropy Attack
An RSA key generation with low-entropy randomness (e.g., seeded with a predictable value) is dangerous because:
aLow entropy means the prime generation algorithm terminates early, producing smaller-than-expected primes.
bThe set of candidate primes is small and predictable, allowing an attacker to search through likely primes and factor \(n\) efficiently.
cLow entropy causes \(e\) and \(\phi(n)\) to share a factor, making \(d\) undefined.
dThe public key \((e, n)\) becomes identical across different users.
Explanation
✓ Correct: (b)
Real-world attack: Heninger et al. (2012) scanned millions of RSA public keys from the internet and found that many devices (routers, VPN appliances) generated keys with low-entropy seeds during boot. Many generated the same prime — when two devices generated \(n_1 = p \cdot q_1\) and \(n_2 = p \cdot q_2\) (sharing one prime), computing \(\gcd(n_1, n_2) = p\) with a simple GCD revealed both private keys instantly. 0.2% of examined hosts were vulnerable. Lesson: RSA key generation requires strong entropy, especially at first boot.

Minimum RSA Key Size
The minimum recommended RSA key size for new deployments today is:
a512 bits
b1024 bits
c2048 bits
d16384 bits
Explanation
✓ Correct: (c)
512-bit RSA was broken in 1999 (RSA-155 factored). 768-bit was broken in 2009 (RSA-768). 1024-bit is considered at risk and is deprecated by NIST. The current minimum is 2048 bits, recommended until 2030. For longer-term security, 3072 or 4096 bits are recommended. Note: RSA key sizes need to be much larger than symmetric keys for equivalent security — a 128-bit symmetric key provides roughly the same security as a 3072-bit RSA key, because factoring is sub-exponential while symmetric brute-force is fully exponential.

RSA vs Symmetric Speed
RSA is approximately how much slower than AES for encrypting the same amount of data?
a2–5× slower
b50–100× slower
c1,000–10,000× slower
dRSA and AES have the same speed when the RSA key is the same length as the AES key in bits.
Explanation
✓ Correct: (c)
RSA encryption/decryption involves modular exponentiation with thousands-of-bit numbers — each operation requires hundreds to thousands of big-integer multiplications. AES-128 processes 16 bytes in 10 rounds of simple byte-level operations, running at gigabytes per second on modern hardware. Benchmarks show RSA-2048 is typically 1,000–10,000× slower than AES-128 for equivalent throughput. This is why RSA is only used to exchange small items (session keys, digital signatures), not for bulk data encryption.

RSA Textbook Correctness
Textbook RSA correctness (\(m^{ed} \equiv m \pmod{n}\)) requires:
a\(m\) must be exactly \(n/2\) bits long.
b\(m\) must satisfy \(0 \leq m < n\) and be coprime to \(n\) (for strict Euler's theorem; in practice, the result holds for all \(m < n\) when using Carmichael's \(\lambda(n)\)).
c\(e\) and \(d\) must satisfy \(e + d = \phi(n)\).
d\(p\) and \(q\) must be equal for the mathematical proof to hold.
Explanation
✓ Correct: (b)
For Euler's theorem to guarantee \(m^{ed} \equiv m \pmod{n}\), we need \(\gcd(m, n) = 1\). Since \(n = pq\) with large primes, a random \(m < n\) is almost always coprime to \(n\) (the probability of sharing a factor is \(\approx 2/p \approx 2^{-1023}\)). Additionally, \(m\) must satisfy \(0 \leq m < n\) so it represents a unique element. In practice, OAEP padding ensures the padded message is always in the valid range. Using Carmichael's \(\lambda(n) = \text{lcm}(p-1, q-1)\) instead of \(\phi(n)\) makes the correctness hold for all \(m\).

Intermediate CA
Why do public PKI hierarchies use intermediate CAs instead of having the root CA directly sign all end-entity certificates?
aRoot CAs use 4096-bit keys that are too slow for high-volume certificate signing.
bRoot CA private keys are kept strictly offline (air-gapped); intermediate CAs handle online signing, so compromising an intermediate CA does not compromise the root.
cIntermediate CAs add an extra layer of encryption to protect the root's key material.
dBrowser trust stores cannot hold more than 50 root CA certificates, requiring the hierarchy.
Explanation
✓ Correct: (b)
Root CA private keys are the ultimate trust anchors. If compromised, all certificates issued under them are suspect. Therefore, root CAs are kept completely offline (hardware security modules in physically secure locations, powered on only for certificate signing ceremonies). Day-to-day certificate issuance is delegated to intermediate CAs with online connectivity. If an intermediate CA is compromised, its certificate can be revoked and a new one issued by the root — the root itself remains uncompromised.

✓ Correct: (b)

OCSP vs CRL
The main advantage of OCSP (Online Certificate Status Protocol) over CRLs is:

Explanation
✓ Correct: (b)

DH Generator
In Diffie-Hellman, the generator \(g\) must be chosen such that:
a\(g = 2\) always, since 2 is the smallest non-trivial generator.
b\(g\) must equal \(p - 1\) to ensure the full group is covered.
c\(g\) is a primitive root modulo \(p\) (a generator of the entire multiplicative group \(\mathbb{Z}_p^*\)), so that powers of \(g\) cover all non-zero elements.
d\(g\) must be a random prime larger than \(p/2\).
Explanation
✓ Correct: (c)
For DH security, \(g\) should be a primitive root modulo \(p\): its successive powers \(g^1, g^2, \ldots, g^{p-1}\) produce every element in \(\mathbb{Z}_p^* = \{1, 2, \ldots, p-1\}\). This ensures the shared secret \(g^{ab}\) can be any element of the full group, providing maximum entropy. If \(g\) generates only a small subgroup, the discrete log problem becomes easier (the DLP is only as hard as the largest prime factor of the subgroup order — this is why safe primes \(p = 2q+1\), where \(q\) is also prime, are preferred).

DHE vs static DH
In TLS, DHE (Ephemeral Diffie-Hellman) provides an advantage over static DH because:
aDHE uses larger DH groups, providing more bits of security per key exchange.
bDHE generates a fresh DH key pair for each session and discards it afterward, providing Perfect Forward Secrecy — compromising the server's long-term key cannot retroactively decrypt recorded sessions.
cDHE encrypts the DH public values with AES before transmission, preventing MITM.
dDHE is authenticated by the client's certificate instead of the server's, reversing the trust direction.
Explanation
✓ Correct: (b)
Static DH: the server uses the same DH private key for all sessions (the server's DH public key is in its certificate). If the server's private key is later stolen, all past sessions can be decrypted by an adversary who recorded the traffic. DHE: a fresh random DH private key is generated for each TLS handshake. The session key derives from this ephemeral key; the ephemeral key is discarded post-handshake. Stolen long-term key → still cannot decrypt past sessions. TLS 1.3 removed static DH entirely — only DHE and ECDHE are permitted.

ElGamal Public Key
In ElGamal encryption with public parameters \((p, g)\), the public key is \(y\) where:
a\(y = x \cdot g \bmod p\), where \(x\) is the private key.
b\(y = g^x \bmod p\), where \(x\) is the private key.
c\(y = x^g \bmod p\), where \(g\) is the generator.
d\(y = g^{-1} \bmod x\), the modular inverse of \(g\) with respect to \(x\).
Explanation
✓ Correct: (b)
ElGamal is built directly on DH. The private key is a random \(x \in [2, p-2]\). The public key is \(y = g^x \bmod p\). Recovering \(x\) from \(y\) requires solving the discrete logarithm problem in \(\mathbb{Z}_p^*\) — hard for large enough \(p\). Encryption: pick random nonce \(K\), send \((C_1, C_2) = (g^K \bmod p,\; m \cdot y^K \bmod p)\). Decryption: compute \(m = C_2 \cdot (C_1^x)^{-1} \bmod p\), exploiting \(C_1^x = g^{Kx} = y^K\).

ElGamal DSS Nonce
In ElGamal DSS, the random nonce \(K\) used per signature must satisfy:
a\(K\) must equal the hash of the message to ensure uniqueness.
b\(K\) must be unique (never reused across different messages), random, and kept secret — reuse immediately reveals the private key.
c\(K\) must be the same as the private key \(x\) to ensure correctness of verification.
d\(K\) must be publicly disclosed with each signature for the verification equation to hold.
Explanation
✓ Correct: (b)
ElGamal/DSA signature with reused \(K\): given two signatures \((r, s_1)\) and \((r, s_2)\) (same \(r = g^K\) implies same \(K\)) on messages \(m_1\) and \(m_2\): \(s_1 - s_2 \equiv K^{-1}(m_1 - m_2) \pmod{p-1}\). Solving: \(K \equiv (m_1 - m_2)(s_1-s_2)^{-1}\), then \(x = r^{-1}(Ks_1 - m_1) \pmod{p-1}\). The entire private key is exposed. DLP Hardness
The Discrete Logarithm Problem (DLP) is: given \(g, y, p\), find \(x\) such that \(g^x \equiv y \pmod{p}\). Why is this hard?
aDivision is not defined in \(\mathbb{Z}_p^*\), preventing direct inversion of exponentiation.
bNo polynomial-time classical algorithm is known; the best algorithms (index-calculus) require sub-exponential time, making DLP infeasible for large enough groups.
cThe discrete logarithm is always an irrational number, which cannot be stored exactly.
dThe group \(\mathbb{Z}_p^*\) has no structure, preventing any algorithmic shortcut.
Explanation
✓ Correct: (b)
Computing \(x\) from \(g^x \bmod p\) is believed hard. The best known algorithms (Baby-step Giant-step, Pollard's rho, Index Calculus for prime fields) run in sub-exponential time — much faster than brute force but still impractical for large \(p\). For 2048-bit prime \(p\), the security is comparable to 112-bit symmetric security. Note: Shor's quantum algorithm solves DLP in polynomial time — quantum computers would break DH and ElGamal, motivating post-quantum cryptography (lattice-based, hash-based, etc.).

ElGamal vs RSA
Compared to RSA for the same security level, ElGamal encryption:
aRequires a smaller key because DLP is harder than factoring at the same bit length.
bIs deterministic — encrypting the same message twice always gives the same ciphertext.
cIs probabilistic (randomized per encryption due to the nonce \(K\)) and produces ciphertext twice the size of the plaintext, unlike RSA which is deterministic in textbook form.
dUses the same key structure as RSA, since both rely on modular exponentiation.
Explanation
✓ Correct: (c)
ElGamal is inherently probabilistic: each encryption uses a fresh random nonce \(K\), so encrypting the same message twice gives different ciphertexts (different \(C_1 = g^K\)). This is a security advantage — CPA (Chosen Plaintext Attack) security, since the adversary cannot distinguish encryptions of the same message. Textbook RSA is deterministic (\(c = m^e \bmod n\) is always the same for the same \(m\)). The trade-off: ElGamal ciphertext is twice the plaintext size, RSA is roughly the same size.

Collision vs Pre-image
Correctly ordered from easiest to hardest for an attacker to break:
aCollision resistance (easiest: \(O(2^{n/2})\) via birthday attack) → Second pre-image resistance (\(O(2^n)\)) → Pre-image resistance (hardest: \(O(2^n)\)).
bPre-image → Second pre-image → Collision (hardest).
cAll three are equally hard for any secure hash function.
dSecond pre-image → Collision → Pre-image.
Explanation
✓ Correct: (a)
Complexity for an \(n\)-bit hash: (1) Collision: find any \(m_1 \neq m_2\) with \(H(m_1)=H(m_2)\). The attacker chooses both messages freely — birthday attack gives \(O(2^{n/2})\). (2) Second pre-image: given specific \(m_1\), find \(m_2\). Less freedom → \(O(2^n)\). (3) Pre-image: given only \(h\), find any \(m\). No starting message to help → \(O(2^n)\). Note: collision resistance is strictly weaker than second pre-image resistance: if an algorithm can find second pre-images, it can find collisions (just use any message as \(m_1\)).

Birthday Bound Calculation
A hash function produces a 160-bit output (like SHA-1). Its collision resistance (birthday bound) is approximately:
a\(2^{160}\) operations
b\(2^{80}\) operations
c\(2^{40}\) operations
d\(2^{20}\) operations
Explanation
✓ Correct: (b)
For an \(n\)-bit hash, the birthday bound is \(2^{n/2}\). For SHA-1 with \(n=160\): collision resistance \(\approx 2^{80}\). In 2005, Wang et al. demonstrated SHA-1 collisions with fewer than \(2^{69}\) operations — below the birthday bound — meaning SHA-1 has structural weaknesses. In 2017, Google's SHAttered project produced the first practical SHA-1 collision (two different PDF files with identical SHA-1 hashes). SHA-1 is now deprecated for all security uses.

Length Extension Attack
Why is \(H(K \| m)\) insecure as a MAC for Merkle-Damgård hash functions?
aAn attacker can compute \(H(K)\) from \(H(K \| m)\) without knowing \(K\).
bThe final hash state equals the internal compression state; an attacker knowing \(H(K \| m)\) can continue hashing to produce valid \(H(K \| m \| \text{pad} \| \text{extra})\) for any chosen \(\text{extra}\), without knowing \(K\).
cPrepending the key makes frequency analysis of the message trivial for an attacker.
dThe key \(K\) is revealed in the first block of the hash output due to Merkle-Damgård's structure.
Explanation
✓ Correct: (b)
Merkle-Damgård chains state: after hashing \(K \| m\), the final state \(H_k\) is fully determined and equals the output. An attacker treats \(H_k\) as a starting state and hashes additional blocks (the "extension"). Since the padding of \(K \| m\) is predictable, the attacker knows the exact padded form and can produce a valid \(H(K \| m \| \text{pad} \| \text{extra})\). This forges a valid MAC for a new message \(m \| \text{pad} \| \text{extra}\) — an authenticated but attacker-chosen message. Flickr's API was vulnerable to exactly this in 2009.

HMAC Inner/Outer
In HMAC, the outer hash \(H((K \oplus \text{opad}) \| \text{inner})\) serves to:
aEncrypt the inner hash output using the key.
bPrevent length extension attacks against the inner hash: an attacker cannot extend the HMAC because the inner state is hidden behind a second keyed hash.
cProvide confidentiality by hashing the message a second time to produce ciphertext.
dAuthenticate the key itself, preventing key substitution attacks.
Explanation
✓ Correct: (b)
HMAC's double application is deliberate: the inner hash \(H((K \oplus \text{ipad}) \| m)\) produces a keyed hash of the message. If HMAC stopped here, it would be vulnerable to length extension (an attacker knowing the inner hash could extend it). The outer hash takes the inner result and hashes it again under a differently-padded key (\(K \oplus \text{opad}\)). An attacker cannot perform a length extension without knowing the outer key — the inner state is completely hidden by the outer hash. This two-layer construction makes HMAC provably secure even against length extension.

Merkle-Damgård Padding
Merkle-Damgård strengthening appends the message length to the padding. This is important because:
aIt ensures the message is always exactly 512 bits, making the hash constant-time.
bWithout the length, two messages with the same content but different lengths (e.g., \(m\) and \(m \| \text{pad}\)) could produce the same padded message and thus the same hash, breaking collision resistance.
cThe length field provides additional entropy to the hash function.
dAppending the length enables parallel hash computation across multiple blocks.
Explanation
✓ Correct: (b)
The Merkle-Damgård strengthening theorem requires the padding to encode the message length. Without it: message \(m_1\) padded to 512 bits might look identical to the first 512 bits of message \(m_2\) which happens to start with the same content followed by the natural padding. With the length appended in the last block, each distinct message length produces a distinct padded form, ensuring that the security reduction from the compression function to the hash remains valid. All standard hash functions (MD5, SHA-1, SHA-256) include the length in their padding.

MAC Unforgeability
A secure MAC (Message Authentication Code) is existentially unforgeable under chosen-message attacks, meaning:
aThe MAC key cannot be reused for more than one message.
bEven after obtaining valid MACs for many chosen messages, an adversary cannot produce a valid MAC for any new message they have not previously queried.
cThe MAC output cannot be decrypted even with knowledge of the secret key.
dThe MAC function produces a unique output for every possible input, acting as a perfect hash.
Explanation
✓ Correct: (b)
Existential unforgeability under chosen-message attacks (EUF-CMA): the adversary can adaptively query the MAC oracle for any messages of their choice and receive valid tags. Despite this, they cannot forge a valid tag for any new message (one not in their query set). This is a strong and necessary security definition — without it, an attacker observing legitimate MACs could construct MACs for fraudulent messages. HMAC-SHA-256 achieves EUF-CMA security under standard hash assumptions.

SHA-3 vs SHA-2
SHA-3 (Keccak) was standardized by NIST in 2015 primarily because:
aSHA-2 was broken by practical collision attacks in 2012.
bNIST wanted a backup hash function with a fundamentally different design (sponge construction, not Merkle-Damgård) in case weaknesses in SHA-2's structure were discovered.
cSHA-3 is significantly faster than SHA-2 on all hardware, making it the preferred replacement.
dSHA-3 uses a 1024-bit internal state, making collisions physically impossible.
Explanation
✓ Correct: (b)
SHA-2 (SHA-256, SHA-512) remains secure as of 2025 — no practical attacks exist. SHA-3 was standardized as a backup with a completely different structure: the sponge construction (absorb/squeeze phases over a large permutation state) rather than the Merkle-Damgård design used by SHA-1, SHA-2, MD5. Hash in Digital Signature
Why is a message always hashed before being signed in RSA (i.e., \(s = H(m)^d \bmod n\) rather than \(s = m^d \bmod n\))?
aHashing speeds up the signing operation because \(H(m)\) is shorter than \(m\).
bHashing ensures the signature length equals the key length regardless of message size.
cBoth (a) and (b): the hash is always a fixed size (fits in the RSA block), and hashing prevents existential forgery attacks where an attacker chooses \(s\) and computes \(m = s^e\) as a "signed" message.
dHashing provides confidentiality by obscuring the plaintext from the verifier.
Explanation
✓ Correct: (c)
Two reasons: (1) Fixed size: RSA operates on values \(< n\). A long message might exceed \(n\) and need blocking; a hash always fits in one RSA block. (2) Security: textbook RSA signatures are vulnerable to existential forgery — an attacker can pick any \(s\), compute \(m = s^e \bmod n\), and claim Alice signed \(m\). aTruncated to the block size by discarding the excess bytes.
bHashed down to the output size \(H(K)\), which is then used as the effective key.
cSplit into multiple sub-keys, each of block size, all XORed together.
dRejected with an error — HMAC requires the key to be exactly the block size.
Explanation
✓ Correct: (b)

Next-Bit Test
A CSPRNG must pass the next-bit test. This means:
aEach output bit must be produced within one clock cycle.
bGiven any prefix of the output sequence, no polynomial-time algorithm can predict the next bit with probability better than \(1/2 + \epsilon\) for any negligible \(\epsilon\).
cThe PRNG must produce the exact next value in a sequence within one step.
dEach output bit must pass the NIST SP 800-22 statistical randomness test suite.
Explanation
✓ Correct: (b)
The next-bit test (Yao, 1982) defines computational unpredictability: given bits \(b_1, b_2, \ldots, b_k\) of the CSPRNG output, no efficient (polynomial-time) algorithm can guess \(b_{k+1}\) with probability meaningfully better than random (50%). This is stronger than just passing statistical tests — a sequence can look random statistically but still be predictable by an algorithm. LCGs can pass many statistical tests while being completely predictable algebraically. Practical CSPRNGs like ChaCha20-based generators are designed to be computationally indistinguishable from truly random sequences.

BBS Output Bits
The Blum Blum Shub (BBS) generator's recurrence is \(x_i = x_{i-1}^2 \bmod n\). The safe way to extract output bits per step is:

bUse only \(O(\log \log n)\) least significant bits of \(x_i\) per step — specifically, just the LSB is provably secure.

State Compromise
If an attacker learns the complete internal state of a PRNG at time \(T\), they can:
aDetermine only the next 100 outputs before the state diverges.
bCompute all future outputs deterministically (since PRNGs are deterministic) and, if the algorithm is invertible, all past outputs as well.
cDetermine only past outputs — future outputs are randomized by re-seeding.
dBreak only the current session's keys but not future or past sessions.
Explanation
✓ Correct: (b)
PRNGs are deterministic state machines: \(s_{n+1} = f(s_n)\), output = \(g(s_n)\). Given the full state at time \(T\), computing all future outputs is trivial — just run the algorithm forward. Past outputs depend on invertibility: LCGs and simple shift registers are easily reversed; AES-CTR-DRBG's state cannot be run backward (AES decryption is needed, which is possible if the key is known — so past outputs are recoverable too). Mitigations: "forward secrecy" for PRNGs means periodically re-seeding with fresh entropy to bound the damage from state exposure.

Shannon Entropy Calculation
A six-sided die has Shannon entropy of approximately:
a1 bit (same as a coin flip)
b2 bits
c\(\log_2 6 \approx 2.58\) bits
d6 bits (one bit per face)
Explanation
✓ Correct: (c)
For a uniform distribution over \(k\) outcomes: \(H = \log_2 k\) bits. For a fair die: \(H = \log_2 6 \approx 2.585\) bits. Statistical tests (frequency tests, autocorrelation, etc.) check for patterns in the output distribution. A PRNG can pass all statistical tests while being completely predictable algorithmically. Example: an LCG can be tuned to pass virtually all statistical tests yet is broken in three output values. True CSPRNG certification requires proof (or strong evidence) of computational unpredictability — typically through security reductions to hard problems (like AES security or factoring) or through NIST FIPS 140-2/3 certification which includes both statistical testing and algorithmic analysis. Statistical tests are a floor, not a ceiling.
