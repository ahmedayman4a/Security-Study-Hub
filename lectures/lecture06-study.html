Why Randomness Is Critical in Security Part 1

The Stakes

There is a line that appears in almost every cryptographic protocol: "A chooses a random number…" The entire security of these protocols often hangs on that number being genuinely unpredictable. If an attacker can predict your "random" numbers:

  • Session keys: A predictable AES session key means the attacker can try all likely keys trivially.
  • Nonces in Needham-Schroeder/Kerberos: Predictable nonces allow replay attacks — the freshness protection collapses.
  • RSA prime generation: Predictable seeds during prime generation produce predictable primes. Two devices with similar seeds may share a common prime factor (the "Mining Ps and Qs" attack — 0.5% of all TLS servers broken this way).
  • DH private values: Predictable $X_A$ or $X_B$ destroys perfect forward secrecy.
  • ElGamal signing nonce $K$: Reusing or predicting $K$ directly reveals the private key (the PS3 hack).
  • Password salts: Predictable salts allow pre-computation rainbow table attacks.
  • IVs for CBC/CTR: Predictable IVs enable chosen-plaintext attacks.
🚨 The Core Principle

Cryptography is only as strong as its randomness. A brilliant algorithm running on a bad random number generator is broken. Randomness is not an implementation detail — it is a foundational security requirement.

🧠 Knowledge Check
Name three cryptographic operations that require random numbers and explain what breaks if those numbers are predictable.
Show Answer ▾
(1) RSA prime generation: predictable primes → shared factors → private key compromise.
(2) DH ephemeral key generation: predictable $X_A$ → shared secret predictable → PFS lost.
(3) ElGamal signing nonce $K$: reusing/predicting $K$ across two signatures → private key $X$ recoverable from two signature equations.

PRNG vs TRNG Part 2

Properties of True Random Numbers

Two essential properties define what we actually want from a random number source:

  • Uniform Distribution: All numbers in the designated range must occur equally often. If generating bits, 0s and 1s must each occur with probability $1/2$. Biased generators make some outputs more predictable.
  • Independence (Unpredictability): Even if you know every number generated so far, you cannot predict the next one. Independence is the cryptographic property. A generator that is deterministic once you know its state fails this requirement.

💡 The Practical Solution

True randomness is precious and slow. The solution: use a TRNG to generate a small, high-quality seed. Feed that seed into a CSPRNG. The CSPRNG stretches a small amount of true randomness into a large stream of cryptographically secure pseudorandom bits at high speed.

🧠 Knowledge Check
Why can't a purely algorithmic PRNG provide true randomness?
Show Answer ▾
A PRNG is a deterministic algorithm — given the same seed, it always produces the same sequence. An attacker who discovers the seed (or the algorithm and enough outputs) can reproduce the entire sequence. True randomness requires unpredictability that no algorithm can provide from first principles — only physical phenomena are fundamentally unpredictable.

PRNG — Linear Congruential Generator Part 3

The Formula

The LCG generates pseudorandom numbers using a simple recurrence:

$$X_{n+1} = (a \times X_n + c) \bmod m$$

Parameters:

  • $m$ — modulus ($m > 0$)
  • $a$ — multiplier ($0 < a < m$)
  • $c$ — increment ($0 \leq c < m$)
  • $X_0$ — seed ($0 < X_0 < m$)
WORKED EXAMPLE LCG: a=7, c=0, m=32, seed=1
$n$$X_n$Computation
01seed
17$(7 \times 1) \bmod 32 = 7$
217$(7 \times 7) \bmod 32 = 49 \bmod 32 = 17$
323$(7 \times 17) \bmod 32 = 119 \bmod 32 = 23$
41$(7 \times 23) \bmod 32 = 161 \bmod 32 = 1$ ← cycles!

Period = 4. The sequence $\{7, 17, 23, 1, 7, 17, \ldots\}$ repeats every 4 steps — extremely short and useless for cryptography.

Getting a Better Period

When $m$ is prime and $c = 0$, and $a$ is a primitive root modulo $m$, the recursion produces a sequence of period $m - 1$ (every non-zero value appears exactly once). A well-studied LCG uses $m = 2^{31} - 1$ (a Mersenne prime) and $a = 7^5 = 16807$.

Why LCG Is NOT Cryptographically Secure

❌ Cryptographic Failure

Given any current value $X_n$, the next value $X_{n+1} = (aX_n + c) \bmod m$ is completely predictable if $a$, $c$, $m$ are known (they almost always are). Even if unknown, observing three consecutive outputs $(X_n, X_{n+1}, X_{n+2})$ allows solving for all parameters algebraically. An LCG should never be used to generate session keys, nonces, RSA primes, or any cryptographic material.

PRACTICE LCG: a=5, c=3, m=13, seed=7

Generate the next 4 values:

  • $X_1 = (5 \times 7 + 3) \bmod 13 = 38 \bmod 13 = 12$
  • $X_2 = (5 \times 12 + 3) \bmod 13 = 63 \bmod 13 = 11$
  • $X_3 = (5 \times 11 + 3) \bmod 13 = 58 \bmod 13 = 6$
  • $X_4 = (5 \times 6 + 3) \bmod 13 = 33 \bmod 13 = 7$ ← back to seed!

Period = 4.

CSPRNG — ANSI X9.17/X9.31 Part 4

Overview

The ANSI X9.17/X9.31 algorithm is a CSPRNG standard used in financial transaction systems and email exchange. It uses two-key 3DES (EDE) as its underlying cryptographic primitive, making outputs unpredictable even to attackers who know the algorithm.

Inputs

  • $K_1$, $K_2$: Two 56-bit DES keys (long-term secret). Written as $[K_1, K_2]$.
  • $DT_j$: 64-bit timestamp (current date/time). Changes every iteration.
  • $V_j$: 64-bit state from previous iteration. Initialized with seed $V_0$.

The Formulas

At each step $j$, compute the output $R_j$ and next state $V_{j+1}$:

$$R_j = \text{EDE}([K_1, K_2],\ V_j \oplus \text{EDE}([K_1, K_2],\ DT_j))$$ $$V_{j+1} = \text{EDE}([K_1, K_2],\ R_j \oplus \text{EDE}([K_1, K_2],\ DT_j))$$

Between two consecutive random numbers, there are nine DES encryptions total (three EDE operations × three DES per EDE = 9). This computational cost makes it virtually impossible to predict the next output even knowing all previous outputs.

⚠️ Speed Limitation

The 9 DES operations per output make X9.31 significantly slower than simple PRNGs. It is unsuitable for applications needing millions of random bytes per second (e.g., bulk encryption). It is designed for cryptographic key generation, not streaming random data.

🧠 Knowledge Check
What two inputs to the X9.31 algorithm change at every iteration?
Show Answer ▾
$DT_j$ (the current date/time timestamp, different for each iteration) and $V_j$ (the state from the previous iteration, computed from the previous output $R_{j-1}$). The keys $K_1$ and $K_2$ remain fixed as the long-term secret.

CSPRNG — Blum Blum Shub (BBS) Part 5

Motivation

BBS generates a pseudorandom bit stream with a provable security reduction to the hardness of factoring large integers. If you can predict BBS outputs, you can factor $n$ — and vice versa. This is the strongest theoretical guarantee of any CSPRNG covered in this course.

Setup

  • 1
    Choose two large primes $p$ and $q$ such that $p \equiv q \equiv 3 \pmod{4}$ (i.e., $p \bmod 4 = 3$ and $q \bmod 4 = 3$).
  • 2
    Compute $n = p \times q$ (same structure as RSA modulus).
  • 3
    Choose a seed $s$ coprime to $n$: $\gcd(s, n) = 1$ (i.e., $p$ and $q$ are not factors of $s$).

Generation

$$X_0 = s^2 \bmod n$$ $$X_{i+1} = X_i^2 \bmod n$$ $$B_i = X_i \bmod 2 \quad \text{(the LSB of } X_i\text{)}$$

The output bit $B_i$ is the least significant bit of $X_i$. Outputting only the LSB (rather than the full $X_i$) maximizes security — the LSB is the hardest bit to predict from the factoring problem.

WORKED EXAMPLE BBS: p=11, q=19, s=3

Verify: $11 \bmod 4 = 3$ ✓, $19 \bmod 4 = 3$ ✓. $n = 11 \times 19 = 209$. $\gcd(3, 209) = 1$ ✓

$i$$X_i$Computation$B_i$ (LSB)
09$3^2 \bmod 209 = 9$1
181$9^2 \bmod 209 = 81$1
282$81^2 = 6561 \bmod 209 = 82$0
336$82^2 = 6724 \bmod 209 = 36$0

Bit stream: $1, 1, 0, 0, \ldots$

⚠️ BBS Speed

BBS is slow — each bit requires a modular squaring of a large number. It is suitable for generating cryptographic keys (where quality matters more than speed) but not for bulk pseudorandom byte generation.

🧠 Knowledge Check
Why must the BBS primes satisfy $p \equiv q \equiv 3 \pmod{4}$?
Show Answer ▾
This condition ensures that $-1$ is a quadratic non-residue modulo $n$. It is essential for the mathematical security proof of BBS — it guarantees that the quadratic residuosity problem (on which BBS security is based) is computationally hard. Without this condition, the security reduction to factoring breaks down.
Why does BBS output only the LSB of each $X_i$ instead of all bits?
Show Answer ▾
Outputting more bits per iteration would allow an attacker to recover the state $X_i$ and predict future values. The LSB is the hardest individual bit to predict from the quadratic residuosity problem. Extracting more bits reduces the work factor for an attacker. For maximum security, only the LSB per step is extracted.

True Random Number Generators — Entropy Sources Part 6

The Fundamental Difference

A PRNG must be initialized with a seed — without it, nothing works. A TRNG works without seeds because it derives randomness from physical phenomena that are genuinely unpredictable. No algorithm can provide true unpredictability. Only analog physical processes can:

  • Thermal noise in electronic components (random electron motion)
  • Radioactive decay (quantum-level randomness)
  • Atmospheric noise
  • Photon behavior in quantum optical systems

Shannon Entropy

The amount of randomness (information) in a source is quantified by Shannon entropy:

$$H = -\sum_i p_i \log_2 p_i \quad \text{(bits per word)}$$

Where $p_i$ is the probability of the $i$-th output word. For a perfectly uniform source (maximum entropy), all $N$ values have equal probability $p_i = 1/N$, giving $H = \log_2 N$ bits per word. For a byte source with uniform distribution, $H = \log_2 256 = 8$ bits — maximum.

Low entropy means the source is biased — some outputs are far more common than others — and an attacker can exploit this predictability.

Limitation of Entropy Sources

💡 Why TRNGs Can't Replace CSPRNGs

Entropy sources are inherently slow — typically a few hundred bits of entropy per second from software sources. This is far too slow for bulk cryptographic operations (AES key streams, TLS session keys). The solution: use a TRNG to seed a fast CSPRNG. A small amount of genuine entropy is stretched into a large stream of cryptographically secure pseudorandom bits.

The Boot-Time Entropy Problem

At device boot time, the entropy pool is nearly empty. The system has been running for milliseconds — barely any keystrokes, mouse events, disk operations, or network packets have occurred. Devices that generate long-term keys (like RSA private keys for SSH servers) immediately at boot may do so with near-zero entropy in their seed.

Different devices in the same batch, booting in similar states, may generate nearly identical seeds — and therefore nearly identical primes. This is precisely the mechanism behind the "Mining Ps and Qs" attack that broke 0.5% of all TLS server keys. The fix: wait for sufficient entropy accumulation, or use a hardware TRNG (like Intel RDRAND) unaffected by this.

Software & Hardware Entropy Sources Part 7

Hardware-Based: Intel DRNG

Intel's Digital Random Number Generator (DRNG), code-named "Bull Mountain," is an on-chip hardware TRNG. It uses thermal noise in transistors as its entropy source and can produce random bits at approximately 3 GHz.

Intel exposes this via the machine-code instruction RDRAND for 64-bit processors. Calling RDRAND fetches a fresh 64-bit random number from the DRNG hardware. The DRNG architecture:

  1. Analog thermal noise source → truly random bits.
  2. Conditioning circuit (AES-CBC-MAC) → whitens the raw noise.
  3. AES-CTR CSPRNG seeded from the conditioned entropy → high-throughput output.
  4. RDRAND instruction → user accesses the CSPRNG output.

Software Entropy Sources

Every computer produces phenomena with unpredictable fine-grained timing:

  • Exact nanosecond timing of keystrokes
  • Mouse movement interrupt timings
  • Disk I/O completion times
  • Network packet arrival times
  • Log file contents and system command outputs (ps, netstat)

User-Space Entropy Daemons

  • EGD (Entropy Gathering Daemon): A Perl-based daemon that gathers entropy from system events and makes it available to user-space programs via a Unix domain socket. Useful when kernel-space entropy is insufficient.
  • PRNGD (Pseudo Random Number Generator Daemon): Similar to EGD; provides a daemon interface for pseudorandom data to processes without root access to /dev/random.
✅ Practical Recommendation

For RSA key generation: use /dev/random (or hardware TRNG if available). For session keys, IVs, and nonces: /dev/urandom is sufficient and much faster. For embedded systems or devices that generate keys at boot: use a hardware TRNG like Intel RDRAND to avoid the boot-time entropy problem.

🧠 Knowledge Check
Why might /dev/random block at device boot time?
Show Answer ▾
/dev/random only outputs bits when the kernel's entropy pool has enough genuine randomness. At boot time, the pool is nearly empty — very few events (keystrokes, disk I/O, network packets) have occurred yet. If a process requests random bytes, /dev/random blocks until sufficient entropy has been accumulated. This is why generating RSA keys immediately at boot is dangerous.
What does /dev/urandom output when the entropy pool is empty?
Show Answer ▾
/dev/urandom never blocks. It uses the kernel's entropy to seed an internal CSPRNG. When the entropy pool is temporarily low, it continues producing CSPRNG output (pseudorandom, not truly random). For most purposes (session keys, IVs), this is acceptable. For long-term key generation, /dev/random or a hardware TRNG is preferred.

Review Questions

🧠 Final Knowledge Check
Explain why strong collision resistance implies weak collision resistance (connecting to the definitions).
Show Answer ▾
Strong: cannot find any two messages $M_1 \neq M_2$ with same hash (attacker chooses both freely). Weak: given fixed $M_1$, cannot find $M_2 \neq M_1$ with same hash (attacker must use the given $M_1$). Weak is strictly easier for the attacker (less freedom). If strong holds — no collision pair can be found at all — then certainly no pair starting from a fixed $M_1$ can be found. Strong ⊇ weak.
Why is the LCG dangerous for cryptographic use, and what would a CSPRNG require instead?
Show Answer ▾
The LCG has a linear structure: $X_{n+1} = (aX_n + c) \bmod m$. Knowing $m$, $a$, $c$ (usually public) and any current value makes all future values trivially predictable. A CSPRNG must satisfy "next-bit unpredictability" — even observing all past outputs gives no advantage in predicting the next bit. BBS achieves this via the quadratic residuosity assumption; X9.31 achieves it via the security of 3DES.
Describe the architecture that balances true randomness with performance in a real system.
Show Answer ▾
Use a TRNG (hardware or /dev/random) to generate a small, high-quality seed. Feed that seed into a fast CSPRNG. The CSPRNG amplifies a small amount of true entropy into a large stream of cryptographically secure pseudorandom bits. The TRNG provides the unpredictability guarantee; the CSPRNG provides the throughput. /dev/urandom is exactly this architecture in Linux.
