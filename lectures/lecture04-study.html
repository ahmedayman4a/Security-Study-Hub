📌 Three Questions This Lecture Answers
  1. Key exchange: RSA is too slow for bulk encryption. How do parties establish a session key efficiently and securely?
  2. Authenticating public keys: How does B know the public key really belongs to A and not an impostor?
  3. Forward secrecy: If an attacker records traffic today and obtains your private key tomorrow, can they decrypt everything?

Key Exchange & Man-in-the-Middle Attack Part 1

The Direct Key Exchange Protocol

RSA is too slow for encrypting large messages, but it can exchange a short symmetric session key. The direct protocol works as follows:

  • 1
    A generates a fresh RSA key pair $[PU_A, PR_A]$ and sends $PU_A$ and $ID_A$ (A's IP address) to B — unencrypted.
  • 2
    B generates a random session key $K_S$ and sends $E(PU_A, K_S)$ — the session key encrypted with A's public key.
  • 3
    A decrypts with $PR_A$ to recover $K_S$. Both parties now share $K_S$ and use it with AES for actual data.

This looks clean. But it has a fatal flaw: B has no way to verify that $PU_A$ actually belongs to A.

The MITM Attack

An adversary $V$ who can intercept messages can completely subvert the protocol without being detected:

INTENDED A → B: $PU_A$, $ID_A$
ATTACK V intercepts. B never sees this. V → B: $PU_V$, $ID_A$ (impersonating A)
INTENDED B → A: $E(PU_A, K_S)$
ATTACK V intercepts. B sent $E(PU_V, K_S)$, so V decrypts with $PR_V$ and gets $K_S$. V → A: $E(PU_A, K_S)$ (re-encrypted)

Result: A and B both have $K_S$ and believe the exchange was secure — but V also has $K_S$ and can decrypt, read, and modify all future messages.

⚠️ Root Cause

The protocol fails because public key distribution is unauthenticated. V successfully impersonated A by substituting their own public key. The fix: digital certificates — a trusted third party (CA) cryptographically binds a public key to a verified identity, making substitution detectable.

🧠 Knowledge Check
In the MITM attack on the direct key exchange, what specific step does the adversary exploit?
Show Answer ▾
The adversary intercepts A's first message (which sends $PU_A$ in plaintext) and substitutes their own public key $PU_V$ before it reaches B. Since B trusts any key it receives, it encrypts $K_S$ to $PU_V$ instead of $PU_A$, allowing V to decrypt it.

Certificates & Public Key Infrastructure (PKI) Part 2

What Is a Digital Certificate?

A digital certificate is a data structure that binds a public key to an identity, signed by a trusted third party. Specifically, A's certificate is:

$$CA = E(PR_{CA},\ [T,\ ID_A,\ PU_A])$$

where $T$ is a timestamp/validity period, $ID_A$ is A's identity, $PU_A$ is A's public key, and the whole bundle is signed with the CA's private key $PR_{CA}$. When B receives this certificate, B uses the CA's public key (which B already trusts) to verify the signature — proving that $PU_A$ genuinely belongs to $ID_A$, as vouched for by the CA.

The CA Hierarchy

Certificate Authorities are organized in a tree hierarchy:

  • Root CAs — the ultimate trust anchors (VeriSign/DigiCert, Comodo, etc.). Self-signed.
  • Intermediate CAs — certified by root CAs; can issue end-entity certificates.
  • Registration Authorities (RA) — handle administrative verification but delegate signing to a CA.

Your OS and browser come pre-loaded with the public keys of trusted root CAs. This is the foundational trust assumption: you trust that the OS/browser vendor has included legitimate root CA keys.

Certificate Validation — Chain of Trust

When your browser connects to https://example.com:

  • 1
    Browser receives the server's certificate, issued by an intermediate CA.
  • 2
    Browser looks up the intermediate CA's certificate (issued by a higher CA) — may also be sent by the server in a bundle.
  • 3
    Chain validation recurses upward until a pre-loaded root CA is reached.
  • 4
    Every signature in the chain is verified. If all are valid and the root CA is trusted, the certificate is accepted.

PKI and X.509

Public Key Infrastructure (PKI) is the complete set of standards, policies, and software for creating, distributing, using, and revoking certificates. X.509 is the most widely used certificate format standard.

An X.509 certificate's signature is created by:

  1. Computing a message digest (hash with SHA-1 or SHA-256) of all the other certificate fields.
  2. Encrypting the digest with the CA's private key.

Verification: compute the hash of the fields, decrypt the signature with the CA's public key, compare — if they match, the certificate is authentic.

Certificate Revocation

  • CRL (Certificate Revocation List): A CA-published list of certificate serial numbers that have been revoked before expiry. Clients should check this before trusting a certificate.
  • Root CA compromise: If a root CA is compromised, the only fix is a software update to all devices — affecting billions of endpoints.
  • Intermediate CA compromise: Affected certificates are added to the CRL.

Certificate Forgery

  • 2008: Weak collision resistance of MD5 was exploited to create a forged certificate. (MD5 was still used by some CAs for signatures.)
  • 2011: A rogue certificate for an intermediate CA was created, used for DNS cache poisoning and phishing.
⚠️ The Fake Root CA Attack

Creating a fake root CA with OpenSSL is technically easy. The challenge is getting parties to add it to their trusted root store — but social engineering has achieved this. Once a fake root CA is trusted, it can sign any certificate, allowing HTTPS impersonation of any domain.

🧠 Knowledge Check
Why is revoking a root CA certificate essentially impossible at scale?
Show Answer ▾
Root CA public keys are pre-loaded into all operating systems, browsers, and devices. Removing a root CA requires pushing a software update to every device worldwide — billions of endpoints. This is why root CAs are extremely carefully guarded.
What does a digital certificate contain? What is its purpose?
Show Answer ▾
A certificate contains: the owner's identity, the owner's public key, validity period, and the CA's digital signature over these fields. Its purpose is to bind a public key to a verified identity — so when you receive a public key, you can verify it really belongs to the claimed entity.

Diffie-Hellman Key Exchange Part 3

Motivation: Perfect Forward Secrecy

With RSA key exchange, if an attacker records your encrypted traffic today and later obtains the server's private key, they can decrypt all past session keys and hence all past conversations. RSA-based key exchange has no forward secrecy.

Perfect Forward Secrecy (PFS): even if long-term private keys are compromised in the future, previously established session keys remain secure. DH achieves this by having both parties independently compute the same shared secret — it is never transmitted in any form.

Mathematical Background: The Group $\mathbb{Z}_p^*$

For a prime $p$, the set $\{1, 2, \ldots, p-1\}$ forms a multiplicative group modulo $p$, denoted $\mathbb{Z}_p^*$.

  • The order of the group $\mathbb{Z}_p^*$ is $p - 1$ (number of elements).
  • The order of an element $a$ is the smallest $t$ such that $a^t \equiv 1 \pmod{p}$.
  • A generator (primitive root) $g$ generates a cyclic subgroup by $\{g^0, g^1, g^2, \ldots\} \bmod p$.
  • For DH, $g$ is chosen so its subgroup has large prime order $n$ (a large prime factor of $p-1$).
  • The triple $(p, g, n)$ is made public.

The DH Key Exchange Protocol

Setup (public): $p$ (large prime), $g$ (generator), $n$ (order of subgroup).

  • 1
    A picks random private key $X_A$ with $1 \leq X_A < n$.
  • 2
    A computes public key $Y_A = g^{X_A} \bmod p$ and sends to B.
  • 3
    B picks random private key $X_B$ with $1 \leq X_B < n$.
  • 4
    B computes public key $Y_B = g^{X_B} \bmod p$ and sends to A.
  • 5
    A computes shared secret: $K = Y_B^{X_A} \bmod p$
  • 6
    B computes shared secret: $K = Y_A^{X_B} \bmod p$

Why Both Parties Get the Same Key (Proof)

$$K_A = Y_B^{X_A} \bmod p = (g^{X_B})^{X_A} \bmod p = g^{X_B X_A} \bmod p$$ $$K_B = Y_A^{X_B} \bmod p = (g^{X_A})^{X_B} \bmod p = g^{X_A X_B} \bmod p$$

Since $X_A X_B = X_B X_A$, both arrive at $K = g^{X_A X_B} \bmod p$. The secret key was never transmitted.

WORKED EXAMPLE DH Key Exchange: $p = 17$, $g = 2$, $n = 8$

A chooses $X_A = 5$:

$$Y_A = 2^5 \bmod 17 = 32 \bmod 17 = 15$$

B chooses $X_B = 7$:

$$Y_B = 2^7 \bmod 17 = 128 \bmod 17 = 9 \quad (128 = 7 \times 17 + 9)$$

They exchange: A sends 15 to B; B sends 9 to A.

A computes:

$$K_A = 9^5 \bmod 17 = 59049 \bmod 17 = 8 \quad (59049 = 3473 \times 17 + 8)$$

B computes:

$$K_B = 15^7 \bmod 17 = 170859375 \bmod 17 = 8$$

Shared secret: $K = 8$ ✓

An eavesdropper sees $\{17, 2, 8, 15, 9\}$ but cannot recover $X_A = 5$ or $X_B = 7$ without solving a discrete logarithm.

PRACTICEDH Exchange with $p=11$, $g=2$, $X_A=3$, $X_B=7$
Using $p = 11$, $g = 2$: A chooses $X_A = 3$, B chooses $X_B = 7$. Compute $Y_A$, $Y_B$, and the shared secret $K$ (from both perspectives).

$Y_A = 2^3 \bmod 11 = 8$

$Y_B = 2^7 \bmod 11 = 128 \bmod 11 = 7$ (since $128 = 11 \times 11 + 7$)

$K_A = 7^3 \bmod 11 = 343 \bmod 11 = 2$ (since $343 = 31 \times 11 + 2$)

$K_B = 8^7 \bmod 11 = 2097152 \bmod 11 = 2$ (since $2097152 = 190650 \times 11 + 2$)

Shared secret: $K = 2$ ✓

PRACTICEDH Exchange with $p=23$, $g=5$, $X_A=6$, $X_B=15$
Using $p = 23$, $g = 5$: A chooses $X_A = 6$, B chooses $X_B = 15$. Find the shared secret.

$Y_A = 5^6 \bmod 23 = 15625 \bmod 23 = 8$ (since $15625 = 679 \times 23 + 8$)

$Y_B = 5^{15} \bmod 23 = 19$ (use successive squaring or trust the computation)

$K_A = 19^6 \bmod 23 = 2$    $K_B = 8^{15} \bmod 23 = 2$ ✓

Shared secret: $K = 2$

DH Security & DHE Part 4

The Computational DH Assumption

DH security rests on two related hardness assumptions:

1. Discrete Logarithm is Hard: Given $Y_A = g^{X_A} \bmod p$ and knowing $(g, Y_A, p)$, computing $X_A$ is infeasible for large $p$. Formally: $X_A = \log_g Y_A \bmod p$.

2. Computational DH Assumption: Even if individual discrete logs are hard, given $(g^a \bmod p)$ and $(g^b \bmod p)$, computing $g^{ab} \bmod p$ is also hard.

DH is Vulnerable to MITM

Unlike the RSA MITM (where the attacker substitutes a key), the DH MITM works differently: the adversary establishes two separate DH sessions — one with A, one with B — and relays messages between them:

  1. Adversary intercepts $Y_A$ and replaces it with their own $Y_A'$.
  2. Adversary intercepts $Y_B$ and replaces it with their own $Y_B'$.
  3. A establishes shared secret $K_1$ with the adversary (thinking it's with B).
  4. B establishes shared secret $K_2$ with the adversary (thinking it's with A).
  5. Adversary knows both keys and can relay, read, and modify everything.

Defense: Authenticated DH. Before the DH exchange, each party must verify the other's identity using certificates. DH values are digitally signed. This is what TLS does in practice.

DHE — Diffie-Hellman Ephemeral

DHE means fresh private values $X_A$ and $X_B$ are generated for every session and discarded immediately after. This provides Perfect Forward Secrecy (PFS):

  • Even if the server's long-term private key is later compromised, past sessions cannot be decrypted — the ephemeral $X_A$ values were discarded and never transmitted.
  • Contrast with static RSA key exchange where $K_S$ was sent encrypted with a long-term key; compromise of that key retroactively breaks all past sessions.

The Logjam Vulnerability

A critical weakness in deployed DHE: many servers use the same DH parameters from RFC 2412 (OAKLEY):

  • Group 1: 768-bit prime
  • Group 2: 1024-bit prime
  • Group 5: 1536-bit prime

Since so many servers use the exact same primes, an attacker can pre-compute a massive table for one specific prime, then quickly break discrete logs for all servers using that prime. Solving DL for 768-bit primes is now within academic reach; 1024-bit primes are within nation-state reach. The Logjam attack (2015) demonstrated this practically.

⚠️ Recommended Minimum

Use at least 2048-bit DH primes. Standardized safe primes of 768 or 1024 bits should no longer be used for security-critical applications.

🧠 Knowledge Check
Why does DHE provide PFS but static RSA key exchange does not?
Show Answer ▾
With DHE, fresh random private values $X_A$ and $X_B$ are generated per session and immediately discarded. The session key was never transmitted in any form. Even if the server's long-term key is later compromised, there is nothing to decrypt retroactively. With static RSA, $K_S$ was sent as $E(PU_B, K_S)$ — compromising $PR_B$ later lets an attacker decrypt all recorded session keys.
What is the Logjam vulnerability's root cause?
Show Answer ▾
Vast numbers of servers reuse the same standard DH prime numbers from RFC 2412. This allows an attacker to amortize the expensive pre-computation of discrete log tables over millions of targets — one pre-computation breaks all servers using that prime. If each server used unique primes, this attack would be impractical.

ElGamal Encryption Part 5

From Key Exchange to Encryption

DH is an interactive key exchange — both parties must be online simultaneously. ElGamal turns DH's mathematics into an asynchronous encryption scheme: B can encrypt a message to A using A's fixed public key, without A being online.

Key Setup

A has a long-term key pair based on DH parameters $(p, g, n)$:

  • Private key: $X_A$ (random, kept secret)
  • Public key: $Y_A = g^{X_A} \bmod p$
  • Full public key tuple: $(p, g, Y_A)$

Encryption (by B)

  • 1
    B generates fresh random ephemeral $X_B$ and computes $Y_B = g^{X_B} \bmod p$.
  • 2
    B computes shared secret $K = Y_A^{X_B} \bmod p$ (using A's public key).
  • 3
    B encrypts: $C = M \times K \bmod p$.
  • 4
    B sends the pair $(Y_B,\ C)$ to A.

Decryption (by A)

  • 1
    A computes $K = Y_B^{X_A} \bmod p$ (using B's ephemeral public key and A's own private key). This gives the same $K$ because $Y_A^{X_B} = g^{X_A X_B} = Y_B^{X_A}$.
  • 2
    A recovers: $M = C \times K^{-1} \bmod p$.
💡 Semantic Security

Each encryption uses a fresh $X_B$, so encrypting the same message twice gives different ciphertexts (since $K$ changes each time). This is called semantic security — the ciphertext reveals nothing about the plaintext even if the same message is encrypted repeatedly.

ElGamal Digital Signature Standard (DSS) Part 6

What Is a Digital Signature?

A digital signature proves:

  • Authentication: the message came from the claimed sender (only they have the private key).
  • Integrity: the message has not been altered since signing.
  • Non-repudiation: the signer cannot deny having signed.

The general process: compute a hash of the message → encrypt the hash with the signer's private key → attach the resulting signature to the message. To verify: decrypt with the signer's public key, compare hashes.

ElGamal DSS — Key Setup

Generate large prime $p$ and random $g, X$ with $g, X < p$:

  • Compute $Y = g^X \bmod p$
  • Public key: $(p, g, Y)$
  • Private key: $X$

Signing a Message $M$

  • 1
    Generate a fresh random $K$ with $0 < K < p-1$ and $\gcd(K, p-1) = 1$. This $K$ must be unique for every signature.
  • 2
    Compute: $$\text{sig}_1 = g^K \bmod p$$
  • 3
    Compute: $$\text{sig}_2 = K^{-1} \times (M - X \times \text{sig}_1) \bmod (p-1)$$
  • 4
    Signature: the pair $(\text{sig}_1, \text{sig}_2)$.

Verifying the Signature

Anyone with the public key $(p, g, Y)$ checks:

$$Y^{\text{sig}_1} \times \text{sig}_1^{\text{sig}_2} \equiv g^M \pmod{p}$$

If this holds, the signature is valid.

Why This Works (Proof Sketch)

If the signature is valid: $\text{sig}_2 = K^{-1}(M - X \cdot \text{sig}_1) \bmod (p-1)$, so $K \cdot \text{sig}_2 \equiv M - X \cdot \text{sig}_1 \pmod{p-1}$, giving $M \equiv X \cdot \text{sig}_1 + K \cdot \text{sig}_2 \pmod{p-1}$. By Fermat's Little Theorem:

$$g^M \equiv g^{X \cdot \text{sig}_1 + K \cdot \text{sig}_2} = (g^X)^{\text{sig}_1} \cdot (g^K)^{\text{sig}_2} = Y^{\text{sig}_1} \cdot \text{sig}_1^{\text{sig}_2} \pmod{p}$$
🚨 Critical: Never Reuse K

If the same $K$ is used to sign two different messages $M_1$ and $M_2$, the two signature equations can be combined to solve for $X$ (the private key). This is not theoretical — the PlayStation 3 was compromised in 2010 because Sony's firmware signing used the same $K$ for every signature. Always generate a truly random, unique $K$ per signature.

🧠 Knowledge Check
What is the difference between ElGamal encryption and ElGamal DSS?
Show Answer ▾
ElGamal encryption uses A's long-term public key $Y_A$ to allow B to encrypt a message to A — provides confidentiality. ElGamal DSS uses A's private key $X$ to sign a message — provides authenticity and integrity. Encryption hides content; signatures prove authorship.

The Discrete Logarithm Problem Part 7

The Problem

Given integers $g$, $k$, and prime $p$, find $s$ such that:

$$g^s \equiv k \pmod{p}$$

This is the discrete logarithm $s = \log_g k \bmod p$. Exponentiation (forward direction: given $g$, $s$, compute $k$) is easy with square-and-multiply. The inverse — given $g$, $k$, find $s$ — is believed to be computationally hard for large $p$.

Known Algorithms

AlgorithmComplexityNotes
Brute force$O(p)$Try all values of $s$
Baby-step giant-step$O(\sqrt{p})$Much better but still exponential in bit size
Pollard's rho$O(\sqrt{p})$Similar complexity, less memory
Index calculusSubexponentialBest for $\mathbb{Z}_p^*$; analogous to NFS for factoring

For $\mathbb{Z}_p^*$, the best algorithms are subexponential in the size of $p$ (not polynomial). For 768-bit primes, the discrete log is solvable by researchers. For 1024-bit primes, nation-state attackers can solve it. This is the basis of the Logjam vulnerability.

Connection to DH and ElGamal

If the discrete logarithm could be computed efficiently, an eavesdropper who sees $Y_A = g^{X_A} \bmod p$ could compute $X_A$ directly and break both DH and ElGamal. The computational hardness of this problem is what makes both protocols secure against passive eavesdropping.

💡 Analogy to RSA

RSA's security depends on integer factorization being hard. DH/ElGamal's security depends on the discrete logarithm being hard. Both are believed to be hard but neither has been formally proven so. The quantum computer threat: Shor's algorithm can solve both problems in polynomial time — which is why post-quantum cryptography is an active research area.

EXAM Q39Preventing MITM
Which mechanism prevents a man-in-the-middle attack during public key exchange?
aUsing a large modulus $n$
bShared secret key agreement
cEncryption combined with digital certificates
dSymmetric key distribution
Answer: (c). Certificates (verified by a CA) authenticate the public keys being exchanged, preventing the MITM from substituting their own public key. The MITM attack on the direct key exchange protocol succeeds precisely because there are no certificates — B cannot distinguish A's real key from V's substituted key. Encryption alone doesn't help if you're encrypting to the wrong key.
EXAM Q40CRL Purpose
What is the primary purpose of a Certificate Revocation List (CRL)?
aListing active certificates
bRegistering new certificates
cIdentifying certificates that are no longer valid
dRotating encryption keys
Answer: (c). A CRL lists certificate serial numbers that have been revoked before their natural expiration date — due to key compromise, change of ownership, etc. Clients should check the CRL before trusting a certificate. Note the distinction: CRL is about revocation (early invalidation), not expiration (natural end-of-life).
EXAM Q41Root Certificate
What distinguishes a root certificate from other certificates in a PKI hierarchy?
aIt is self-signed and serves as the ultimate trust anchor
bIt is issued by a Registration Authority
cIt has a shorter validity period
dIt uses symmetric encryption
Answer: (a). A root CA certificate is self-signed — there is no higher authority above it in the hierarchy, so it must vouch for itself. It is the ultimate trust anchor: pre-loaded in all operating systems and browsers. All other certificates in a PKI chain trace their trust back to a root certificate. Note: root CAs are not issued by RAs — RAs are subordinates of CAs, not the other way around.
EXAM Q42Certificate Contents
What does a digital certificate contain?
aThe user's private key
bOnly the CA's signature
cA symmetric encryption key
dThe owner's public key, identity, and the CA's digital signature
Answer: (d). A certificate binds a public key to an identity (name, domain), and this binding is authenticated by the CA's digital signature over both pieces of information — plus metadata like validity period and serial number. The private key is never included in a certificate; it stays exclusively with the owner. Including the private key would completely defeat the purpose of asymmetric cryptography.
EXAM Q43RA Role in PKI
What is the role of a Registration Authority (RA) in PKI?
aIt generates encryption keys on behalf of users
bIt verifies the identity of certificate applicants before the CA issues a certificate
cIt revokes certificates
dIt stores users' private keys
Answer: (b). An RA is the front-end administrative entity that handles identity verification (checking documents, domain ownership, legal records) but delegates the actual cryptographic signing of certificates to the CA. Think of it as a "reseller" that does the paperwork. The RA does not sign certificates — that is exclusively the CA's role. It does not store private keys (that would be a key escrow service, a separate concept).
EXAM Q44 ⚠️ DISPUTED DH Primary Use
Which of the following best describes the primary use of the Diffie-Hellman algorithm?
aKey exchange
bNon-repudiation
cDigital signatures
dPassword storage

Answer: (a) Key exchange.

⚠️ Questions of Doubt Clarification

This question was disputed. The answer is definitively (a) key exchange. DH does not provide non-repudiation — it does not authenticate who you are talking to, which is why it is vulnerable to MITM attacks. Non-repudiation requires proving identity (that only one specific party could have performed an action), which DH alone cannot do. Digital signatures (c) are provided by ElGamal DSS or similar, not plain DH. DH's sole purpose is to allow two parties to agree on a shared secret over an insecure channel.

EXAM Q45DH Security Basis
What property makes DH secure against passive eavesdroppers?
aThe hardness of computing discrete logarithms
bThe difficulty of factoring large integers
cThe use of symmetric encryption
dThe randomness of the key size
Answer: (a). An eavesdropper sees $Y_A = g^{X_A} \bmod p$ and $Y_B = g^{X_B} \bmod p$. To compute the shared secret $K = g^{X_A X_B} \bmod p$, they need either $X_A$ or $X_B$ — which requires solving the discrete logarithm: $X_A = \log_g Y_A \bmod p$. This is computationally infeasible for large $p$. (b) is the basis of RSA security — DH and RSA rely on different hard problems.
EXAM Q46DHE Advantage
What advantage does DHE (Diffie-Hellman Ephemeral) provide over static RSA key exchange?
aFaster computation
bLarger key sizes
cPerfect Forward Secrecy
dBetter hash functions
Answer: (c). DHE generates fresh private values $X_A$ and $X_B$ for every session, discards them immediately, and never transmits them. Even if the server's long-term private key is later compromised, there is nothing to use to recover past session keys. Static RSA key exchange sends $K_S$ as $E(PU_B, K_S)$ — if $PR_B$ is ever obtained, all past sessions are retroactively broken. PFS is the property that past sessions remain secure even after future key compromise.
EXAM Q47ElGamal Algorithm Type
What type of cryptographic algorithm is the ElGamal DSS?
aSymmetric encryption
bHash function
cStream cipher
dAsymmetric encryption / digital signature scheme
Answer: (d). ElGamal DSS is an asymmetric (public-key) algorithm. It uses a public key $(p, g, Y)$ for signature verification and a private key $X$ for signing. The security is based on the discrete logarithm problem in $\mathbb{Z}_p^*$, the same mathematical structure as DH. It is a Digital Signature Standard — used for authentication, not for symmetric encryption or hashing.
EXAM Q48ElGamal Purpose
What is the purpose of the ElGamal signature scheme?
aTo encrypt large files efficiently
bTo verify the authenticity and integrity of a message
cTo generate symmetric session keys
dTo replace AES for data encryption
Answer: (b). A digital signature proves that the message came from the claimed sender (authenticity) and has not been altered (integrity). Anyone with the signer's public key $(p, g, Y)$ can verify the signature, but only the holder of the private key $X$ can create it. ElGamal is not used for bulk file encryption (too slow; AES handles that) and does not generate session keys directly (that's DH key exchange).
