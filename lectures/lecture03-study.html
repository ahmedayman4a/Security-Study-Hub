Public-Key Cryptography Part 1

The Key Distribution Problem — The Circular Dependency

Every symmetric cipher — DES, 3DES, AES — shares a fundamental assumption that is easy to miss: both parties already share the secret key before any secure communication begins. Think carefully about what this means. Symmetric encryption solves the problem of secrecy after trust is established. It does not solve the problem of establishing that trust in the first place.

Imagine Alice wants to send Bob a confidential message. She decides to use AES. She encrypts her message — but Bob cannot decrypt without the key. She would need to tell Bob the key, but she cannot do that securely without already having a secure channel. That is a circular dependency. The key distribution problem plagued cryptography for centuries.

Public-key cryptography dissolves the problem by changing the setup entirely. Instead of requiring both parties to own the same secret before communicating, it makes one direction of the key operation safe to publish. Bob publishes a lock that anyone can snap shut — but only Bob can open. Structurally: symmetric cryptography requires a shared secret before communication; public-key crypto removes that requirement by exposing one operation safely.

Key Pairs

In public-key cryptography, every party has a key pair:

  • Public key ($PU$): Published openly. Anyone can have it. Think of it as a padlock anyone can snap shut.
  • Private key ($PR$): Kept secret. Only the owner knows it. Think of it as the key that opens the padlock.

The public key is used for locking (encryption or signature verification). The private key is used for unlocking (decryption or signing). The link between them is mathematically one-way: knowing the public key does not let you compute the private key (assuming the underlying hard problem is hard).

The Three Services — Designed Around Who Should See What

Understanding these services is easier if you treat them as design problems: who must be able to perform each operation, and who must be excluded? The protocol structure follows directly from those answers.

1. Confidentiality (A sending to B)

Goal: only Bob can read Alice's message. Who must encrypt? Anyone. Who must be excluded from decrypting? Everyone except Bob. The only operation unique to Bob is his private key. Therefore: encrypt with Bob's public key, decrypt with Bob's private key.

$$C = E(PU_B,\ M) \qquad M = D(PR_B,\ C)$$

2. Digital Signature / Authentication / Non-Repudiation

Goal: anyone can verify the message came from Alice, but only Alice could have created the signature. Who must sign? Only Alice. Who must verify? Anyone. The only operation exclusive to Alice is her private key. Therefore: sign with Alice's private key, verify with Alice's public key. It is tempting to think of this as "encrypting with the private key" — resist that framing. The goal is not encryption; it is creating a value only Alice can produce and anyone can check.

$$C = E(PR_A,\ M) \qquad M = D(PU_A,\ C)$$

3. Confidentiality + Digital Signature

Goal: secrecy and authenticity. The order matters. Alice signs first (with $PR_A$), then encrypts the signed message (with $PU_B$). Bob decrypts first (with $PR_B$), then verifies the signature (with $PU_A$).

You might ask: why not encrypt first and then sign? If Alice signed the ciphertext, Bob could verify someone signed an encrypted blob — but not that Alice committed to the underlying message. Anyone with access to the ciphertext could reattach a signature. Signing the plaintext first means Alice is committing to the actual message content.

$$C = E\!\left(PU_B,\ E(PR_A,\ M)\right) \qquad M = D\!\left(PU_A,\ D(PR_B,\ C)\right)$$

Does This Replace Symmetric Cryptography?

⚠️ Common Misconception — Hybrid is Inevitable

Public-key cryptography does not replace symmetric ciphers. The mathematical hardness that makes RSA secure is also what makes it slow — you cannot separate the two. In practice: RSA handles the narrow tasks of authentication and key exchange; AES encrypts the actual data using a session key established via RSA. Real protocols (TLS/HTTPS) always use this hybrid design. It is not a workaround — it is the intended architecture.

This is not a workaround; it is the intended architecture of every serious protocol.

RSA — Mathematical Foundation Part 2

Designing the Right Kind of Arithmetic

RSA needs two operations that undo each other — but with a catch: one must be easy for the owner to compute while the inverse must be hard for everyone else to find without a secret. Ordinary arithmetic does not give you this. If you multiply by 3, dividing by 3 undoes it — but anyone who knows the multiplier also knows the divisor. There is no asymmetry.

Modular exponentiation has the right properties — but only if the exponents are chosen carefully. Euler's theorem tells us how.

Euler's Theorem — The Exponent Cycle

Euler's theorem states: for every positive integer $n$ and every integer $a$ coprime to $n$:

$$a^{\phi(n)} \equiv 1 \pmod{n}$$

where $\phi(n)$ is Euler's totient function — the count of integers from 1 to $n$ coprime to $n$.

This means exponents wrap around in cycles of length $\phi(n)$ in the modular world defined by $n$. Raising $a$ to any multiple of $\phi(n)$ always gives 1 — the exponent cycle resets. A useful consequence: if $k = k_1 \cdot \phi(n) + k_2$, then:

$$a^k = a^{k_1 \cdot \phi(n) + k_2} = \left(a^{\phi(n)}\right)^{k_1} \cdot a^{k_2} \equiv 1^{k_1} \cdot a^{k_2} \equiv a^{k_2} \pmod{n}$$

Any multiple of $\phi(n)$ in the exponent cancels out. Only the remainder modulo $\phi(n)$ matters.

The RSA Insight: Inverses in the Exponent Cycle

Here is the central design idea. We want two operations — encrypt with exponent $e$, decrypt with exponent $d$ — such that applying both returns the original message. That requires $M^{ed} \equiv M$, which (by the cycle idea) happens when $ed \equiv 1 \pmod{\phi(n)}$, because then $ed = 1 + k\phi(n)$:

$$M^{e \cdot d} = M^{1 + k \cdot \phi(n)} = M \cdot \left(M^{\phi(n)}\right)^k \equiv M \cdot 1^k \equiv M \pmod{n}$$

So the requirement is: choose $e$ and $d$ to be multiplicative inverses modulo $\phi(n)$. Encrypting (raise to $e$) is undone by decrypting (raise to $d$) — in the exponent cycle, they compose to "do nothing." This is the complete mathematical core of RSA. Every design decision in key generation exists to make this construction possible while keeping $d$ computationally hidden from an attacker who sees only $n$ and $e$.

RSA Correctness vs. RSA Security — Two Separate Questions

Correctness (why RSA works for the owner): Euler's theorem guarantees $M^{ed} \equiv M \pmod{n}$ when $ed \equiv 1 \pmod{\phi(n)}$. The owner knows $d$ and can do this.

Security (why attackers cannot break it): to compute $d$, an attacker needs $e^{-1} \bmod \phi(n)$. To compute $\phi(n) = (p-1)(q-1)$, they need $p$ and $q$. To find $p$ and $q$ from $n$, they must factor $n$ — computationally infeasible for large primes.

These are different arguments. Euler's theorem is about correctness; factoring hardness is about security. Both must hold.

Why $n = p \cdot q$?

When $n$ is the product of two distinct primes:

$$n = p \times q \qquad \phi(n) = (p-1)(q-1)$$

This choice is the practical bridge between correctness and security — it simultaneously serves two opposite needs. For the key owner who knows $p$ and $q$: computing $\phi(n)$ is instant, finding $d$ takes milliseconds. For the attacker who sees only $n$: computing $\phi(n)$ requires knowing $p$ and $q$, which requires factoring $n$ — infeasible for large primes. The two-prime structure makes key generation easy and $d$-recovery hard. That is why no simpler choice of $n$ works.

Computing $d = e^{-1} \bmod \phi(n)$ requires knowing $\phi(n)$. Computing $\phi(n) = (p-1)(q-1)$ requires knowing $p$ and $q$. Finding $p$ and $q$ from $n$ requires factoring $n$ — computationally infeasible for large enough primes. $\phi(n)$ is the hidden lever: the owner knows it; the attacker cannot compute it.
If $p = 11$ and $q = 7$, what is $\phi(n)$?
Show Answer ▾
$\phi(n) = (p-1)(q-1) = 10 \times 6 = 60$. ($n = 77$.)

RSA Key Generation Part 3

Key Generation as a Chain of Necessities

The key generation steps look like a checklist, but each step exists because the next step requires it. Missing or weakening any step does not just reduce security — it makes the whole scheme fail. We need primes to compute $\phi(n)$; we need $\phi(n)$ to find $d$; we need $d$ to decrypt. The chain cannot be reordered.

  • 1
    Choose two large distinct primes $p$ and $q$. We need two secret primes because they let us compute $\phi(n) = (p-1)(q-1)$, which lets us find $d$. Why distinct? If $p = q$, then $n = p^2$ and an attacker computes $\sqrt{n} = p$ immediately — the "hard factoring" problem becomes a single square root. Why random? Randomness is not hygiene — it is the precondition for the hardness assumption. If two devices generate the same prime due to low entropy, a GCD scan across their public keys breaks both private keys instantly.
  • 2
    Compute the modulus: $$n = p \times q$$
  • 3
    Compute the totient: $$\phi(n) = (p-1)(q-1)$$
  • 4
    Choose the public exponent $e$ such that: $$1 < e < \phi(n) \quad \text{and} \quad \gcd(e, \phi(n)) = 1$$ The coprimality condition is not a preference — it is a mathematical requirement. A multiplicative inverse $d$ satisfying $ed \equiv 1 \pmod{\phi(n)}$ exists if and only if $\gcd(e, \phi(n)) = 1$. Choosing an $e$ that shares a factor with $\phi(n)$ means no valid $d$ exists — decryption is literally impossible. Typical values: 65537 ($= 2^{16}+1$, strongly preferred), 17, 3 (insecure). 65537 has only 2 bits set in binary ($10000000000000001_2$) — very fast exponentiation.
  • 5
    Compute the private exponent $d$: $$d = e^{-1} \bmod \phi(n)$$ $d$ is the modular multiplicative inverse of $e$ modulo $\phi(n)$, computed using the Extended Euclidean Algorithm. It satisfies $e \cdot d \equiv 1 \pmod{\phi(n)}$.
  • 6
    Public Key = $[e,\ n]$ — publish this to the world.
  • 7
    Private Key = $[d,\ n]$ — guard this with your life.
⚠️ What Must Be Kept Secret — And Why Each Leak Is Fatal

$p$ and $q$ — leaking either gives the attacker $\phi(n)$ and thus $d$ immediately.
$\phi(n)$ — leaking it directly gives $d = e^{-1} \bmod \phi(n)$.
$d$ — the private decryption exponent; knowing it means the key is fully compromised.
$n$ and $e$ are public by design.

Why $e = 3$ Is Dangerous

If $e = 3$ and a message $M$ is short enough that $M^3 < n$ (the modular reduction never triggers), then $C = M^3$ exactly. An attacker takes the cube root to recover $M$ — no factoring needed. The deeper issue: raw RSA is algebraically transparent when the input has structure. Padding (OAEP) destroys predictable structure before encryption, addressing the root cause.

Choosing $e$ — The Coprimality Condition

$\gcd(e, \phi(n)) = 1$ is equivalent to requiring:

  • $\gcd(e, p-1) = 1$ — equivalently $p \bmod e \neq 1$
  • $\gcd(e, q-1) = 1$ — equivalently $q \bmod e \neq 1$

If you want a specific $e$ (like 65537), just generate $p$ and $q$ until both conditions hold — which happens quickly in practice.

WORKED EXAMPLE RSA Key Generation — Lecture Toy Example

Given: $p = 197$, $q = 211$ (design a 16-bit RSA cipher)

Step 1–2: $n = 197 \times 211 = 41{,}567$

Step 3: $\phi(n) = (197-1)(211-1) = 196 \times 210 = 41{,}160$

Step 4 — Choose $e$ (we need $\gcd(e, \phi(n)) = 1$ so that a valid inverse $d$ exists):

  • $e = 3$: $\gcd(3, 210) = 3 \neq 1$ ✗ — 3 divides $q-1 = 210$, so no inverse $d$ exists. Decryption would be impossible.
  • $e = 5$: $\gcd(5, 210) = 5 \neq 1$ ✗ — 5 also divides $q-1$, same problem.
  • $e = 17$: $\gcd(17, 196) = 1$ ✓ and $\gcd(17, 210) = 1$ ✓ — 17 shares no factor with $p-1$ or $q-1$. A valid $d$ exists. Use $e = 17$.

Step 5 — Compute $d$: We need $17 \cdot d \equiv 1 \pmod{41160}$ — the inverse of 17 in the exponent cycle of length 41160. The Extended Euclidean Algorithm constructs this inverse (it does not merely name it).

Result: $d = 26{,}633$. Verify: $17 \times 26633 = 452761 = 11 \times 41160 + 1$ ✓ (that $+1$ is exactly $17d \equiv 1 \pmod{\phi(n)}$)

Public key: $[e = 17,\ n = 41567]$     Private key: $[d = 26633,\ n = 41567]$

WORKED EXAMPLE RSA Encryption and Decryption — Small Numbers

Given: $p = 3$, $q = 11$

$n = 33$, $\phi(n) = 2 \times 10 = 20$

Choose $e = 7$: $\gcd(7, 20) = 1$ ✓

Compute $d$: $7d \equiv 1 \pmod{20}$ → $d = 3$ (since $7 \times 3 = 21 = 1 \times 20 + 1$) ✓

Encrypt $M = 2$:

$$C = 2^7 \bmod 33 = 128 \bmod 33 = 128 - 3 \times 33 = 128 - 99 = 29$$

Decrypt $C = 29$:

$$M = 29^3 \bmod 33 = 24389 \bmod 33 = 2 \checkmark$$

(Since $24389 \div 33 = 739.06...$, $739 \times 33 = 24387$, $24389 - 24387 = 2$.)

RSA Encryption & Decryption Part 4

Messages as Numbers in a Modular World

RSA treats the message as an integer in the range $0$ to $n-1$ — a number living in the modular universe defined by $n$. Encryption is not "hiding" in the everyday sense; it is raising that number to a public exponent inside a number system where only the key owner can reliably reverse it. The entire setup of key generation — choosing $e$, computing $d$ so that $ed \equiv 1 \pmod{\phi(n)}$ — was specifically to manufacture exponents that compose back to "do nothing."

You might ask: why use exponentiation rather than some simpler invertible arithmetic — say, multiplication by $e$ modulo $n$? Because linear operations can be inverted by anyone. If $C = M \cdot e \bmod n$, then $M = C \cdot e^{-1} \bmod n$, and anyone can compute $e^{-1}$ from $e$ and $n$. Exponentiation in a modular world makes the inverse exponent $d$ computationally hidden behind the factoring problem. The choice of exponentiation is the source of the asymmetry.

$$C = M^e \bmod n \quad \text{(Encrypt with public key } [e, n]\text{)}$$

$$M = C^d \bmod n \quad \text{(Decrypt with private key } [d, n]\text{)}$$

Both operations are modular exponentiation. Correctness: $C^d = (M^e)^d = M^{ed} \equiv M \pmod{n}$ because $ed = 1 + k\phi(n)$ and $M^{\phi(n)} \equiv 1$, so the extra full cycles cancel.

Why Decryption Is Slower Than Encryption

The public exponent $e$ is chosen to be small with few set bits — for example, $e = 65537 = (10000000000000001)_2$ has only 2 bits set. Computing $M^e \bmod n$ requires at most about $2 \times 17 = 34$ multiplications using square-and-multiply.

The private exponent $d$, however, is computed as the modular inverse of $e$ and is roughly the same size as $n$ — often 2048 bits with approximately 1024 set bits. Computing $C^d \bmod n$ requires about $3 \times 2048 = 6144$ multiplications. This is why decryption is the computationally expensive operation in RSA, and why CRT is used to speed it up.

Block Size

When RSA is used as a block cipher, the block size is chosen to be smaller than $n$ — typically about half the number of bits needed to represent $n$. For example, a 1024-bit modulus allows roughly 512-bit message blocks (with padding filling the rest). The message integer $M$ must always satisfy $0 \leq M < n$.

📌 Message Padding is Essential

Before encrypting, the message is padded with random bytes to fill the entire block (using schemes like OAEP — Optimal Asymmetric Encryption Padding). This prevents attacks based on short messages, ensures the modular reduction always happens, and provides semantic security (same plaintext encrypts to different ciphertexts each time).

Fast Modular Exponentiation Part 5

Why Naive Computation Is Impossible

RSA requires computing $M^e \bmod n$. Even with $e = 65537$, the value $M^{65537}$ before reduction would have an astronomical number of digits — billions of digits for typical RSA keys. Storing or multiplying such numbers is computationally infeasible. Two ideas together solve this: reduce modulo $n$ at every step (numbers never exceed $n^2$), and decompose the exponent in binary so only $\log_2 e$ squarings are needed.

The Square-and-Multiply Algorithm — The Invariant

The invariant that makes this algorithm obvious rather than magical: at every stage, base means "the current power of $A$ I might need next" — it starts at $A^1$ and doubles (squares) at every bit. And result means "the product of all the powers I have selected so far." We accumulate into result only when the current bit is 1, because that bit says this power contributes to the binary decomposition.

Express the exponent $B$ in binary: $B = (b_k \ldots b_1 b_0)_2 = \sum_{b_i \neq 0} 2^i$, so $A^B = \prod_{b_i = 1} A^{2^i}$.

result = 1
base = A mod n
for each bit b of B (from LSB to MSB):
    if b == 1:
        result = (result × base) mod n    ← this power is in the decomposition
    base = (base × base) mod n            ← advance to next power regardless
return result

We square base because moving one bit left in binary doubles the exponent — $A^{2^{i+1}} = (A^{2^i})^2$. We reduce at every step because without it, intermediate values grow without bound — after $k$ squarings, base could have $2^k$ times as many digits as $n$. The reduction is not an optimization; it is what makes the computation feasible.

This requires at most $2 \log_2 B$ multiplications — one squaring per bit, one extra multiplication per set bit. Numbers never grow larger than $n^2$.

RSA-CRT Decryption Speedup Part 6

The Costly Computation and Its Split

RSA decryption requires $M = C^d \bmod n$ where $d$ is roughly the same size as $n$ — for 2048-bit RSA, $d$ is a 2048-bit number. Modular exponentiation with $k$-bit operands costs roughly $O(k^3)$ bit operations. At 2048 bits, this is measurable.

The Chinese Remainder Theorem (CRT) provides a ~4× speedup by splitting the decryption into two smaller exponentiations, each in a world half the size. The key owner knows $p$ and $q$, so they can work modulo $p$ and modulo $q$ separately. An attacker who does not know $p$ and $q$ cannot use this shortcut — CRT decryption is only available to the key owner.

Why CRT Works — Satisfying Two Constraints at Once

Compute two partial results in the smaller modular worlds:

$$V_p = C^d \bmod p \qquad V_q = C^d \bmod q$$

These are much cheaper because $p$ and $q$ are each half the size of $n$ (so each multiplication costs $1/4$ as much, and the exponent is also shorter via Fermat). Now we need a single number $M$ in $[0, n)$ that satisfies both $M \equiv V_p \pmod{p}$ and $M \equiv V_q \pmod{q}$ simultaneously.

The CRT combination coefficients are engineered for this. Define:

$$X_p = q \cdot (q^{-1} \bmod p) \qquad X_q = p \cdot (p^{-1} \bmod q)$$

$X_p$ has a remarkable property: $X_p \equiv 1 \pmod{p}$ and $X_p \equiv 0 \pmod{q}$. It contributes exactly 1 in the $p$-world and vanishes in the $q$-world. $X_q$ is the mirror: 0 mod $p$, 1 mod $q$. They act as "selectors" — each partial result survives only where it belongs.

$$M = (V_p \cdot X_p + V_q \cdot X_q) \bmod n$$

Check: modulo $p$, the $V_q \cdot X_q$ term vanishes (since $X_q \equiv 0 \pmod{p}$), leaving $V_p \cdot 1 = V_p$. Modulo $q$, the $V_p \cdot X_p$ term vanishes, leaving $V_q$. The formula "glues" two local answers into one global answer. Once you see CRT as satisfying two modular constraints simultaneously, the formula stops looking arbitrary.

Further Speedup: Fermat's Little Theorem (Same Principle as RSA!)

For prime $p$, $a^{p-1} \equiv 1 \pmod{p}$ — the same exponent-cycling idea from the RSA mathematical foundation, now applied in the smaller world mod $p$. Write $d = u(p-1) + v$; then $C^d \equiv C^v \pmod{p}$. The exponent shrinks from $\sim \log_2 n$ bits to $\sim \log_2 p$ bits.

Combined CRT + Fermat: roughly one-quarter the cost of naive $C^d \bmod n$.

What Is NOT Part of CRT Decryption

❌ Common Exam Trap

"Factor $n$ dynamically during decryption" is NOT part of RSA-CRT. The legitimate key holder already knows $p$ and $q$ — they used them to generate the key. Dynamically factoring $n$ is what an attacker would need to do. CRT is a speed optimization for someone who already has the secret factors, not a shortcut for someone who does not.

🧠 Knowledge Check
Why does splitting one decryption into two smaller modular worlds make the work much cheaper?
Show Answer ▾
Each modular multiplication with a $k$-bit modulus costs $O(k^2)$ bit operations. Using modulus $p \approx n^{1/2}$ costs $O((k/2)^2) = O(k^2/4)$ — 4× cheaper per multiplication. With Fermat's reduction, the exponents are also halved. Two partial exponentiations at 4× lower cost each give roughly 4× total speedup. CRT recombination is cheap compared to the exponentiations it replaces.
What do the CRT coefficients $X_p$ and $X_q$ actually do, and why are they constructed the way they are?
Show Answer ▾
$X_p = q \cdot (q^{-1} \bmod p)$ satisfies $X_p \equiv 1 \pmod{p}$ and $X_p \equiv 0 \pmod{q}$. It lets $V_p$ "survive" in the $p$-world while vanishing in the $q$-world. $X_q$ is the mirror. Together, $V_p \cdot X_p + V_q \cdot X_q$ is simultaneously $V_p \pmod{p}$ and $V_q \pmod{q}$ — the two local answers are glued into one global answer.

RSA Security — Threats & Vulnerabilities Part 7

A Map of RSA Failure Modes

RSA fails for different reasons depending on what assumption is violated. Grouping the attacks by failure type makes them easier to understand and remember than treating them as an unrelated list.

  • Mathematical hardness fails: Factoring becomes tractable — directly yields $p$, $q$, $\phi(n)$, $d$.
  • Algebraic misuse: Raw, structured inputs expose mathematical relationships that leak the message without factoring.
  • Randomness failure: The "hard" factoring problem collapses to a trivial GCD when primes share a factor due to poor entropy.
  • Protocol design limitation: RSA is used correctly as a cipher, but its role in the protocol creates a long-term confidentiality problem.

1. Mathematical Hardness — Factoring

RSA's security depends on the practical infeasibility of factoring $n = pq$. Various algorithms exist (trial division, Fermat's method, quadratic sieve, number field sieve, Pollard's rho), but none can practically factor numbers of modern RSA sizes. Importantly: "factoring is hard" is an assumption, not a theorem. It holds only when primes were generated sanely with real randomness.

2. Small Exponent / Raw RSA Attack (Algebraic Misuse)

If $e = 3$ and a message $M$ is short enough that $M^3 < n$, then $C = M^3$ exactly — no modular reduction. An attacker computes $\sqrt[3]{C}$ to recover $M$ with no cryptanalysis.

The deeper issue is not $e$'s size — it is that textbook RSA is algebraically transparent when the input has predictable structure. If the message is guessable, the attacker can encrypt all candidates and compare to $C$. Raw RSA leaks algebraic structure unless the message representation is carefully randomized.

Defense: OAEP padding randomizes the message before encryption, destroying algebraic predictability and ensuring $M^e$ always exceeds $n$. Padding addresses the root cause (algebraic structure); increasing $e$ addresses only one symptom.

3. Common Modulus Attack (Algebraic Misuse)

If two parties share the same modulus $n$ but use different coprime exponents $e_1$ and $e_2$, and the same message is encrypted under both: find $a, b$ with $e_1 a + e_2 b = 1$ (Extended Euclidean), then $C_1^a \cdot C_2^b \equiv M^{e_1 a + e_2 b} = M \pmod{n}$. The message is recovered without factoring. RSA's multiplicative algebra becomes a weapon when structural shortcuts (shared moduli) are reused. Every key pair must have its own independent modulus.

4. Low Entropy — Shared Prime Factor Attack (Randomness Failure)

If two different RSA moduli $n_1$ and $n_2$ share a common prime factor $p$, then $\gcd(n_1, n_2) = p$ — computable in milliseconds. Both private keys are immediately broken. The entire "hard factoring" claim rests on the assumption that primes were generated unpredictably. If that assumption fails, there is no mathematical barrier at all.

⚠️ Real-World Impact

A 2012 study ("Mining your Ps and Qs") harvested 5+ million TLS certificates and 4 million SSH keys. Using only GCD computations, they recovered the private keys for 0.50% of all TLS servers and 0.03% of SSH servers. Root cause: devices generating keys at boot time had low entropy (little randomness), causing some pairs of devices to generate primes with a shared factor.

WORKED EXAMPLE Shared Factor Attack

Given: $n_1 = 221$ and $n_2 = 247$

Compute $\gcd(221, 247)$:

  • $247 = 1 \times 221 + 26$
  • $221 = 8 \times 26 + 13$
  • $26 = 2 \times 13 + 0$ → $\gcd = 13$

Shared factor: $p = 13$. Factor both: $n_1 = 13 \times 17$, $n_2 = 13 \times 19$.

Both private keys are broken with a single GCD computation.

5. Perfect Forward Secrecy — A Protocol Design Limitation

When RSA is used for key exchange (Alice encrypts session key $K_S$ with Bob's public key), an eavesdropper who records all traffic can decrypt it retroactively if they ever obtain Bob's private key — through breach, legal order, or eventual advances in factoring. The long-term private key is a single point of failure for all historical traffic.

This connects to a principle from Lecture 02's key distribution protocols: long-term keys are trust roots; session secrets should be ephemeral. Diffie-Hellman (Lecture 04) generates per-session ephemeral values that are discarded after each handshake — even if the long-term authentication keys are later compromised, past sessions cannot be recovered.

Connection: Key Extraction and Shared-Factor Attacks

The shared-prime-factor attack requires extracting the modulus $n$ from each public key and computing pairwise GCDs across a large collection. This is exactly what RSA parameter extraction tools (Python cryptography library) enable. Keys are not opaque objects — they are structured data. Being able to read $n$ from a key file is the first step of a large-scale security analysis.

EXAM Q30RSA Security Foundation
What is the foundation of RSA's security?
aThe difficulty of factoring large integers
bThe hardness of solving discrete logarithms
cThe complexity of hash collisions
dThe secrecy of its S-boxes
Answer: (a). RSA security rests on integer factoring hardness. An attacker who could factor $n = pq$ would recover $p$ and $q$, compute $\phi(n) = (p-1)(q-1)$, and immediately derive $d = e^{-1} \bmod \phi(n)$. This is the security argument — distinct from the correctness argument (Euler's theorem). (b) is the basis of Diffie-Hellman/ElGamal. (d) is a symmetric cipher concept.
EXAM Q31Key Generation Order
What is the correct order for RSA key generation?
aSelect $e$ → Choose $n$ → Factor $n$ → Compute $d$
bGenerate $d$ → Find $e$ → Randomize $n$ → Pick $p, q$
cCompute $n$ → Solve for $p, q$ → Derive $e$ → Choose $d$
dChoose primes $p, q$ → Compute $n = pq$ → Select $e$ → Calculate $d$
Answer: (d). The order is a chain of necessities: $p, q$ are needed to compute $n$ and $\phi(n)$; $\phi(n)$ is needed to check $e$'s validity and compute $d$; $e$ must be chosen before $d$ because $d$ is defined as $e$'s inverse. You cannot go backward through this chain.
EXAM Q32Condition on $e$
In RSA, what condition must the public exponent $e$ satisfy?
a$e > n$
b$e$ must be prime
c$1 < e < \phi(n)$ and $\gcd(e, \phi(n)) = 1$
d$e \equiv 1 \bmod p$
Answer: (c). The coprimality condition $\gcd(e, \phi(n)) = 1$ is not a preference — it is a mathematical requirement. A modular inverse $d$ exists if and only if $\gcd(e, \phi(n)) = 1$. Without it, no valid $d$ exists and decryption is literally impossible. $e$ need not be prime — common choices (3, 17, 65537) happen to be prime for efficiency, not mathematical necessity.
EXAM Q33Computing $d$
How is the private exponent $d$ computed in RSA?
a$d \equiv e^{-1} \bmod \phi(n)$
b$d = n$
c$d = e \bmod p$
d$d$ is randomly selected
Answer: (a). $d$ is the modular multiplicative inverse of $e$ modulo $\phi(n)$, satisfying $e \cdot d \equiv 1 \pmod{\phi(n)}$. It is computed deterministically using the Extended Euclidean Algorithm — not randomly selected. $d$ cannot be derived without knowing $\phi(n)$, which requires knowing $p$ and $q$.
EXAM Q34Expensive Operation
Which RSA operation is computationally expensive?
aDecryption (using $d$)
bEncryption (using $e$)
cGenerating $p$ and $q$
dCalculating $\gcd$
Answer: (a). Decryption requires $C^d \bmod n$. The private exponent $d$ is as large as $n$ (typically 2048+ bits), requiring thousands of modular multiplications. Encryption with a small $e$ (like 65537 with only 2 set bits) requires only about 34 multiplications. CRT can speed up decryption by ~4x but it is still much slower than encryption.
EXAM Q35Why RSA Not for Bulk Encryption
Why is RSA rarely used for bulk (message content) encryption?
aLack of authentication
bSlow performance
cFixed block size
dInability to encrypt large files
Answer: (b). The mathematical hardness that makes RSA secure is the same property that makes it slow — you cannot separate the two. RSA is used only for authentication and key exchange; AES encrypts the actual traffic using a session key established via RSA. Hybrid design is the intended architecture, not a workaround.
EXAM Q36Common Modulus Attack
Which vulnerability arises from reusing the same modulus $n$ with different public exponents $e$?
aNonce reuse
bTiming attack
cPadding oracle attack
dCommon modulus attack
Answer: (d). If two parties use the same $n$ with different exponents $e_1$, $e_2$ where $\gcd(e_1, e_2) = 1$, and the same message $M$ is encrypted to both: find $a, b$ such that $e_1 a + e_2 b = 1$ (Extended Euclid). Then $C_1^a \cdot C_2^b = (M^{e_1})^a (M^{e_2})^b = M^{e_1 a + e_2 b} = M^1 = M \pmod{n}$. The message is recovered without factoring $n$.
EXAM Q37CRT Purpose
What is the primary purpose of using the Chinese Remainder Theorem (CRT) with RSA?
aTo increase the key size
bTo enable quantum resistance
cTo replace modular exponentiation
dTo speed up decryption/signature generation by approximately 4x
Answer: (d). CRT splits $C^d \bmod n$ into two smaller exponentiations mod $p$ and mod $q$. Halving the modulus reduces each multiplication to $1/4$ the cost (quadratic scaling). Combined with Fermat's Little Theorem (same exponent-cycling idea from the RSA foundation, now applied in the smaller world mod $p$), the exponents are also halved. Total speedup: approximately 4×.
EXAM Q38NOT Part of CRT Decryption
Which step is NOT part of RSA-CRT decryption?
aCompute $m_p = C^{d_p} \bmod p$ and $m_q = C^{d_q} \bmod q$
bFactor $n$ into $p$ and $q$ dynamically during decryption
cRecombine results via CRT
dVerify the result modulo $n$
Answer: (b). The private key holder already knows $p$ and $q$ — they were used when the key was generated and are stored as part of the private key. "Factor $n$ dynamically" is exactly what an attacker would need to do to break RSA. The legitimate decryptor never factors $n$ during normal operation; they simply use the pre-known $p$ and $q$.

Practice Questions — RSA Security

PRACTICEGCD scan and violated assumption
Why is a GCD scan across many public moduli such a devastating attack surface? Explain what hidden assumption in RSA key generation is violated when two moduli share a prime.

$\gcd(n_1, n_2)$ runs in time proportional to the key size — milliseconds per pair, seconds for millions of pairs. RSA's security claim says "factoring $n$ is hard." That claim assumes $p$ and $q$ were chosen randomly and independently. If poor entropy causes two devices to generate the same prime, GCD immediately produces the factorization.

The assumption was never met, so the security guarantee never applied. RSA's hardness is conditional on proper randomness — it is not a property of the mathematics itself.

PRACTICEWhy increasing e is only a partial fix
Why is "choose a larger $e$" only a partial fix for small-exponent attacks? What does OAEP padding address more fundamentally?

Larger $e$ means $M^e$ outgrows $n$ for most message sizes, ensuring modular reduction activates. But the underlying problem is not $e$'s size — it is that raw RSA is algebraically transparent when the input has predictable structure.

If the message is guessable (one of a small set of values), an attacker can encrypt all candidates and compare to $C$. Increasing $e$ addresses one symptom. OAEP padding randomizes the message before encryption, destroying the algebraic relationship between plaintext and ciphertext. Padding addresses the root cause; increasing $e$ does not.

PRACTICEWhy CRT makes decryption cheaper — explain from first principles
Why does splitting one decryption into two smaller modular worlds make the work much cheaper? Explain in terms of operand size, exponentiation cost, and how CRT reconstructs one answer from two local answers.

Each modular multiplication with a $k$-bit modulus costs $O(k^2)$ bit operations. Using modulus $p \approx n^{1/2}$ costs $O((k/2)^2) = O(k^2/4)$ — 4× cheaper. The exponents also reduce (via Fermat's Little Theorem) from $d$ to $d \bmod (p-1)$, roughly halving their length.

Two partial exponentiations mod $p$ and mod $q$, each at 4× lower cost, give the same total answer for roughly $1/4$ the work. CRT recombination finds a number $M$ that is simultaneously $V_p \pmod{p}$ and $V_q \pmod{q}$, using coefficients $X_p$ and $X_q$ that act as "selectors" — each partial result survives only in its own modular world.

"What Breaks If..." — Master Understanding List

🧠 Conceptual Depth — What Breaks If...

1. $\gcd(e, \phi(n)) \neq 1$?
No inverse $d$ exists. Decryption is mathematically impossible — not just hard. The entire scheme fails to be a cipher.

2. $p = q$ instead of distinct primes?
$n = p^2$ and $\sqrt{n} = p$ immediately. Hard factoring collapses to a single square root.

3. Modular reduction is postponed until after computing the full power?
Intermediate values grow without bound — after $k$ squarings, numbers have $2^k$ times as many digits as $n$. The computation becomes infeasible to store or multiply.

4. Two users share the same modulus $n$ with different exponents?
Common modulus attack: if the same message is encrypted under both, it is recoverable using only the Bézout identity for the two exponents. No factoring needed.

5. RSA encrypts raw short messages without padding?
If $M^e < n$, $C = M^e$ exactly and the attacker takes the $e$-th root. For guessable messages, the attacker encrypts candidates and compares.

6. RSA is used for key transport and the long-term key is later compromised?
All past sessions encrypted under that key can be decrypted retroactively. Ephemeral Diffie-Hellman prevents this.

7. Prime generation uses poor entropy and two devices share a prime factor?
$\gcd(n_1, n_2) = p$ in milliseconds. The hardness assumption was never satisfied — there is no mathematical barrier to break.

8. You know $n$ and $e$ but not $\phi(n)$?
You cannot compute $d = e^{-1} \bmod \phi(n)$. Computing $\phi(n)$ requires $p$ and $q$. $\phi(n)$ is the real hidden lever: the owner knows it; the attacker cannot compute it without factoring.
