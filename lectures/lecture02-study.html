Four Problems That Drive This Entire Lecture

Before diving into algorithms, it helps to see the shape of the whole lecture at once — because every topic below is really an answer to one of four recurring design pressures.

PROBLEM 1 — Repeated structure leaks

A block cipher is deterministic. The same key + same input = same output, always. Without a strategy, repeated plaintext blocks produce repeated ciphertext blocks — visible to any observer without breaking the cipher.

PROBLEM 2 — Reused keystreams destroy secrecy

Any two ciphertexts encrypted with the same keystream can be XOR-ed together, cancelling the key entirely and exposing \(P^{(1)} \oplus P^{(2)}\). With natural-language text, both plaintexts are recoverable. This is the two-time pad — one structural failure shared by all keystream modes.

PROBLEM 3 — Stale messages enable replay

A cryptographically valid message is just as dangerous re-sent as when first sent. Any protocol that distributes keys or authenticates identities must prove not just what is being said but when — that the message was generated specifically for this session, right now.

PROBLEM 4 — Pairwise trust doesn't scale

For \(n\) users to all share unique keys pairwise requires \(n(n-1)/2\) keys. For 1,000 users: 499,500 keys. Any deployable system needs a way to derive session keys on demand from a much smaller set of persistent secrets.

With this in mind: block cipher modes answer Problems 1 and 2. RC4 and stream ciphers explore Problem 2 further. Needham-Schroeder solves Problems 3 and 4 in their simplest form. Kerberos applies those solutions under real-world operational constraints. When a design choice seems arbitrary — why does CBC XOR before encryption? why does Kerberos split the KDC in two? — the answer is always one of these four pressures.

2DES and the Meet-in-the-Middle Attack

DES Is Too Weak — Two Ways the Obvious Fix Can Fail

DES's 56-bit key was its Achilles' heel. By 1998, dedicated hardware could crack it in under 56 hours. The simplest response: apply DES twice with two independent 56-bit keys.

$$C = E_{K_2}(E_{K_1}(P))$$ $$P = D_{K_1}(D_{K_2}(C))$$

The combined key is \(2^{112}\). A brute-force search over \(2^{112}\) is completely out of reach. So is 2DES secure? The answer is no — and there are actually two completely separate reasons it could fail. It is worth understanding both, because they are different threats and the fix for one does not address the other.

Failure Mode 1 — Is DES a Group?

The most devastating failure would be if applying DES twice always produced the same result as applying it once with some other key — if the set of DES permutations were closed under composition (forming a mathematical group). That would mean there always exists a \(K_3\) such that \(E(K_2, E(K_1, P)) = E(K_3, P)\) for all plaintexts. If true, 2DES would be completely pointless — the attacker need only search the \(2^{56}\) single-key space.

This was investigated seriously. The set of all permutations on 64-bit blocks has size \((2^{64})!\) — incomprehensibly larger than \(10^{10^{20}}\). DES generates at most \(2^{56}\) distinct permutations — a tiny speck in that space. The probability that composing two DES permutations lands on another DES permutation is negligibly small. Confirmed: DES is not a group. 2DES is genuinely different from 1DES.

Failure Mode 2 — The Meet-in-the-Middle Attack

Even knowing DES is not a group, 2DES still fails to deliver \(2^{112}\) security. The meet-in-the-middle attack reduces the effective security to \(\approx 2^{57}\) — barely more than single DES.

The key insight is geometric: 2DES computation has a natural seam. There must exist an intermediate value \(X\) satisfying both sides simultaneously:

$$X = E_{K_1}(P) \qquad \text{and} \qquad X = D_{K_2}(C)$$

The attacker does not search the \(2^{112}\) combined space. Instead, they enumerate both halves separately and look for a collision at the shared intermediate state. This is a time-memory tradeoff: spending \(2^{56}\) storage to reduce computation from \(2^{112}\) to \(\approx 2^{57}\). It applies to any construction where computation splits cleanly into two independently searchable halves. Assume the attacker has a known plaintext–ciphertext pair \((P, C)\).

1
Build the Forward Table
For every possible \(K_1\) (all \(2^{56}\) values), compute \(X = E_{K_1}(P)\) and store the pair \((X, K_1)\) in a lookup table, sorted by \(X\). Cost: \(2^{56}\) DES encryptions.
2
Search from the Right
For every possible \(K_2\) (all \(2^{56}\) values), compute \(X' = D_{K_2}(C)\) and look up \(X'\) in the table. Any match gives a candidate key pair \((K_1, K_2)\). Cost: \(2^{56}\) DES decryptions.
3
Eliminate False Alarms
There will be approximately \(2^{48}\) false matches (pairs that coincidentally produce the same \(X\) but are not the real key). Verify each candidate on a second known pair \((P', C')\). Typically only one candidate survives.

The Bottom Line — Two-Time Pad of Double Encryption
Doubling the key from 56 to 112 bits added roughly 1 bit of real security. The effective security of 2DES is \(\approx 2^{57}\) — barely more than single DES. The \(2^{112}\) apparent strength is an illusion created by the searchable seam between the two DES applications. The group question asks whether 2DES is different from 1DES; the MITM attack asks whether it is secure. Both questions must be answered.

📋 2025 MIDTERM Q14
What would be the theoretical security impact if DES were a group?
aDES would become resistant to differential cryptanalysis
bAll multiple-DES variants would collapse to single-DES security
cKey size could be reduced to 40 bits
dThe Feistel network would become redundant
Explanation
✓ Answer: (b)
If DES were closed under composition (a group), then \(E(K_2, E(K_1, P)) = E(K_3, P)\) for some \(K_3\). 2DES would be equivalent to searching the \(2^{56}\) single-DES key space. Triple DES would similarly collapse. DES is not a group — confirmed by mathematical analysis — which is why multi-DES has genuine value.
❓ What is the "intermediate value" X in the MITM attack?▾
\(X = E_{K_1}(P)\) — the output after the first DES encryption. It also equals \(D_{K_2}(C)\) — the input to the second DES going backward from the ciphertext. The attack finds the \(K_1, K_2\) pair for which both paths produce the same X.
❓ Why does the MITM attack produce ~2⁴⁸ false alarms?▾
The forward table has \(2^{56}\) X values over a \(2^{64}\)-bit output space. For each of the \(2^{56}\) backward queries, the probability of a random match is \(2^{56}/2^{64} = 2^{-8}\). Total expected false matches: \(2^{56} \times 2^{-8} = 2^{48}\). A second plaintext-ciphertext pair is used to filter these out.

3DES — EDE Mode and Backward Compatibility

Fixing 2DES: Triple Encryption

The meet-in-the-middle attack teaches a deep lesson: simply stacking block ciphers does not multiply their security. The key insight for defeating MITM is to require three layers, not two. Triple DES (3DES) applies DES three times in a carefully chosen pattern:

$$C = E_{K_3}\!\bigl(D_{K_2}\!\bigl(E_{K_1}(P)\bigr)\bigr)$$

This is EDE mode — Encrypt, then Decrypt, then Encrypt. The decryption in the middle may seem strange. Why not EEE (Encrypt–Encrypt–Encrypt)?

The Backward Compatibility Argument for EDE

Consider what happens when all three keys are set equal: \(K_1 = K_2 = K_3 = K\). Trace through step by step:

$$E_K\!\Bigl(\underbrace{D_K\!\bigl(\underbrace{E_K(P)}_{\text{produces some } X}\bigr)}_{\text{= }P}\Bigr) = E_K(P)$$

The inner \(E_K\) produces \(X\). Applying \(D_K\) immediately undoes it, recovering \(P\). The outer \(E_K\) then applies once — this is exactly single DES.

When all three keys are the same, 3DES EDE reduces exactly to single DES. This enabled a gradual, compatible migration: organizations deploying new 3DES hardware could communicate with legacy DES-only systems by using the same key three times — no protocol changes, no compatibility breaks.

With three distinct keys, 3DES achieves effective security of approximately \(2^{112}\). The meet-in-the-middle attack has been extended to 3DES with two keys (where \(K_1 = K_3 \neq K_2\)), but its cost is of order \(2^{112}\) — sufficient for most applications. 3DES was widely used until AES made it obsolete — AES is faster and provides stronger security. All new systems use AES.

❓ What is the effective security of 3DES with three distinct keys vs. two keys?▾
With three distinct keys (168-bit total key): effective security ≈ \(2^{112}\) due to the extended MITM attack. With two keys (\(K_1 = K_3 \neq K_2\), 112-bit total): also ≈ \(2^{112}\), but slightly easier to attack. Three distinct keys is stronger but 168 bits is "unwieldy" for many applications (slower, more key storage). Two-key 3DES-EDE was the common practical choice.

Block Cipher Modes — ECB

Why Modes Are Necessary

A block cipher encrypts exactly one fixed-size block. Real messages are almost never exactly that size. Worse, encrypting a long message as a series of independent blocks creates a fundamental security problem: if the same plaintext block appears multiple times (common in text, images, and structured data), it will produce the same ciphertext block every time. Pattern structure in the plaintext leaks directly into the ciphertext.

Block cipher modes of operation define how to apply a block cipher to multi-block messages. There are five standard modes: ECB, CBC, CFB, OFB, and CTR. Each makes different tradeoffs.

ECB — Electronic Codebook Mode

In ECB mode, each plaintext block is encrypted independently with the same key. No interaction exists between blocks:

$$C_i = E_K(P_i)$$ $$P_i = D_K(C_i)$$

This is the most obvious approach and the most dangerous. The cipher acts exactly like a codebook: the same plaintext block always maps to the same ciphertext block, regardless of position.

You might wonder: if AES is secure, how can this be broken? The answer is that AES itself is not broken. The attacker does not decrypt the blocks. Instead, they observe that some ciphertext blocks are identical, and conclude that the corresponding plaintext blocks are also identical. This reveals structural information about the plaintext without breaking the cipher at all.

The most famous demonstration is the "ECB penguin": encrypting a bitmap image with AES-ECB produces a ciphertext image where the outline of the original penguin is still clearly visible, because regions of uniform color produce runs of identical ciphertext blocks.

ECB Rule: Never Use for Multi-Block Messages
ECB is appropriate only for encrypting a single block or very short, non-repeating messages. For any real content, it leaks pattern information and should be considered broken.

Block Cipher Modes — CBC

Cipher Block Chaining — Designed From ECB's Failure

CBC's design logic starts directly from ECB's failure. The question is: what is the simplest change you could make to ECB that would ensure identical plaintext blocks always produce different ciphertext blocks?

You need the input to the block cipher to differ even when the plaintext is identical. The solution is to XOR the plaintext with the previous ciphertext block before feeding it to the cipher:

$$C_i = E_K(P_i \oplus C_{i-1})$$ $$P_i = D_K(C_i) \oplus C_{i-1}$$

Now the cipher's input for block \(i\) is \(P_i \oplus C_{i-1}\). Even if \(P_3 = P_{17}\), the inputs are \(P_3 \oplus C_2\) and \(P_{17} \oplus C_{16}\), which differ because \(C_2 \neq C_{16}\). The cipher sees different inputs → produces different outputs. ECB's pattern leakage is solved.

Why XOR Before Encryption, Not After?
Suppose instead you XORed after: \(C_i = E_K(P_i) \oplus C_{i-1}\). The block cipher still receives the raw plaintext \(P_i\). When \(P_3 = P_{17}\), the cipher outputs are identical — \(E_K(P_3) = E_K(P_{17})\) — and you only mask those equal outputs with different \(C_{i-1}\) values. An attacker can XOR consecutive ciphertexts to recover the chaining differences and extract structural information. XORing before is the only arrangement that changes what the cipher sees, not just what it outputs.

For the first block there is no previous ciphertext. The Initialization Vector (IV) fills this role:

$$C_1 = E_K(P_1 \oplus \text{IV}) \qquad P_1 = D_K(C_1) \oplus \text{IV}$$

The IV: Random, Not Secret

The IV must be random and unpredictable — this is the critical security requirement. It does not need to be secret; it is routinely sent alongside the ciphertext (the receiver needs it to decrypt the first block). But if the IV is predictable, an attacker who knows the first plaintext block \(P_1\) can check whether \(E_K(P_1 \oplus \text{IV})\) matches \(C_1\), leaking plaintext information.

IV Rules
  • Must be random (unpredictable) — this is the critical requirement.
  • Does NOT need to be secret — transmit it openly with the ciphertext.
  • Must NEVER be reused with the same key — reuse reveals whether the first plaintext blocks of two messages are identical.

Error Propagation in CBC

Suppose ciphertext block \(C_i\) is corrupted in transmission. What happens during decryption?

i
Block i is fully garbled
\(P_i = D_K(C_i) \oplus C_{i-1}\). Since \(C_i\) is wrong, \(D_K(C_i)\) is garbage. All bits of \(P_i\) are corrupted.
i+1
Block i+1 has one bit flipped (per corrupted bit in Cᵢ)
\(P_{i+1} = D_K(C_{i+1}) \oplus C_i\). \(D_K(C_{i+1})\) is correct, but the corrupted \(C_i\) is XORed in. Only the bits that were flipped in \(C_i\) are wrong.
i+2+
All subsequent blocks are correct
\(C_{i+1}\) and all later blocks are undamaged. Decryption proceeds correctly from block \(i+2\) onward.

Encryption: \(C_i = E_K(P_i \oplus C_{i-1})\). You need \(C_{i-1}\) before computing \(C_i\) — strictly sequential. Decryption: \(P_i = D_K(C_i) \oplus C_{i-1}\). All ciphertext blocks are already available, so you can compute \(D_K(C_i)\) for all \(i\) simultaneously, then XOR each with \(C_{i-1}\) — parallelizable.

Block Cipher Modes — CFB & OFB

CFB — Cipher Feedback Mode

CBC requires plaintext to arrive in complete blocks. But what if you need to encrypt data as it arrives — one byte at a time from a keyboard or serial port? You cannot buffer 128 bytes before encrypting. CFB solves this by converting the block cipher into a stream cipher that operates on \(s\)-bit segments (typically \(s = 8\), one byte).

The block cipher is not applied to the plaintext directly — it is used to generate a keystream. A shift register (initialized with the IV) is encrypted to produce keystream bytes; those bytes are XORed with plaintext. The resulting ciphertext byte is then fed back into the shift register:

1
Encrypt the shift register
Apply the block cipher to the \(b\)-bit shift register. Retain the leftmost \(s\) bits of the output.
2
Encrypt plaintext
XOR those \(s\) bits with the next \(s\) plaintext bits to produce \(s\) ciphertext bits.
3
Update the shift register
Shift the register left by \(s\) bits and insert the new \(s\) ciphertext bits at the right end.
4
Repeat
Go to step 1 for the next \(s\) bits of plaintext.
$$C_i = P_i \oplus \text{MSB}_s(E_K(\text{shift register}))$$

Only the encryption direction of the block cipher is ever used at both ends — no block cipher decryption is needed.

Why Does Ciphertext Feed Back, Not Plaintext?
If plaintext fed back into the shift register, the keystream for segment \(i+1\) would depend on \(P_i\) — which an attacker can observe or influence. An attacker who can choose \(P_i\) could predict the register's next state and therefore predict the next keystream segment. Ciphertext feedback makes the keystream derive from the encrypted output that both sender and receiver share, not the plaintext that only the sender knows. This independence from attacker-visible plaintext is the source of the mode's security.

Error propagation in CFB: A corrupted ciphertext segment enters the shift register and contaminates the keystream for \(b/s\) subsequent decryptions — until the corrupted segment has completely shifted out. Errors are local but cascade briefly, unlike OFB.

OFB — Output Feedback Mode

CFB achieves byte-at-a-time encryption, but because ciphertext enters the shift register, a corrupted ciphertext byte contaminates the keystream for the next several steps. For satellite links and deep-space telemetry where random bit errors are common, this cascade is unacceptable.

OFB asks the natural repair question: what if the shift register were fed by the keystream itself rather than the ciphertext? Then transmission errors could never enter the keystream generation loop.

$$O_i = E_K(O_{i-1}), \quad O_0 = \text{IV}$$ $$C_i = P_i \oplus O_i \qquad P_i = C_i \oplus O_i$$

Because the keystream \(O_i\) is generated entirely independently of the plaintext and ciphertext (the feedback loop never sees ciphertext), OFB has a crucial advantage:

OFB vs CFB — Error Isolation
In OFB, the keystream sequence \(O_1, O_2, \ldots\) is generated entirely from the key and IV — no ciphertext ever enters this loop. A corrupted ciphertext bit \(C_i\) flips exactly the corresponding plaintext bit \(P_i\), and the \(O\) sequence continues generating correctly. One bit in, one bit wrong, full stop. In CFB, the corrupted byte enters the register and poisons the keystream for \(b/s\) subsequent blocks. OFB is strictly better for noisy channels; CFB is self-synchronizing for dropped segments, which OFB is not.
What Breaks If OFB Reuses the Same IV?
Both messages are XORed with the identical keystream sequence. \(C^{(1)} \oplus C^{(2)} = P^{(1)} \oplus P^{(2)}\) — the two-time pad. This is the same structural failure as CTR nonce reuse and RC4 key reuse. The mechanism differs; the consequence is identical. Any keystream cipher fails catastrophically if the keystream is used twice — this is one principle, not three separate vulnerabilities.

❓ Why does OFB have better noise tolerance than CFB?▾
CFB: the ciphertext is fed back into the shift register. A corrupted ciphertext byte enters the register and contaminates the keystream for b/s subsequent decryptions. ❓ In which modes is only the encryption operation of the block cipher needed?▾
CFB, OFB, and CTR. In all three, the block cipher is applied to generate a keystream (either by encrypting the shift register, feeding back the output, or encrypting counter values). The plaintext or ciphertext is then XORed with the keystream — no block cipher decryption is involved.

Block Cipher Modes — CTR

Counter Mode

In CTR mode, there is no feedback from previous blocks at all. Instead, a counter is encrypted directly to produce a keystream, and each plaintext block is XORed with its corresponding keystream block:

$$\text{Keystream}_i = E_K(\text{Nonce} \| \text{Counter}_i)$$ $$C_i = P_i \oplus \text{Keystream}_i$$ $$P_i = C_i \oplus \text{Keystream}_i$$

The counter starts at an initial value (typically a nonce concatenated with 0, 1, 2, ...) and is incremented modulo \(2^b\) for each block. Since each keystream block depends only on its own counter value, all blocks are completely independent of each other.

Advantages of CTR Mode

PARALLELIZABLE

Each keystream block \(E_K(\text{Nonce}\|i)\) depends only on the counter value \(i\). All blocks can be computed simultaneously. Both encryption and decryption are fully parallelizable.

RANDOM ACCESS

To decrypt block \(j\) without decrypting preceding blocks, compute \(E_K(\text{Nonce}\|j)\) and XOR with \(C_j\). No sequential processing needed.

NO PADDING

XOR the keystream with plaintext byte-by-byte. Truncate to the exact message length. No block-alignment padding is required.

PRECOMPUTATION

The keystream can be computed before the plaintext is known. At transmission time, just XOR with the pre-computed keystream blocks.

The Critical Danger: IV Reuse

CTR mode is deadly if the same nonce (IV) is reused with the same key, even once. If two messages \(P^{(1)}\) and \(P^{(2)}\) are encrypted with the same key and same nonce:

$$C^{(1)}_i = P^{(1)}_i \oplus \text{KS}_i \quad \text{and} \quad C^{(2)}_i = P^{(2)}_i \oplus \text{KS}_i$$

The attacker computes:

$$C^{(1)}_i \oplus C^{(2)}_i = P^{(1)}_i \oplus P^{(2)}_i$$

With both plaintexts in natural language (or any structured format), the XOR can often be exploited to recover both plaintexts through "crib dragging" — guessing words in one plaintext and checking if the implied other plaintext makes sense.

The Unifying Pattern: "Generate Keystream, XOR"

CFB, OFB, CTR, and RC4 are all variations on the same core idea: generate a pseudorandom keystream, then XOR with plaintext. The block cipher (or RC4's state machine) is never applied to plaintext directly. They differ only in how the keystream is generated:

  • CFB: encrypt the shift register (which contains recent ciphertext)
  • OFB: encrypt recent keystream blocks — no ciphertext ever feeds back
  • CTR: encrypt counter values — no feedback at all, fully independent
  • RC4: output from a continuously-evolving permutation state

All four therefore share one vulnerability: keystream reuse is catastrophic (the two-time pad). This is one principle, not four. CBC does not share this vulnerability because its block cipher receives XORed plaintext as input — the chaining changes what the cipher sees, not what it outputs.

Note also that CBC's IV must be unpredictable (to prevent chosen-plaintext attacks), while CTR's nonce must be unique (to prevent keystream reuse). Different failures, different requirements.

Stream Ciphers & RC4

What is a Stream Cipher?

A stream cipher takes a different approach from block ciphers: instead of grouping plaintext into fixed blocks and encrypting each block as a unit, a stream cipher encrypts data continuously, one byte at a time. It generates a long sequence of pseudorandom bytes called a keystream, derived from the secret key. Each plaintext byte is XORed with the corresponding keystream byte:

$$C_i = P_i \oplus \text{KS}_i \qquad P_i = C_i \oplus \text{KS}_i$$

The security rests entirely on the quality of the keystream. A good keystream must be statistically indistinguishable from random: all 256 byte values should appear with equal frequency, the sequence should not repeat for a very long time, and knowledge of one byte should give no information about the next.

Stream ciphers are naturally suited to real-time communications where data arrives byte-by-byte and cannot wait for a block to fill — voice calls, video streams, and satellite communications have historically used stream ciphers. Modern practice often uses AES in CTR or OFB mode to achieve stream-cipher-like behavior with a well-analyzed block-cipher foundation.

RC4 — Rivest Cipher 4

RC4 (designed 1987, leaked publicly 1994) is one of the most widely deployed stream ciphers in history: SSL/TLS, WEP, and many other protocols used it. It is now deprecated due to statistical biases. To appreciate its design choices, start from a simpler, obviously weak alternative and ask what RC4 adds at each step.

The naive approach that fails: Initialize a 256-entry array \(S\) from the key, then walk through it in a fixed pattern — output \(S[0], S[1], \ldots, S[255], S[0], S[1], \ldots\). This fails immediately: the keystream repeats with period 256 bytes. Any message longer than 256 bytes reuses the entire keystream — the two-time pad applied to a single message.

The fix — continuously evolve the state: RC4 modifies \(S\) as it uses it, by continuously swapping entries. Each step changes two entries in the permutation, so the state at position 1000 is different from position 0. With \(256!\) possible permutations (\(\approx 10^{507}\) states), the keystream effectively never cycles. RC4's central data structure is a state vector \(S\) — an array of 256 bytes, always forming a permutation of values 0–255 (swapping never duplicates or loses values).

Phase 1: State Initialization

Fill \(S\) with values 0 through 255 in order — the identity permutation:

S[0] = 0x00, S[1] = 0x01, ..., S[255] = 0xFF

Phase 2: Build Temporary Vector T

If the key \(K\) has \(\text{keylen}\) bytes, fill a 256-element vector \(T\) by cycling through the key:

$$T[i] = K[i \bmod \text{keylen}], \quad 0 \leq i \leq 255$$

For example, with a 3-byte key \([k_0, k_1, k_2]\), the vector \(T\) repeats the key pattern 85 full times (and then once more for the final byte): \([k_0, k_1, k_2, k_0, k_1, k_2, \ldots]\).

Key Scheduling Algorithm (KSA)

The KSA transforms the identity permutation into a key-dependent scrambled one:

j = 0
for i = 0 to 255:
    j = (j + S[i] + T[i]) mod 256
    SWAP(S[i], S[j])

The index \(j\) accumulates across all 256 iterations — it is never reset. This accumulation is the critical design choice. After step \(i\), the value of \(j\) depends on all previous values of \(S\) and all previous key bytes, creating a complex interdependence. If the \(S[i]\) term were removed (so \(j = (j + T[i]) \bmod 256\)), then \(j\) would be determined entirely by the key bytes with no feedback from the evolving permutation. Each swap would become an independent, easily invertible operation, making the key setup far simpler to analyze and reverse.

Keystream Generation (PRGA)

i, j = 0
while (true):
    i = (i + 1) mod 256
    j = (j + S[i]) mod 256
    SWAP(S[i], S[j])
    k = (S[i] + S[j]) mod 256
    output S[k]    ← XOR this with plaintext byte

The output byte is \(S[k]\) where \(k = S[i] + S[j] \bmod 256\) — a third position derived from the two just-swapped values. Why not output \(S[i]\) or \(S[j]\) directly? Because \(i\) is a public counter — an observer who knows \(i\) could narrow down the state if \(S[i]\) were output. Using \(k = S[i] + S[j]\) means the observer needs to know \(j\) as well, which depends on hidden accumulated state history. The indirection hides which position of \(S\) was read.

If the PRGA removed the swap step and only advanced indices, the permutation would be fixed after initialization. The keystream would then be a fixed walk through one permutation, periodic with period at most \(256^2 = 65536\) — any message over 65536 bytes would reuse the entire keystream.

RC4 Key Reuse — The Two-Time Pad

RC4 shares the same critical vulnerability as CTR, OFB, and every other keystream cipher: never use the same key twice. Same key → same keystream. Two messages encrypted with the same keystream:

$$C^{(1)} \oplus C^{(2)} = (P^{(1)} \oplus \text{KS}) \oplus (P^{(2)} \oplus \text{KS}) = P^{(1)} \oplus P^{(2)}$$

The key cancels entirely — no cipher analysis needed. This is the two-time pad, and it is structurally identical to CTR nonce reuse or OFB IV reuse. The cipher is different; the failure is the same.

WEP used RC4 with a 24-bit IV prepended to the key: \(\text{RC4}(\text{IV} \| K)\). With only \(2^{24} \approx 16\) million possible IVs, in a busy Wi-Fi network the same IV+key combination reappears within hours, triggering keystream reuse across many packets. Combined with known statistical biases in RC4's early output (some byte positions are predictable with above-chance probability), WEP was completely broken — practical tools recovered the WEP key from captured traffic in minutes.

Needham-Schroeder Key Distribution Protocol

The Problem: Key Distribution at Scale

For \(n\) users to each share a unique secret key with every other user requires \(n(n-1)/2\) keys. For 1,000 users: 499,500 keys. As users join or leave, all those keys must be regenerated and redistributed. The system is undeployable.

A Key Distribution Center (KDC) solves the scaling problem: each user shares exactly one long-term master key with the KDC — \(n\) keys total. To communicate with any other user, you ask the KDC to generate a temporary session key for that conversation. But the KDC model immediately creates three new problems:

  1. How does Alice know the session key she received actually came from the KDC, not an impersonator?
  2. How does Bob know the key was authorized by the KDC specifically for Alice, and not fabricated by Alice herself?
  3. How can Bob be sure the key is fresh — not replayed from an old, compromised session?

Building the Protocol from First Principles

The naive approach and its failure. Suppose the KDC sent \(K_S\) directly to Alice and separately to Bob, each encrypted under their master key. Bob decrypts and gets \(K_S\) — but how does he know this came from the KDC? Alice also knows Bob's key was used to send something; she could have generated \(K_S\) herself and sent it to Bob pretending the KDC authorized it. There is no proof of KDC authorization for Bob.

The ticket idea. The fix is to include, in Alice's bundle, a sub-packet that only the KDC could have created: something encrypted under Bob's master key \(K_B\), which Alice does not have. When Bob decrypts this with \(K_B\) and finds a valid session key and Alice's identity inside, he can reason: "Only the KDC knows \(K_B\). This must have come from the KDC." Alice received this sub-packet but cannot read or modify it — she forwards it as an opaque credential. This is the ticket: \(E(K_B, [K_S, ID_A])\).

The freshness problem — the ticket alone is not enough. The ticket proves KDC authorization but carries no timestamp. An attacker who captured message 3 from an old session could replay it. Bob would receive a valid ticket and believe a fresh session is starting — with a compromised old key. To prove Alice is currently alive and holds \(K_S\), Bob generates a fresh nonce \(N_2\), encrypts it with \(K_S\), and challenges Alice. Only someone currently holding \(K_S\) can respond correctly.

Alice's own freshness need — \(N_1\). Similarly, when Alice receives the KDC's response, she needs to verify it is fresh. She includes a nonce \(N_1\) in her request; the KDC echoes it in the encrypted response. When Alice decrypts and sees \(N_1\), she knows: "This response was generated after my specific request — not a replay."

These three insights — the ticket, \(N_1\) for request freshness, and \(N_2\) for session freshness — produce the five messages below. Every message is now derivable from the problem it solves.

2

$$E(K_A,\ [K_S,\ ID_A,\ ID_B,\ N_1,\ E(K_B,\ [K_S,\ ID_A])])$$ KDC generates \(K_S\) and sends A: the session key, the original request echoed back (A verifies \(N_1\) to confirm freshness), and a ticket for B — \(E(K_B, [K_S, ID_A])\). 3
A → B: Forward the ticket
$$E(K_B,\ [K_S,\ ID_A])$$ A forwards the ticket to B. It is still encrypted under \(K_B\) — A cannot tamper with it or read the session key out of it. Only B can decrypt this.
4

$$E(K_S,\ [N_2])$$ B decrypts the ticket using \(K_B\), obtains \(K_S\) and \(ID_A\). B now has the session key but needs to verify A is really on the other end. B generates a new nonce \(N_2\) and sends it encrypted under \(K_S\). Only someone with \(K_S\) (i.e., A or the KDC) can decrypt this.
5
A → B: Response to challenge
$$E(K_S,\ [N_2 - 1])$$ A decrypts \(N_2\), applies the transformation, and sends \(N_2 - 1\) back. B verifies: (1) the sender knows \(K_S\); (2) since \(N_2\) was just generated by B, the response is fresh — no replay. Session established.
The Residual Replay Vulnerability (Denning-Sacco, 1981)
Messages 4–5 prove liveness: the sender currently holds \(K_S\). They do not prove ticket freshness. An attacker who previously compromised \(K_S\) from an old session does hold it — so they can respond correctly to Bob's \(N_2\) challenge. Bob cannot detect that the ticket was issued years ago. This attack was discovered three years after the protocol was published. The fix — used in Kerberos — is to embed a timestamp in the ticket and reject stale ones on receipt.

Kerberos

Needham-Schroeder Under Operational Pressure

Needham-Schroeder proves the central idea. Under the operational demands of a large real network, it has four concrete problems that make it undeployable as-is:

  1. KDC must be involved in every session. Every service access requires a round-trip to the KDC.
  2. Password-derived key exercised repeatedly. Every KDC contact uses \(K_\text{Client}\), the most sensitive key in the system.
  3. The replay vulnerability. A compromised old session key allows replaying old tickets through the challenge-response.
  4. No single sign-on. The user must re-authenticate for every individual service.

Kerberos (MIT, 1980s) addresses all four with three architectural decisions. Understanding each decision as the solution to a specific problem is what makes Kerberos feel inevitable rather than arbitrary.

Design Decision 1: The Ticket-Granting Ticket (Solves Problems 1, 2, and 4)

The expensive, password-involving step is the initial authentication — proving who you are. That only needs to happen once per login. What if the KDC, after authenticating you once, issued a reusable credential that lets you request further credentials — without re-authenticating?

This is the Ticket-Granting Ticket (TGT). The client authenticates to the Authentication Server (AS) using \(K_\text{Client}\) (derived from the password). AS issues a TGT (encrypted under the TGS's key — opaque to the client) and a session key \(K_\text{Client-TGS}\). For all subsequent service access, the client presents the TGT to the Ticket-Granting Server (TGS) and uses \(K_\text{Client-TGS}\) — never the password-derived key again. This is single sign-on: one password entry, unlimited service access within the TGT's validity period.

The AS/TGS split also has a direct security benefit: \(K_\text{Client}\) is only ever touched at the AS interaction at login. All subsequent operations use session keys. A compromise of TGS does not expose any client's password-derived key. Combining AS and TGS into one server would make the more-frequently-accessed ticket-granting path also the path that handles password-derived keys — a larger attack surface for the most sensitive credential.

Design Decision 2: Timestamps Instead of Nonces (Solves Problem 3)

Needham-Schroeder uses a nonce challenge-response (messages 4–5) to prove freshness. This adds a round-trip and — critically — only proves liveness, not ticket freshness. Kerberos replaces this with timestamps embedded in every ticket and every authenticator:

  • The ticket contains an issue time and validity period (e.g., 8 hours).
  • The authenticator — sent alongside the ticket — contains the client's current timestamp, encrypted under the session key.
  • TGS and Service Providers reject authenticators with timestamps more than ±5 minutes old.

This eliminates the nonce challenge-response and directly addresses ticket staleness. The price is real: Kerberos requires synchronized clocks. If a clock drifts more than 5 minutes, authentication fails. NTP (Network Time Protocol) becomes a security-critical dependency — not just a convenience. An attacker who can manipulate a server's clock can reopen replay windows. Needham-Schroeder avoided this assumption entirely by using nonces. Kerberos trades that freedom for fewer round-trips and stronger replay protection.

What Breaks If Timestamps Are Removed From Authenticators?
Any captured authenticator becomes permanently valid — the server cannot distinguish a fresh one from one recorded in a previous session. The TGT's validity period still expires, but within that window any captured authenticator can be replayed indefinitely. Without timestamps, Kerberos degrades to exactly the replay vulnerability that Needham-Schroeder has.
What Breaks If K_Client Is Used for Every Service Exchange?
\(K_\text{Client}\) (the password-derived key) would be exercised on every network interaction. A single interception — a captured authentication attempt, a timing side-channel — gives the attacker indefinite access to all Kerberos services for that user. The AS/TGS separation confines \(K_\text{Client}\) exposure to one moment at login.
NS Tickets vs Kerberos TGT — Same Pattern, Generalized
The Kerberos TGT is the Needham-Schroeder ticket idea generalized. In NS, \(E(K_B, [K_S, ID_A])\) is an opaque credential: A carries it to B but cannot read it; only B can decrypt it, verifying KDC authorization. The TGT \(E(K_\text{TGS}, [\ldots])\) works identically — the client carries it opaquely to TGS. Kerberos adds a two-level hierarchy (TGT for TGS, service tickets for services) and the SSO reuse property, but the fundamental opaque-credential pattern is unchanged.
❓ Why does Kerberos separate AS from TGS? Could one server do both?▾
A single server could technically handle both. The separation exists for two reasons. First, scalability and SSO: the client authenticates to AS only once per login, then uses the TGT for all TGS interactions without re-authenticating. Second, security: AS is the only server that ever touches \(K_\text{Client}\) (the password-derived key). TGS only sees session keys. A compromise of TGS cannot expose client passwords. If combined, the more-frequently-accessed ticket-granting path would also handle the most sensitive key.
❓ How does Kerberos prevent a stolen TGT from being used indefinitely?▾
TGTs embed an explicit validity period (e.g., 8 hours). TGS checks the current time against this window on every request and rejects expired TGTs unconditionally. A stolen TGT is therefore useful for at most 8 hours from issuance. Higher-security environments set shorter validity periods — trading convenience for a smaller exposure window.
❓ Kerberos uses timestamps; Needham-Schroeder uses nonces. What is the tradeoff?▾
Timestamps: eliminate the nonce challenge-response round-trip, directly address ticket staleness, and prevent replay of both tickets and authenticators. Cost: all realm clocks must be synchronized within ±5 minutes — NTP becomes a security dependency. Nonces: no clock synchronization required, but an extra round-trip is needed per session, and ticket staleness is not directly prevented. Kerberos trades clock dependency for fewer messages and stronger replay protection.
❓ What is the single sign-on (SSO) property of Kerberos?▾
The client authenticates to AS once per login session and receives a TGT. For each service it needs, the client presents the TGT to TGS and receives a service-specific ticket — using \(K_\text{Client-TGS}\), not the password-derived key. The user never re-enters a password for individual services. AS is contacted only at login; TGS handles all subsequent service authorizations.

Understanding Questions

❓ Why does doubling DES's key length fail to double its security? Name the general pattern.▾
The computation splits at a natural seam — the intermediate state between the two DES applications. The MITM attack enumerates both halves independently (\(2^{56}\) each) and matches at the collision point, costing \(\approx 2^{57}\) total. This is a time-memory tradeoff: \(2^{56}\) storage buys an exponential reduction in computation time. The seam created by sequential double encryption makes the full \(2^{112}\) space impossible to require of an attacker. This pattern applies to any double-encryption construction where the two stages are independently invertible.
❓ Why does CBC fix ECB's pattern-leakage specifically by XORing before encryption rather than after?▾
XORing before changes what the cipher sees: even if \(P_i = P_j\), their cipher inputs \((P_i \oplus C_{i-1})\) and \((P_j \oplus C_{j-1})\) differ because \(C_{i-1} \neq C_{j-1}\). XORing after would leave the cipher receiving the raw plaintext — identical inputs still produce identical outputs — and only mask the results. The masking could be reversed by XOR-ing consecutive ciphertexts. The fix must change the cipher's input, not its output.
❓ RC4 key reuse is "structurally the same failure" as CTR nonce reuse. Prove it with algebra.▾
Both produce the same keystream for the same key/nonce. Given \(C^{(1)} = P^{(1)} \oplus \text{KS}\) and \(C^{(2)} = P^{(2)} \oplus \text{KS}\): \(C^{(1)} \oplus C^{(2)} = P^{(1)} \oplus P^{(2)}\). The keystream cancels entirely — no cipher analysis needed. The cipher's internals are irrelevant. This is the two-time pad, and it applies to any keystream cipher. The name changes; the algebra is identical.
❓ Walk through Needham-Schroeder without N₁, then without N₂. What attack does each removal enable?▾
Without \(N_1\): Alice's request contains no fresh value. An attacker who captured a previous valid KDC response can replay it. Alice decrypts an old, potentially compromised session key, believing it is fresh. Without \(N_2\): Bob accepts the ticket alone. An attacker who captured message 3 from any prior session replays it. Bob extracts the old session key and believes a fresh session is starting. If the attacker also holds the old \(K_S\), they can communicate with Bob while Bob believes he is talking to Alice.
❓ Kerberos uses timestamps; Needham-Schroeder uses nonces. What does each stop, and what does each cost?▾
Timestamps stop: replay of stale tickets and authenticators (any message outside the clock window is rejected), and ticket staleness is directly addressed. Cost: clocks across the realm must be synchronized within ±5 minutes — NTP is now a security dependency, and clock manipulation reopens replay windows. Nonce challenge-response stops: replay of the ticket by someone who does not currently hold \(K_S\). Does not stop replay by someone who does hold old \(K_S\) (the Denning-Sacco attack). Cost: one extra round-trip per session. Kerberos trades the round-trip and the clock-free assumption for stronger protection.
❓ Why is reusing long-term keys acceptable in Kerberos while reusing session keys or nonces is dangerous?▾
Long-term keys are the persistent trust anchors of the system — they are designed to be reused. Their security comes from secrecy, maintained by careful management. Session keys and nonces are freshness mechanisms — their security depends on being unique and non-repeatable. A reused session key defeats isolation: compromise of one session compromises all sharing that key. A reused nonce defeats the replay protection it was designed to provide. The categories serve fundamentally different security roles, and their requirements derive from those roles.

"What Breaks If..." — Master List

Use these to test your deepest understanding. Each answer is explained in the corresponding section.

❓ What breaks if 3DES uses EEE (not EDE) with K₁ = K₂ = K₃?▾
Backward compatibility is lost. EEE with equal keys does not reduce to single DES (because DES is not a group: \(E_K(E_K(E_K(P))) \neq E_K(P)\) in general). Legacy DES hardware cannot interoperate with EEE-based 3DES. The EDE arrangement was chosen specifically so that the degenerate same-key case collapses cleanly to exactly single DES.
❓ What breaks if CBC XORs with the previous ciphertext AFTER encryption instead of before?▾
The block cipher still receives raw plaintext. Identical plaintext blocks still produce identical cipher outputs. The post-encryption XOR masks those equal outputs with different chaining values, but an attacker can XOR consecutive ciphertexts to recover those differences. ECB's core problem — identical inputs to the cipher — survives. XOR must happen before to change what the cipher sees.
❓ What breaks if CBC uses a fixed IV (always zero) with the same key?▾
Two messages with identical first plaintext blocks produce identical first ciphertext blocks, leaking that the messages share a common prefix. Against a chosen-plaintext attacker, a predictable IV enables verifying guesses about plaintext content (BEAST attack in TLS 1.0).
❓ What breaks if CFB feeds back plaintext instead of ciphertext?▾
The keystream for segment \(i+1\) depends on \(P_i\), which an attacker can observe. An attacker who can choose or predict \(P_i\) can predict the register contents and therefore the next keystream segment, compromising confidentiality of all subsequent bytes.
❓ What breaks if OFB or CTR reuse their IV/nonce for two messages?▾
Both messages are XORed with the identical keystream. \(C^{(1)} \oplus C^{(2)} = P^{(1)} \oplus P^{(2)}\) — the two-time pad. Both plaintexts are recoverable via crib-dragging without breaking the cipher. This is one structural failure shared by all keystream modes, not two separate issues.
❓ What breaks if RC4's KSA removes S[i] from the j update?▾
\(j\) is determined entirely by key bytes with no feedback from the evolving permutation. The 256 swaps become independent, each computable from one key byte alone. The key setup provides far weaker obfuscation — easier to analyze and invert.
❓ What breaks if RC4's PRGA stops swapping?▾
The permutation is fixed after initialization. The keystream becomes a fixed walk through one permutation, periodic with period at most \(256^2 = 65536\). Any message over 65536 bytes reuses the entire keystream — the two-time pad inflicted on a single message.
❓ What breaks if Needham-Schroeder omits the challenge-response (messages 4–5)?▾
An attacker who captured message 3 (the ticket) from any old session can replay it. Bob extracts the old session key and believes a fresh session is starting. With the old \(K_S\), the attacker can communicate with Bob while Bob believes he is talking to Alice. There is no liveness proof.
❓ What breaks if Kerberos removes timestamps from authenticators?▾
Captured authenticators can be replayed indefinitely. The server cannot distinguish a fresh one from one recorded in a previous session. Kerberos degrades to the same replay vulnerability as Needham-Schroeder without the nonce challenge-response.

Comprehensive Review Questions

💡 REVIEW
An attacker captures a 2DES-encrypted message along with a known plaintext-ciphertext pair (P, C). The attacker builds a forward table from P and a backward table from C. How many false alarms should the attacker expect, and how do they resolve them?
Explanation
✓ ~2⁴⁸ false alarms
The forward table has \(2^{56}\) entries over a \(2^{64}\)-bit space. For each of the \(2^{56}\) backward queries, the probability of a random match is \(2^{56}/2^{64} = 2^{-8}\). Expected false matches: \(2^{56} \times 2^{56} / 2^{64} = 2^{48}\). To resolve: test each candidate \((K_1, K_2)\) on a second known pair \((P', C')\). The probability of a false alarm surviving is \(2^{-64}\) per additional pair, so one or two additional pairs typically leave only the real key.
💡 REVIEW
A developer encrypts user session tokens using AES-ECB. A security researcher claims they can detect when two different users have the same role (admin, user, guest) embedded in their token without breaking AES. Explain how and propose a fix.
Explanation
✓ ECB pattern leakage
If the role string ("admin", "user") is aligned to a 128-bit block boundary, identical role fields produce identical ciphertext blocks. The researcher collects tokens from multiple users, identifies repeated ciphertext blocks, and concludes those users share the same role value. Fix: use CBC or CTR mode with a fresh random IV per token. The IV ensures that even identical plaintext blocks produce different ciphertext, eliminating the detectable pattern.
💡 REVIEW
In a Kerberos deployment, the TGS server is compromised by an attacker. What can the attacker do, and what are they prevented from doing?
Explanation
✓ Limited damage by AS/TGS separation
The attacker gains access to \(K_\text{TGS}\) (used to encrypt TGTs) and \(K_\text{SP}\) values (used to issue service tickets). They can forge service tickets for any service provider and impersonate clients to services. However, they do NOT have access to clients' long-term keys \(K_\text{Client}\) (derived from passwords) — these are known only to AS and each individual client. So they cannot impersonate the KDC to clients, cannot obtain clients' passwords, and cannot issue new TGTs. The AS/TGS separation limits the damage scope.
