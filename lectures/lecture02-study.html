2DES and the Meet-in-the-Middle Attack

DES Is Too Weak — The Obvious Fix

DES's 56-bit key was its Achilles' heel. By 1998, dedicated hardware could crack it in under 56 hours. The simplest response: apply DES twice with two independent 56-bit keys, so the combined key is 112 bits.

$$C = E_{K_2}(E_{K_1}(P))$$ $$P = D_{K_1}(D_{K_2}(C))$$

The key space is \(2^{56} \times 2^{56} = 2^{112}\). A brute-force attack over \(2^{112}\) is completely out of reach. So is 2DES secure? The answer is a surprising no — and the reason reveals an elegant structural vulnerability.

Is DES a Group? (A Prerequisite Question)

Before discussing the attack, consider an even more fundamental question: when you apply DES twice with two keys, do you get something genuinely different from a single DES application? Or could there always be a third key \(K_3\) such that \(E(K_2, E(K_1, P)) = E(K_3, P)\)?

If such a \(K_3\) always existed — if the set of all DES permutations were closed under composition (forming a group) — then 2DES would be completely pointless. Double-encrypting would be equivalent to encrypting once with some other key. The attacker would only need to search the \(2^{56}\) space of single-DES keys.

This was investigated seriously. The set of all permutations on 64-bit blocks has size \((2^{64})!\), which is incomprehensibly larger than \(10^{10^{20}}\). The DES-generated permutations number at most \(2^{56} \approx 10^{17}\) — a tiny speck in that vast space. The probability that composing two DES permutations accidentally lands on another DES permutation is negligibly small. It has been mathematically confirmed: DES is not a group. 2DES is genuinely stronger than 1DES.

The Meet-in-the-Middle Attack

Even though 2DES is not equivalent to 1DES, it still fails to deliver \(2^{112}\) security. The meet-in-the-middle attack reduces the effective security to approximately \(2^{57}\).

The key insight is that there exists an intermediate value \(X\) — the output after the first DES and the input to the second DES — that satisfies both:

$$X = E_{K_1}(P) \quad \text{and} \quad X = D_{K_2}(C)$$

The attacker approaches the problem from both sides simultaneously and looks for a match at this middle value. Assume the attacker has a known plaintext–ciphertext pair \((P, C)\).

1
Build the Forward Table
For every possible \(K_1\) (all \(2^{56}\) values), compute \(X = E_{K_1}(P)\) and store the pair \((X, K_1)\) in a lookup table, sorted by \(X\). Cost: \(2^{56}\) DES encryptions.
2
Search from the Right
For every possible \(K_2\) (all \(2^{56}\) values), compute \(X' = D_{K_2}(C)\) and look up \(X'\) in the table. Any match gives a candidate key pair \((K_1, K_2)\). Cost: \(2^{56}\) DES decryptions.
3
Eliminate False Alarms
There will be approximately \(2^{48}\) false matches (pairs that coincidentally produce the same \(X\) but are not the real key). Verify each candidate on a second known pair \((P', C')\). Typically only one candidate survives.

The Bottom Line
Doubling the key from 56 to 112 bits added essentially no real security. The effective security of 2DES is \(\approx 2^{57}\) — barely more than single DES at \(2^{56}\). The \(2^{112}\) apparent strength is an illusion created by the double-encryption structure.

bAll multiple-DES variants would collapse to single-DES security

Explanation
✓ Answer: (b)
If DES were closed under composition (a group), then \(E(K_2, E(K_1, P)) = E(K_3, P)\) for some \(K_3\). 2DES would be equivalent to searching the \(2^{56}\) single-DES key space. Triple DES would similarly collapse. DES is not a group — confirmed by mathematical analysis — which is why multi-DES has genuine value.
❓ What is the "intermediate value" X in the MITM attack?▾
\(X = E_{K_1}(P)\) — the output after the first DES encryption. It also equals \(D_{K_2}(C)\) — the input to the second DES going backward from the ciphertext. The attack finds the \(K_1, K_2\) pair for which both paths produce the same X.
❓ Why does the MITM attack produce ~2⁴⁸ false alarms?▾
The forward table has \(2^{56}\) X values over a \(2^{64}\)-bit output space. For each of the \(2^{56}\) backward queries, the probability of a random match is \(2^{56}/2^{64} = 2^{-8}\). Total expected false matches: \(2^{56} \times 2^{-8} = 2^{48}\). A second plaintext-ciphertext pair is used to filter these out.

3DES — EDE Mode and Backward Compatibility

Fixing 2DES: Triple Encryption

The meet-in-the-middle attack teaches a deep lesson: simply stacking block ciphers does not multiply their security. The key insight for defeating MITM is to require three layers, not two. Triple DES (3DES) applies DES three times in a carefully chosen pattern:

$$C = E_{K_3}\!\bigl(D_{K_2}\!\bigl(E_{K_1}(P)\bigr)\bigr)$$

This is EDE mode — Encrypt, then Decrypt, then Encrypt. The decryption in the middle may seem strange. Why not EEE (Encrypt–Encrypt–Encrypt)?

The Backward Compatibility Argument for EDE

Consider what happens when all three keys are set equal: \(K_1 = K_2 = K_3 = K\). Trace through step by step:

$$E_K\!\Bigl(\underbrace{D_K\!\bigl(\underbrace{E_K(P)}_{\text{produces some } X}\bigr)}_{\text{= }P}\Bigr) = E_K(P)$$

The inner \(E_K\) produces \(X\). Applying \(D_K\) immediately undoes it, recovering \(P\). The outer \(E_K\) then applies once — this is exactly single DES.

When all three keys are the same, 3DES EDE reduces exactly to single DES. This enabled a gradual, compatible migration: organizations deploying new 3DES hardware could communicate with legacy DES-only systems by using the same key three times — no protocol changes, no compatibility breaks.

With three distinct keys, 3DES achieves effective security of approximately \(2^{112}\). The meet-in-the-middle attack has been extended to 3DES with two keys (where \(K_1 = K_3 \neq K_2\)), but its cost is of order \(2^{112}\) — sufficient for most applications. 3DES was widely used until AES made it obsolete — AES is faster and provides stronger security. All new systems use AES.

Three distinct keys is stronger but 168 bits is "unwieldy" for many applications (slower, more key storage). Two-key 3DES-EDE was the common practical choice.

Block Cipher Modes — ECB

Why Modes Are Necessary

A block cipher encrypts exactly one fixed-size block. Real messages are almost never exactly that size. Worse, encrypting a long message as a series of independent blocks creates a fundamental security problem: if the same plaintext block appears multiple times (common in text, images, and structured data), it will produce the same ciphertext block every time. Pattern structure in the plaintext leaks directly into the ciphertext.

Block cipher modes of operation define how to apply a block cipher to multi-block messages. There are five standard modes: ECB, CBC, CFB, OFB, and CTR. Each makes different tradeoffs.

ECB — Electronic Codebook Mode

In ECB mode, each plaintext block is encrypted independently with the same key. No interaction exists between blocks:

$$C_i = E_K(P_i)$$ $$P_i = D_K(C_i)$$

This is the most obvious approach and the most dangerous. The cipher acts exactly like a codebook: the same plaintext block always maps to the same ciphertext block, regardless of position.

You might wonder: if AES is secure, how can this be broken? The answer is that AES itself is not broken. The attacker does not decrypt the blocks. Instead, they observe that some ciphertext blocks are identical, and conclude that the corresponding plaintext blocks are also identical. This reveals structural information about the plaintext without breaking the cipher at all.

The most famous demonstration is the "ECB penguin": encrypting a bitmap image with AES-ECB produces a ciphertext image where the outline of the original penguin is still clearly visible, because regions of uniform color produce runs of identical ciphertext blocks.

ECB Rule: Never Use for Multi-Block Messages
ECB is appropriate only for encrypting a single block or very short, non-repeating messages. For any real content, it leaks pattern information and should be considered broken.

Block Cipher Modes — CBC

Cipher Block Chaining

In CBC mode, each plaintext block is XORed with the previous ciphertext block before encryption. This creates a dependency chain: each ciphertext block depends on all previous plaintext blocks.

$$C_i = E_K(P_i \oplus C_{i-1})$$ $$P_i = D_K(C_i) \oplus C_{i-1}$$

For the first block, there is no previous ciphertext. An Initialization Vector (IV) fills this role as \(C_0\):

$$C_1 = E_K(P_1 \oplus \text{IV})$$ $$P_1 = D_K(C_1) \oplus \text{IV}$$

Because each block's encryption depends on all previous ciphertext, identical plaintext blocks at different positions produce different ciphertext blocks (since \(C_{i-1}\) differs). CBC solves ECB's pattern-leakage problem.

The IV: Random, Not Secret

The IV must be random and unpredictable — this is the critical security requirement. It does not need to be secret; it is routinely sent alongside the ciphertext (the receiver needs it to decrypt the first block). But if the IV is predictable, an attacker who knows the first plaintext block \(P_1\) can check whether \(E_K(P_1 \oplus \text{IV})\) matches \(C_1\), leaking plaintext information.

IV Rules
  • Must be random (unpredictable) — this is the critical requirement.
  • Does NOT need to be secret — transmit it openly with the ciphertext.
  • Must NEVER be reused with the same key — reuse reveals whether the first plaintext blocks of two messages are identical.

Error Propagation in CBC

Suppose ciphertext block \(C_i\) is corrupted in transmission. What happens during decryption?

i
Block i is fully garbled
\(P_i = D_K(C_i) \oplus C_{i-1}\). Since \(C_i\) is wrong, \(D_K(C_i)\) is garbage. All bits of \(P_i\) are corrupted.
i+1
Block i+1 has one bit flipped (per corrupted bit in Cᵢ)
\(P_{i+1} = D_K(C_{i+1}) \oplus C_i\). \(D_K(C_{i+1})\) is correct, but the corrupted \(C_i\) is XORed in. Only the bits that were flipped in \(C_i\) are wrong.
i+2+
All subsequent blocks are correct
\(C_{i+1}\) and all later blocks are undamaged. Decryption proceeds correctly from block \(i+2\) onward.

❓ In CBC mode, why can decryption be parallelized but encryption cannot?▾
Encryption: \(C_i = E_K(P_i \oplus C_{i-1})\). You need \(C_{i-1}\) before computing \(C_i\) — strictly sequential. Decryption: \(P_i = D_K(C_i) \oplus C_{i-1}\). All ciphertext blocks are already available, so you can compute \(D_K(C_i)\) for all \(i\) simultaneously, then XOR each with \(C_{i-1}\) — parallelizable.

Block Cipher Modes — CFB & OFB

CFB — Cipher Feedback Mode

Both CFB and OFB convert a block cipher into a stream cipher — they let you encrypt data one segment (e.g., one byte) at a time, without waiting for a full block to accumulate. This is useful when you cannot pad the message to a block boundary, or when data arrives byte-by-byte in real time.

In CFB mode, a shift register (initialized with the IV) feeds into the block cipher. The output is used to encrypt one byte (or \(s\) bits) of plaintext. The resulting ciphertext byte is then shifted into the register:

1
Encrypt the shift register
Apply the block cipher to the \(b\)-bit shift register. Retain the leftmost \(s\) bits of the output.
2
Encrypt plaintext
XOR those \(s\) bits with the next \(s\) plaintext bits to produce \(s\) ciphertext bits.
3
Update the shift register
Shift the register left by \(s\) bits and insert the new \(s\) ciphertext bits at the right end.
4
Repeat
Go to step 1 for the next \(s\) bits of plaintext.
$$C_i = P_i \oplus \text{MSB}_s(E_K(\text{shift register}))$$

Key property: only the encryption direction of the block cipher is used, at both sending and receiving ends. No decryption operation is required of the underlying cipher.

Error propagation in CFB: A corrupted \(s\)-bit ciphertext segment enters the shift register and stays there for \(b/s\) steps, contaminating the keystream for those steps. After the corrupted segment has been completely shifted out, the error stops propagating. Errors are local but affect more than one segment (unlike OFB, as we will see).

OFB — Output Feedback Mode

OFB mode is nearly identical to CFB, with one critical change: instead of feeding back the ciphertext into the shift register, OFB feeds back the output of the block cipher encryption (the keystream itself) — before XOR with the plaintext.

$$O_i = E_K(O_{i-1}), \quad O_0 = \text{IV}$$ $$C_i = P_i \oplus O_i \qquad P_i = C_i \oplus O_i$$

Because the keystream \(O_i\) is generated entirely independently of the plaintext and ciphertext (the feedback loop never sees ciphertext), OFB has a crucial advantage:

OFB vs CFB Error Property
In OFB, a corrupted ciphertext bit corrupts only the corresponding plaintext bit during decryption. The keystream generation is unaffected (it never sees the ciphertext). This makes OFB preferable over CFB for noisy transmission channels where bit errors are common.

❓ Why does OFB have better noise tolerance than CFB?▾
CFB: the ciphertext is fed back into the shift register. A corrupted ciphertext byte enters the register and contaminates the keystream for b/s subsequent decryptions. OFB: only the keystream (never the ciphertext) is fed back. A corrupted ciphertext bit flips only the corresponding plaintext bit — the keystream generation is completely unaffected.
❓ In which modes is only the encryption operation of the block cipher needed?▾
CFB, OFB, and CTR. In all three, the block cipher is applied to generate a keystream (either by encrypting the shift register, feeding back the output, or encrypting counter values). The plaintext or ciphertext is then XORed with the keystream — no block cipher decryption is involved.

Block Cipher Modes — CTR

Counter Mode

In CTR mode, there is no feedback from previous blocks at all. Instead, a counter is encrypted directly to produce a keystream, and each plaintext block is XORed with its corresponding keystream block:

$$\text{Keystream}_i = E_K(\text{Nonce} \| \text{Counter}_i)$$ $$C_i = P_i \oplus \text{Keystream}_i$$ $$P_i = C_i \oplus \text{Keystream}_i$$

The counter starts at an initial value (typically a nonce concatenated with 0, 1, 2, ...) and is incremented modulo \(2^b\) for each block. Since each keystream block depends only on its own counter value, all blocks are completely independent of each other.

Advantages of CTR Mode

PARALLELIZABLE

Each keystream block \(E_K(\text{Nonce}\|i)\) depends only on the counter value \(i\). All blocks can be computed simultaneously. Both encryption and decryption are fully parallelizable.

RANDOM ACCESS

To decrypt block \(j\) without decrypting preceding blocks, compute \(E_K(\text{Nonce}\|j)\) and XOR with \(C_j\). No sequential processing needed.

NO PADDING

XOR the keystream with plaintext byte-by-byte. Truncate to the exact message length. No block-alignment padding is required.

PRECOMPUTATION

The keystream can be computed before the plaintext is known. At transmission time, just XOR with the pre-computed keystream blocks.

The Critical Danger: IV Reuse

CTR mode is deadly if the same nonce (IV) is reused with the same key, even once. If two messages \(P^{(1)}\) and \(P^{(2)}\) are encrypted with the same key and same nonce:

$$C^{(1)}_i = P^{(1)}_i \oplus \text{KS}_i \quad \text{and} \quad C^{(2)}_i = P^{(2)}_i \oplus \text{KS}_i$$

The attacker computes:

$$C^{(1)}_i \oplus C^{(2)}_i = P^{(1)}_i \oplus P^{(2)}_i$$

With both plaintexts in natural language (or any structured format), the XOR can often be exploited to recover both plaintexts through "crib dragging" — guessing words in one plaintext and checking if the implied other plaintext makes sense.

Stream Ciphers & RC4

What is a Stream Cipher?

A stream cipher takes a different approach from block ciphers: instead of grouping plaintext into fixed blocks and encrypting each block as a unit, a stream cipher encrypts data continuously, one byte at a time. It generates a long sequence of pseudorandom bytes called a keystream, derived from the secret key. Each plaintext byte is XORed with the corresponding keystream byte:

$$C_i = P_i \oplus \text{KS}_i \qquad P_i = C_i \oplus \text{KS}_i$$

The security rests entirely on the quality of the keystream. A good keystream must be statistically indistinguishable from random: all 256 byte values should appear with equal frequency, the sequence should not repeat for a very long time, and knowledge of one byte should give no information about the next.

Stream ciphers are naturally suited to real-time communications where data arrives byte-by-byte and cannot wait for a block to fill — voice calls, video streams, and satellite communications have historically used stream ciphers. Modern practice often uses AES in CTR or OFB mode to achieve stream-cipher-like behavior with a well-analyzed block-cipher foundation.

RC4 — Rivest Cipher 4

RC4 (designed 1987, leaked publicly 1994) is one of the most widely deployed stream ciphers in history: SSL/TLS, WEP Wi-Fi security, and many other protocols used it. It is now deprecated due to statistical biases in its keystream, but it remains an important teaching example.

RC4 is a variable-key-length, byte-oriented stream cipher. Its central data structure is a state vector \(S\) — an array of 256 bytes, always forming a permutation of the values 0–255.

Phase 1: State Initialization

Fill \(S\) with values 0 through 255 in order — the identity permutation:

S[0] = 0x00, S[1] = 0x01, ..., S[255] = 0xFF

Phase 2: Build Temporary Vector T

If the key \(K\) has \(\text{keylen}\) bytes, fill a 256-element vector \(T\) by cycling through the key:

$$T[i] = K[i \bmod \text{keylen}], \quad 0 \leq i \leq 255$$

For example, with a 3-byte key \([k_0, k_1, k_2]\), the vector \(T\) repeats the key pattern 85 full times (and then once more for the final byte): \([k_0, k_1, k_2, k_0, k_1, k_2, \ldots]\).

Key Scheduling Algorithm (KSA)

The KSA uses \(T\) to produce a key-dependent scrambling of \(S\):

j = 0
for i = 0 to 255:
    j = (j + S[i] + T[i]) mod 256
    SWAP(S[i], S[j])

After the KSA, \(S\) is a pseudo-random permutation of 0–255 determined by the key. Swapping never destroys or duplicates values, so \(S\) always remains a valid permutation.

Keystream Generation (PRGA)

Once \(S\) is initialized, generate one keystream byte per iteration:

i, j = 0
while (true):
    i = (i + 1) mod 256
    j = (j + S[i]) mod 256
    SWAP(S[i], S[j])
    k = (S[i] + S[j]) mod 256
    output S[k]    ← XOR this with plaintext byte

Each iteration advances both pointers, swaps two elements (continuously evolving the permutation), computes a third index \(k\) from the sum of the two swapped values, and outputs \(S[k]\). This multi-level indirection makes the output hard to predict.

RC4 Key Reuse Vulnerability

RC4 shares the fundamental stream cipher weakness: never reuse a key. The same key produces the same keystream. If two messages are encrypted with the same keystream:

$$C^{(1)} \oplus C^{(2)} = P^{(1)} \oplus P^{(2)}$$

From the XOR of the two plaintexts, both can often be recovered when the content is natural-language text. WEP (Wi-Fi Encrypted Protocol) was completely broken partly because its short IV led to keystream reuse across messages.

Needham-Schroeder Key Distribution Protocol

The Problem: Key Distribution at Scale

Imagine a secure communication system for 1,000 users. The naïve approach — every user physically exchanges a key with every other user — requires \(\binom{1000}{2} = 499{,}500\) key exchanges. As the group changes, all those keys must be updated. This is completely infeasible.

A more efficient alternative: a Key Distribution Center (KDC). Each user shares exactly one master key with the KDC. To communicate with any other user, they ask the KDC to generate a temporary session key just for that conversation. But this raises immediate security questions: how does A know a response is really from the KDC? How does B know the request really came from A? How can B be sure the session key is fresh and not a replay from an old session?

The Needham-Schroeder protocol (1978) answers all these questions precisely using only symmetric encryption and carefully designed nonces.

KDCTrusted third partyKnows both \(K_A\) and \(K_B\)
\(K_S\)Session keyGenerated fresh by KDC for each session

Protocol Steps

1
A → KDC: Request a session key
$$E(K_A,\ [ID_A,\ ID_B,\ N_1])$$ A sends its identity, B's identity, and a fresh nonce \(N_1\) (a "number used once"), all encrypted under \(K_A\). The nonce is A's fingerprint on this specific request — it ensures the response matches this particular request and is not a replay of an old one.
2
KDC → A: Session key + Ticket for B
$$E(K_A,\ [K_S,\ ID_A,\ ID_B,\ N_1,\ E(K_B,\ [K_S,\ ID_A])])$$ KDC generates \(K_S\) and sends A: the session key, the original request echoed back (A verifies \(N_1\) to confirm freshness), and a ticket for B — \(E(K_B, [K_S, ID_A])\). A cannot read the ticket (no access to \(K_B\)), but B can. Since the whole response is encrypted under \(K_A\), only A can read it → A knows it came from the KDC.
3
A → B: Forward the ticket
$$E(K_B,\ [K_S,\ ID_A])$$ A forwards the ticket to B. It is still encrypted under \(K_B\) — A cannot tamper with it or read the session key out of it. Only B can decrypt this.
4
B → A: Challenge with nonce N₂
$$E(K_S,\ [N_2])$$ B decrypts the ticket using \(K_B\), obtains \(K_S\) and \(ID_A\). B now has the session key but needs to verify A is really on the other end. B generates a new nonce \(N_2\) and sends it encrypted under \(K_S\). Only someone with \(K_S\) (i.e., A or the KDC) can decrypt this.
5
A → B: Response to challenge
$$E(K_S,\ [N_2 - 1])$$ A decrypts \(N_2\), applies the transformation, and sends \(N_2 - 1\) back. B verifies: (1) the sender knows \(K_S\); (2) since \(N_2\) was just generated by B, the response is fresh — no replay. Session established.

What Does Each Step Actually Prove?

MessageWhat it proves
Message 2 includes \(N_1\)KDC's response matches A's specific request (freshness)
Message 2 encrypted under \(K_A\)Sender has \(K_A\) → response really came from KDC
Ticket encrypted under \(K_B\)Only B can extract \(K_S\) and \(ID_A\) — A cannot forge the ticket
Messages 4–5 with \(N_2\)A has \(K_S\) → A is who the KDC said they were. \(N_2\) is fresh → not a replay.

📋 2025 MIDTERM Q24
Which cryptographic primitive does the original Needham-Schroeder protocol rely on?
aHash functions
bSymmetric-key encryption
cDigital signatures
dElliptic curve cryptography
Explanation
✓ Answer: (b)
The protocol uses only symmetric-key encryption. Messages are encrypted with pre-shared master keys (\(K_A\), \(K_B\)) and the session key \(K_S\). No public-key cryptography, hashing, or digital signatures are involved in the original protocol.
❓ Why does the KDC include B's ticket in its response to A, rather than sending it directly to B?▾
The KDC sends everything through A to reduce protocol messages. A forwards the ticket to B. Since the ticket is encrypted with \(K_B\), A cannot read or tamper with it. This is a clean design: the KDC never needs to initiate communication with B.
❓ What specific attack does the nonce-challenge in messages 4–5 prevent?▾
Replay of message 3. An attacker who captured message 3 (the ticket) from an old session could replay it to B. But to respond correctly to B's challenge \(N_2\), they would need \(K_S\). If the old \(K_S\) is unknown, the attack fails. Additionally, \(N_2\) is freshly generated — a replayed response would not contain the right value.

Kerberos

Why Needham-Schroeder is Not Enough

The Needham-Schroeder protocol handles the core key distribution problem, but real networks need more. Consider a university: hundreds of students need access to printers, file servers, database servers, and web services. Each service needs to authenticate the student. The student should be able to log in once and access multiple services without re-authenticating every time — single sign-on (SSO).

Kerberos (originally designed at MIT in the 1980s) solves these scalability requirements while preserving Needham-Schroeder's security model. The key design change: split the KDC into two distinct servers.

The Two-Server Architecture

ServerRole
Authentication Server (AS)Verifies the client's identity (password). Issues a Ticket-Granting Ticket (TGT).
Ticket-Granting Server (TGS)Issues service-specific tickets. Client must present a valid TGT to reach TGS.

The client authenticates to AS once per login session. The TGT it receives can be reused for multiple requests to TGS — one per service needed. This is the SSO property: authenticate once, access many services.

Kerberos Keys

KeyShared BetweenType
\(K_\text{Client}\)AS & ClientLong-term (from password)
\(K_\text{TGS}\)AS & TGSLong-term (ticket encryption key)
\(K_\text{SP}\)TGS & Service ProviderLong-term (ticket encryption key)
\(K_\text{Client-TGS}\)AS → Client → TGSSession key (temporary)
\(K_\text{Client-SP}\)TGS → Client → SPSession key (temporary)
Q29 — Long-Term vs Session Keys
From Questions of Doubt: "Answers (b) and (c) are valid here. Ticket encryption keys, which are \(K_\text{TGS}\) and \(K_\text{SP}\), are long-term keys." Session keys (\(K_\text{Client-TGS}\), \(K_\text{Client-SP}\)) are temporary — never stored in the KDC database. Long-term keys (\(K_\text{Client}\), \(K_\text{TGS}\), \(K_\text{SP}\)) are persistent and stored by the KDC.

Kerberos Message Flow

Phase 1: Client → AS (Authentication)

1
Client → AS (plaintext)
Client sends its identity and the requested service (TGS). No key needed — sent in plaintext.
2+3

Message 3: TGT (encrypted with \(K_\text{TGS}\) — client cannot read it). Both encrypted under \(K_\text{Client}\) so only the legitimate client can decrypt.

4+5
Client → TGS
Message 4: the TGT + requested service ID. Message 5: Client Authenticator = \(E(K_\text{Client-TGS},\ [ClientID,\ \text{Timestamp}])\). TGS decrypts the TGT with \(K_\text{TGS}\) to get \(K_\text{Client-TGS}\), then decrypts the authenticator. The timestamp prevents replay: TGS rejects authenticators more than a few minutes old.

TGS → Client
Message 6: service ticket \(E(K_\text{SP},\ [ClientID,\ ClientIP,\ Validity,\ K_\text{Client-SP}])\). Message 7: \(K_\text{Client-SP}\) encrypted with \(K_\text{Client-TGS}\).

Phase 3: Client → Service Provider

8+9
Client → Service Provider
Message 8: service ticket (encrypted with \(K_\text{SP}\)). Message 9: Authenticator = \(E(K_\text{Client-SP},\ [ClientID,\ \text{Timestamp}])\). SP decrypts ticket to get \(K_\text{Client-SP}\), decrypts authenticator, verifies identity and timestamp.
10

$$E(K_\text{Client-SP},\ [\text{Timestamp} + 1])$$ Confirms SP has \(K_\text{Client-SP}\) and the session key is working. Client verifies this proves they are talking to the real SP.

Which key is reused across multiple sessions?

🔴 From "Questions of Doubt"
✓ Answers: (b) AND (c) — both valid
From Questions of Doubt: "Answers (b) and (c) are valid here. Ticket encryption keys, which are \(K_\text{TGS}\) and \(K_\text{SP}\), are long-term keys." Long-term keys are reused by definition. Ticket encryption keys (\(K_\text{TGS}\), \(K_\text{SP}\)) are also long-term — reused across sessions to encrypt different tickets. Session keys (a) and nonces (d) are never reused.
❓ Why does Kerberos separate AS from TGS?▾
The separation is a security principle: AS handles sensitive password-based authentication (directly touching \(K_\text{Client}\), derived from the password), while TGS handles service authorization (only ever seeing \(K_\text{Client-TGS}\), a temporary session key). A compromise of TGS does not expose password-derived keys. Additionally, AS can be made very restricted (contact only at login), while TGS can be more accessible.
❓ How does Kerberos prevent a stolen TGT from being used indefinitely?▾
TGTs have a validity period (e.g., 8 hours). A stolen TGT automatically expires. The shorter the validity period, the smaller the window for misuse. In high-security environments, the period can be set to minutes.
❓ What is the single sign-on (SSO) property of Kerberos?▾
The client authenticates to AS once per login session. The TGT it receives can be reused for multiple requests to TGS — one per service. The client does not re-enter a password for each service (printer, file server, etc.); it presents the TGT and gets a fresh service ticket each time.

Quick Knowledge Checks

❓ Why does 2DES fail despite a 112-bit combined key?▾
The meet-in-the-middle attack. Build a table of \(2^{56}\) forward encryptions from \(P\). Search \(2^{56}\) backward decryptions from \(C\). Match at the intermediate value. Total cost ≈ \(2^{57}\). The 112-bit key space is irrelevant because the attacker never searches it directly.
❓ What two properties must a good keystream have?▾
1. Statistical randomness: all byte values appear equally often; bytes are uncorrelated. 2. Long period: the sequence should not repeat for an extremely long time. Together these ensure that the keystream gives no information about the key or previous bytes.
❓ What is the difference between a master key and a session key in Needham-Schroeder?▾
Master keys (\(K_A\), \(K_B\)) are long-term — shared between each user and the KDC, used for many sessions. Session keys (\(K_S\)) are temporary — generated fresh by the KDC for one specific conversation and discarded afterward. Session keys limit damage if a key is later compromised.
❓ How does Kerberos achieve single sign-on?▾
The client authenticates to AS once per login session and receives a TGT. For each service it needs, the client presents the TGT to TGS and gets a service-specific ticket — without re-entering a password. AS is contacted only at login; TGS handles all subsequent service authorizations.
❓ Compare: how do Needham-Schroeder and Kerberos prevent replay attacks?▾
Needham-Schroeder: nonce-challenge (\(N_2\), \(N_2-1\)). B generates a fresh nonce; A must respond with the correct transformation. A replay of an old session cannot answer the new nonce. Kerberos: timestamps in authenticators. TGS and SP reject authenticators whose timestamps are more than ~5 minutes old. Kerberos requires synchronized clocks; Needham-Schroeder does not.

Comprehensive Review Questions

💡 REVIEW
An attacker captures a 2DES-encrypted message along with a known plaintext-ciphertext pair (P, C). The attacker builds a forward table from P and a backward table from C. How many false alarms should the attacker expect, and how do they resolve them?
Explanation
✓ ~2⁴⁸ false alarms
The forward table has \(2^{56}\) entries over a \(2^{64}\)-bit space. For each of the \(2^{56}\) backward queries, the probability of a random match is \(2^{56}/2^{64} = 2^{-8}\). Expected false matches: \(2^{56} \times 2^{56} / 2^{64} = 2^{48}\). To resolve: test each candidate \((K_1, K_2)\) on a second known pair \((P', C')\). The probability of a false alarm surviving is \(2^{-64}\) per additional pair, so one or two additional pairs typically leave only the real key.
💡 REVIEW
A developer encrypts user session tokens using AES-ECB. A security researcher claims they can detect when two different users have the same role (admin, user, guest) embedded in their token without breaking AES. Explain how and propose a fix.
Explanation
✓ ECB pattern leakage
If the role string ("admin", "user") is aligned to a 128-bit block boundary, identical role fields produce identical ciphertext blocks. The researcher collects tokens from multiple users, identifies repeated ciphertext blocks, and concludes those users share the same role value. Fix: use CBC or CTR mode with a fresh random IV per token. The IV ensures that even identical plaintext blocks produce different ciphertext, eliminating the detectable pattern.
💡 REVIEW
In a Kerberos deployment, the TGS server is compromised by an attacker. What can the attacker do, and what are they prevented from doing?
Explanation
✓ Limited damage by AS/TGS separation
The attacker gains access to \(K_\text{TGS}\) (used to encrypt TGTs) and \(K_\text{SP}\) values (used to issue service tickets). They can forge service tickets for any service provider and impersonate clients to services. However, they do NOT have access to clients' long-term keys \(K_\text{Client}\) (derived from passwords) — these are known only to AS and each individual client. So they cannot impersonate the KDC to clients, cannot obtain clients' passwords, and cannot issue new TGTs. The AS/TGS separation limits the damage scope.
