What Is a Hash Function? Part 1

The Core Idea

A hash function $h$ takes a variable-length input message $M$ and produces a fixed-length output — the hashcode, message digest, or digest:

$$h : \{0,1\}^* \to \{0,1\}^n$$

For example, SHA-256 accepts messages of essentially any length and always outputs exactly 256 bits. SHA-512 always outputs 512 bits, regardless of whether the input is 1 byte or 1 GB.

The key property: any change to the input — even flipping a single bit — produces a completely different hash. This avalanche effect is what makes hashes useful as tamper-evident fingerprints.

Why Hash Functions Matter

  • File integrity: "Does this downloaded file match what the server sent?" Compare hashes.
  • Password storage: Store $h(\text{password})$, not the password. At login, hash the input and compare.
  • Digital signatures: Hash the document (fast), then sign the hash (small). The signature is binding because changing the document changes its hash.
  • Message authentication: Prove a message was not tampered with in transit.
  • Blockchain: Proof-of-work mining, ownership authentication, double-spending prevention.
💡 SHA-512 Specifics

SHA-512 accepts messages up to $2^{128}$ bits in length and produces a 512-bit digest. It processes input in 1024-bit blocks, uses 80 rounds of processing, and maintains eight 64-bit internal registers.

🧠 Knowledge Check
What happens if you change one bit in the input to a hash function?
Show Answer ▾
The hash output changes completely and unpredictably — this is the avalanche effect. A good hash function ensures that any single-bit change in the input changes approximately 50% of the output bits.
Why can't you store the raw hash in SSL/TLS certificates and use it as a session key?
Show Answer ▾
Hashing is one-way and keyless — there's no secret. Anyone can compute the same hash. A session key must be secret, not publicly computable. Hash functions are for integrity (detecting changes), not confidentiality.

Security Properties of Hash Functions Part 2

Property 1: Pre-Image Resistance (One-Way)

Given a hash value $h$, it is computationally infeasible to find any message $M$ such that $\text{hash}(M) = h$. You cannot reverse the hash. This is what makes password hash storage safe — even if an attacker gets the hash database, they cannot easily recover the original passwords.

Property 2: Weak Collision Resistance (Second Pre-Image Resistance)

Given a specific message $M_1$ and its hash $h(M_1)$, it is computationally infeasible to find a different message $M_2 \neq M_1$ such that:

$$h(M_2) = h(M_1)$$

This protects against substituting a signed document with a different one that has the same hash. The starting message $M_1$ is fixed — the attacker must work from a specific document.

Property 3: Strong Collision Resistance

It is computationally infeasible to find any two different messages $M_1 \neq M_2$ such that $h(M_1) = h(M_2)$. The attacker gets to choose both $M_1$ and $M_2$ freely — no starting point is fixed. This is strictly harder for the attacker to achieve than weak collision.

The Relationship Between Properties

📌 Key Implication

Strong collision resistance implies weak collision resistance. If you cannot find any colliding pair (strong), you certainly cannot find one starting from a fixed message $M_1$ (weak). Weak is a special case of strong. Therefore: strong ⊇ weak.

Pre-image resistance is independent — a hash can be collision resistant without being pre-image resistant, and vice versa.

PropertyGivenGoal (attacker)Effort
Pre-image resistanceHash $h$Find any $M$ with $\text{hash}(M) = h$$\approx 2^n$
Weak collision (2nd pre-image)Message $M_1$Find $M_2 \neq M_1$ with same hash$\approx 2^n$
Strong collision (birthday)Nothing fixedFind any pair $M_1 \neq M_2$ same hash$\approx 2^{n/2}$
🧠 Knowledge Check
A hash has 160-bit output. What is the approximate birthday attack effort?
Show Answer ▾
Birthday attack: $2^{160/2} = 2^{80}$ tries. Pre-image attack: $2^{160}$ tries. The birthday attack is dramatically more efficient — $2^{80}$ is far more feasible than $2^{160}$.
Does strong collision resistance imply pre-image resistance?
Show Answer ▾
No — these are independent properties. A hash function can be strongly collision resistant (no one can find any two colliding messages) but not pre-image resistant (given a hash, you can find an input). They protect against different types of attacks.

Message Authentication Schemes Part 3

Six Standard Schemes

Let $M$ = message, $K$ = shared symmetric key, $PR_A$ = sender A's private key, $h(M)$ = hash of $M$, $S$ = shared secret string.

⚠️ Critical Warning

Sending $M \| h(M)$ with no encryption or shared secret provides zero security. An attacker modifies $M$ to $M'$, recomputes $h(M')$, and sends $M' \| h(M')$. Always protect the hash.

Scheme 1

Authentication + Confidentiality (Symmetric)

A → B: C = E(K, [M || h(M)])

Concatenate message + hash, then encrypt everything. B decrypts, recomputes hash, verifies. ✅ Authentication + Confidentiality

Scheme 2

MAC — Authentication Only

A → B: C = M || E(K, h(M))

Message in cleartext; only the hash is encrypted. Only B (with key $K$) can verify. ✅ Authentication only (Message Authentication Code)

Scheme 3

Digital Signature — Authentication with Private Key

A → B: C = M || E(PR_A, h(M))

A encrypts hash with its private key. Anyone with $PU_A$ can verify. ✅ Authentication + Non-repudiation

Scheme 4

Digital Signature + Symmetric Confidentiality

A → B: C = E(K, [M || E(PR_A, h(M))])

Sign first, then encrypt the whole bundle. ✅ Authentication + Non-repudiation + Confidentiality

Scheme 5

Shared Secret — No Encryption Algorithm

A → B: C = M || h(M || S)

$S$ = shared secret. Hash computed over message + secret. No encryption needed. ✅ Authentication (B verifies by computing $h(M||S)$ and comparing)

Scheme 6

Shared Secret + Confidentiality

A → B: C = E(K, M || h(M || S))

Scheme 5 plus symmetric encryption of the whole message. ✅ Authentication + Confidentiality

Simple Hash Functions & Their Weaknesses Part 4

The XOR Hash

The simplest possible hash: partition the message into $n$-bit blocks $X_1, X_2, \ldots, X_m$ and XOR them all:

$$\Delta(M) = X_1 \oplus X_2 \oplus \cdots \oplus X_m$$

This is trivially broken. An attacker replacing $X_1, \ldots, X_{m-1}$ with any desired $Y_1, \ldots, Y_{m-1}$ simply computes:

$$Y_m = Y_1 \oplus Y_2 \oplus \cdots \oplus Y_{m-1} \oplus \Delta(M)$$

This ensures $Y_1 \oplus \cdots \oplus Y_m = \Delta(M)$ — same hash, completely different content. The attacker has full freedom to replace the entire message.

The Rotated-XOR (ROXR) Hash

A slightly better attempt: after processing each block, rotate the partial hash one bit circularly. This adds position-sensitivity but still fails: an attacker can append a gibberish block to a forged message $M_2$ to force the final hash to equal $h(M_1)$.

The Fix: Include Message Length

✅ Solution

All production hash functions include the message length in what gets hashed (as part of the padding). Changing the length changes the padded input, preventing the "append gibberish to equalize hash" attack. SHA-1 padding includes the 64-bit length field; SHA-512 uses 128 bits for the length field.

SHA-1 Padding

Given message $M$ of $L$ bits:

  1. Append a single 1 bit.
  2. Append $K$ zero bits, where $K$ is the smallest non-negative integer satisfying: $(L + 1 + K) \bmod 512 = 448$
  3. Append the 64-bit binary representation of $L$.

Result: total length is a multiple of 512 bits. The last 64 bits are the length; 448 is chosen because $448 + 64 = 512$.

WORKED EXAMPLE SHA-1 Padding for L = 1000 bits

$(1000 + 1 + K) \bmod 512 = 448$

$1001 + K \equiv 448 \pmod{512}$

$K \equiv 448 - 1001 \equiv -553 \equiv -553 + 2 \times 512 = 471 \pmod{512}$

$K = 471$ zero bits.

Total: $1000 + 1 + 471 + 64 = 1536 = 3 \times 512$ ✓

Birthday Attack — Collision Probability Part 5

Weak Collision Probability (Pre-Image)

With $n$-bit hash and $N = 2^n$ possible values: to find any message matching a specific hash $h$ with probability 0.5, need approximately:

$$k \approx 0.5 \times N = 2^{n-1} \approx 2^n$$

Strong Collision — The Birthday Paradox

The birthday paradox: 23 random people have >50% probability of sharing a birthday. Applied to hashing: a pool of $k$ random messages has probability of at least one matching pair:

$$P(k) = 1 - \frac{N!}{(N-k)! \cdot N^k} \approx 1 - e^{-k(k-1)/(2N)}$$

Setting $P(k) = 0.5$ and solving:

$$k \approx \sqrt{N} = \sqrt{2^n} = 2^{n/2}$$

This is the birthday bound: finding any strong collision requires only $2^{n/2}$ tries — dramatically fewer than $2^n$ for pre-image attacks.

Birthday Attack on Digital Signatures (The Contract Forgery)

The birthday attack can forge digital signatures:

  1. Mr. Creepy creates $k \approx 2^{n/2}$ variations of a legitimate contract $\{c_1, \ldots, c_k\}$ (semantic synonyms, punctuation changes).
  2. Creepy creates $k$ variations of a fraudulent contract $\{f_1, \ldots, f_k\}$.
  3. With probability ≈ 0.5, there exists $(c_i, f_j)$ such that $h(c_i) = h(f_j)$.
  4. Get Mr. BigShot to digitally sign $c_i$ (the legitimate version).
  5. Replace the signed contract with $f_j$ (the fraudulent version). The signature still verifies!
⚠️ Security Implication

The birthday bound halves the effective security. A 256-bit hash gives only 128-bit collision security. SHA-1 (160-bit) offers only $2^{80}$ collision security — considered insufficient. SHA-256 (256-bit) offers $2^{128}$ — the current recommended minimum.

AttackAttacker's GoalEffort
Pre-imageFind $M$ with given hash $h$$\approx 2^n$
2nd Pre-image (weak collision)Find $M_2$ matching hash of given $M_1$$\approx 2^n$
Birthday (strong collision)Find any pair with same hash$\approx 2^{n/2}$
🧠 Knowledge Check
MD5 produces a 128-bit hash. What is the birthday attack cost to find a collision?
Show Answer ▾
$2^{128/2} = 2^{64}$ tries. This is feasible for modern hardware — $2^{64}$ operations can be done in hours with GPUs. MD5 is considered completely broken for collision resistance. SHA-256 ($2^{128}$ cost) is the current minimum.

Merkle-Damgård Construction Part 6

The Structure

Proposed by Ralph Merkle in 1979, the Merkle-Damgård (MD) construction forms the basis of MD5, SHA-1, SHA-2, and Whirlpool. It processes a padded message block by block:

  1. Pad the message and partition into $L$ blocks of $b$ bits each (final block includes message length).
  2. Initialize state: $H_0 = IV$ (a fixed, published constant — the same for every use of the hash function).
  3. For each block $i$: $H_i = f(H_{i-1}, B_i)$ where $f$ is the compression function.
  4. Final output: $H_L$ is the hash digest.
💡 Why "Compression"?

The compression function $f$ takes a $b$-bit block and an $n$-bit state, producing an $n$-bit output. Since $b > n$, it genuinely compresses. It is NOT file compression — it reduces the combined input size down to just the $n$-bit state.

The IV Is Not Secret

Unlike IVs in CBC or CTR mode, the hash function IV is a fixed public constant — the same for every computation of the same hash function. It is published in the hash specification. Its role is to start from a defined, non-trivial state, not to add secrecy.

Security Reduction

If the compression function $f$ is collision resistant, then the overall Merkle-Damgård hash is collision resistant. This is a formal security reduction — cryptographers need only prove the compression function is secure, and the theorem guarantees the full hash is secure. SHA-2 has no known compression function weaknesses as of 2024.

🧠 Knowledge Check
What is the role of the compression function in Merkle-Damgård?
Show Answer ▾
It processes one block of the message at a time, taking the current block ($b$ bits) and the current hash state ($n$ bits), and producing an updated hash state ($n$ bits). It is called compression because $b > n$ — more bits go in than come out.
SHA-3 uses a different structure — what is it?
Show Answer ▾
The sponge construction (Keccak algorithm). Unlike Merkle-Damgård, it is not vulnerable to length extension attacks. It "absorbs" input and then "squeezes" output, using a large fixed-width permutation function.

The SHA Family Part 7

SHA-1 (Broken)

SHA-1 produces a 160-bit hash. Despite being theoretically broken since 2005, it continues to be used in legacy SSL/TLS, SSH, PGP, S/MIME, and IPsec.

YearEvent
2005Theoretical break: $2^{69}$ operations to find collision (less than $2^{80}$ brute force)
2010NIST withdrew SHA-1 approval for U.S. government applications
2017"SHAttered" — Google produced two real PDF files with identical SHA-1 hashes using GPU clusters

SHA-2 (Current Standard)

SHA-256, SHA-384, and SHA-512 are collectively SHA-2. No practical attacks are known as of 2024. NIST recommends SHA-256 as the minimum for new applications.

SHA-512 Internals

  • Block size: $b = 1024$ bits per block
  • Padding: Last 128 bits of the final block encode the message length
  • Message schedule: Each 1024-bit block (16 × 64-bit words) is expanded to 80 words
  • Rounds: 80 rounds of compression per block
  • State: Eight 64-bit registers ($a$ through $h$) — concatenated they form the 512-bit output

MAC and HMAC Part 8

Why Simple Hashing Provides No Security

Sending $M \| h(M)$ with no secret provides zero protection. An adversary who sees $\{M, h(M)\}$ replaces $M$ with any $M'$, recomputes $h(M')$, and sends $M' \| h(M')$. The recipient cannot detect the substitution.

Why Simple XOR-Based MAC Fails

For $M = X_1 \| X_2 \| \cdots \| X_m$, the XOR-based MAC is $\text{MAC}(K, M) = E(K, X_1 \oplus \cdots \oplus X_m)$.

An attacker replaces $X_1, \ldots, X_{m-1}$ with $Y_1, \ldots, Y_{m-1}$ and sets:

$$Y_m = Y_1 \oplus Y_2 \oplus \cdots \oplus Y_{m-1} \oplus \Delta(M)$$

This preserves the XOR, so the MAC is unchanged. The forged message passes verification.

HMAC — The Standard Solution

HMAC is used in IPsec, SSL/TLS, and many other protocols. It defends against length extension by hashing the key twice:

$$\text{HMAC}_K(M) = h\!\left( (K^+ \oplus \text{opad})\ \|\ h\!\left( (K^+ \oplus \text{ipad})\ \|\ M \right) \right)$$

Where:

  • $K^+$: key $K$ zero-padded to length $b$ (hash block size)
  • ipad: byte 0x36 repeated $b/8$ times
  • opad: byte 0x5C repeated $b/8$ times
💡 Why Two Different Pads?

The ipad and opad derive two different key variants from $K^+$. The inner hash uses $K^+ \oplus \text{ipad}$ and the outer hash uses $K^+ \oplus \text{opad}$. Two different keys at two different levels prevent length extension: an attacker who extends the inner hash cannot use that to compute the outer HMAC without knowing $K$.

HMAC vs. Digital Signature

FeatureHMACDigital Signature
Key typeSymmetric (shared secret)Asymmetric (private key)

🧠 Knowledge Check
Why does HMAC use two hashing levels instead of just $h(K \| M)$?
Show Answer ▾
$h(K \| M)$ is vulnerable to length extension attacks in Merkle-Damgård hash functions. Given $h(K \| M)$, an attacker can compute $h(K \| M \| \text{padding} \| M')$ for any extension $M'$ without knowing $K$, because the hash state after processing $K \| M$ is the starting point for the next block. The outer hash in HMAC wraps the result with the key again, breaking this extension chain.
Why can HMAC not provide non-repudiation?
Show Answer ▾
HMAC uses a symmetric key — both sender and receiver hold the same key. Either party could have generated the MAC. There's no way to prove which one created it. Non-repudiation requires asymmetric keys: only the private key holder can create the signature, so the signer cannot deny authorship.

EXAM Q49Purpose of Hash Functions
What is the primary purpose of a cryptographic hash function?
aGenerate a fixed-size output from variable-size input
bEncrypt data for secure transmission
cProvide compression for large files
dCreate digital signatures
Answer: (a). A hash function maps any-length input to a fixed-length digest (fingerprint). (b) is wrong — hashing is one-way and keyless, not encryption. (c) is wrong — the "compression" is mathematical (many inputs → small output), not file compression like zip. (d) is wrong — hashing is a component of signatures but is not itself a signature scheme.
EXAM Q50Collision Resistance
What does "collision resistance" mean in hash functions?
aIt should be hard to find two different inputs with the same hash
bIt should be easy to reverse the hash output
cThe hash output should always be unique
dThe hash function should use a secret key
Answer: (a). Strong collision resistance means it is computationally infeasible to find any $M_1 \neq M_2$ with $h(M_1) = h(M_2)$. (c) is impossible by pigeonhole principle — with infinite inputs and finite outputs, collisions must mathematically exist; the goal is to make them hard to find, not to eliminate them.
EXAM Q51Birthday Attack
Which attack involves finding any two inputs that produce the same hash?
aPre-image attack
bBirthday attack
cBrute-force attack
dMan-in-the-middle attack
Answer: (b). A birthday attack exploits the birthday paradox to find any two messages with the same hash (strong collision), requiring only $\approx 2^{n/2}$ tries. (a) Pre-image attack finds a message with a specific given hash, requiring $\approx 2^n$ tries — harder.
EXAM Q52 ❌ NO CORRECT ANSWER Weak Collision Resistance
What is a "weak collision resistance" in hash functions?
aIt should be hard to find two different inputs with the same hash
bIt should be easy to modify an input without changing its hash
cGiven a hash output, it should be hard to find any input that produces it
dThe hash function should use a secret key

⚠️ Questions of Doubt: None of the options is correct.

(a) describes strong collision resistance. (c) describes pre-image resistance.

The correct definition of weak collision resistance (second pre-image resistance): given a specific message $M_1$ and its hash $h(M_1)$, it is computationally infeasible to find $M_2 \neq M_1$ such that $h(M_2) = h(M_1)$. The key distinction from strong: $M_1$ is fixed by the adversary's challenge, not freely chosen.

EXAM Q53Weak vs Strong Collision
What is the difference between weak (second pre-image) and strong collision resistance?
aWeak collision resistance deals with fixed inputs; strong deals with any two inputs
bWeak is about reversing the hash; strong is about finding any collision
cWeak requires a secret key; strong does not
dThere is no difference
Answer: (a). Weak (second pre-image) resistance: given a fixed $M_1$, cannot find $M_2 \neq M_1$ with same hash. Strong: cannot find any two messages (both chosen freely) with same hash. The attacker's freedom is what differs. Strong is harder to break because the attacker can choose optimally; in weak, the challenge message is imposed.
EXAM Q54 ✅ CONFIRMED (c)
If a hash function is strong collision-resistant, does it imply weak collision resistance?
aNo, because they are independent properties
bOnly if the hash function is also pre-image resistant
cYes, because strong collision resistance is a stronger property
dOnly for keyed hash functions
Answer: (c). Strong collision resistance says: cannot find any two messages with same hash. Weak says: given $M_1$, cannot find $M_2$ with same hash. Strong implies weak — if you cannot find any colliding pair at all, you certainly cannot find one that starts from a specific $M_1$. The strong property is strictly harder to violate.
EXAM Q55Hash Function Structure
What is the most common structural design used in traditional hash functions like SHA-1 and SHA-2?
aSponge construction
bMerkle-Damgård construction
cFeistel network
dDavies-Meyer construction
Answer: (b). SHA-1, SHA-2, MD5, and Whirlpool all use the Merkle-Damgård construction. (a) Sponge is SHA-3/Keccak. (c) Feistel networks are used in DES/block ciphers. (d) Davies-Meyer is a method of building a compression function from a block cipher — it can be used inside Merkle-Damgård, but is not the top-level structure.
EXAM Q56Compression Function
Which component of a hash function processes one block at a time and updates the internal state?
aInitialization Vector (IV)
bPermutation function
cKey schedule
dCompression function
Answer: (d). The compression function $f$ takes the current $b$-bit message block and $n$-bit state, producing an updated $n$-bit state. It is the core iterative engine of Merkle-Damgård. The IV provides the initial state (once). There is no key schedule in a hash function (hashing is keyless).
EXAM Q57IV Role in Hash
What is the role of the "initialization vector (IV)" in a hash function?
aActs as a secret key for encryption
bProvides the initial fixed value for the hash computation
cEnsures collision resistance
dShortens the output hash
Answer: (b). The hash IV is a fixed constant (published in the standard) that initializes the state $H_0$ before the first block is processed. It is not secret — unlike CBC/CTR IVs, the hash IV is identical for every computation of the hash function. It starts the computation from a defined non-trivial state, not to add confidentiality.
EXAM Q58HMAC Requirements
Which of the following components does HMAC require?
aA public key and a block cipher
bAn IV (Initialization Vector)
cA secret key and a hash function
dA digital certificate
Answer: (c). HMAC requires exactly two ingredients: a symmetric secret key $K$ and an underlying hash function (e.g., SHA-256 for HMAC-SHA-256). No public key, no block cipher used directly, no certificate. This is what makes HMAC simple, fast, and widely used.
EXAM Q59HMAC vs Digital Signature
How does HMAC differ from a digital signature?
aHMAC uses symmetric keys; signatures use asymmetric keys
bHMAC is faster but provides non-repudiation
cSignatures are shorter than HMAC outputs
dHMAC requires a certificate authority
Answer: (a). HMAC uses a symmetric shared key — both sender and receiver hold it. Digital signatures use asymmetric keys — private key signs, public key verifies. HMAC does NOT provide non-repudiation (both parties could have created the MAC), while signatures DO. (b) is wrong about non-repudiation. (c) is wrong — HMAC output size equals the underlying hash size.
EXAM Q60HMAC Formula
What is the formula for HMAC?
a$H\!M\!A\!C(K, m) = H(K \| m)$
b$H\!M\!A\!C(K, m) = H(K \oplus \text{opad} \| H(K \oplus \text{ipad} \| m))$
c$H\!M\!A\!C(K, m) = H(m \| K)$
d$H\!M\!A\!C(K, m) = \text{Encrypt}(K, H(m))$
Answer: (b). The correct HMAC formula uses two-level hashing with XOR pads: $$\text{HMAC}_K(M) = h\!\left( (K^+ \oplus \text{opad})\ \|\ h\!\left( (K^+ \oplus \text{ipad})\ \|\ M \right) \right)$$ Options (a) and (c) are naive constructions vulnerable to length extension attacks. (d) is a different construct (MAC using block cipher encryption of hash). The double-hashing structure with different pads is what makes HMAC secure.
